Architecture Fusion Map / 2026-03-27

MiroFish、DeerFlow 与 OpenFons 的架构融合图

这不是“二选一”的对比，而是把三者放回同一个坐标系里看清楚： 谁更擅长需求编译，谁更擅长执行 runtime，谁更适合作为 OpenFons 的长期主骨架。

MiroFish = 垂直产品壳 DeerFlow = Super Agent Harness OpenFons = Opportunity + Evidence Core

一句话结论

保留

OpenFons 必须保留 `OpportunitySpec -> User Confirmation -> TaskSpec / WorkflowSpec -> EvidenceSet -> ReportSpec -> Artifact` 这条受控主链。

借 DeerFlow

借它的 `runtime / skills / tools / sub-agents / sandbox / provider adapters` 组织方式。

借 MiroFish

借它的“让复杂流程被看见”的产品表达和报告交付形态。

1. 三个项目的定位差异

先看中心重心，而不是先看具体技术栈。这样更容易判断“该借哪一层”。

Vertical Product Shell

MiroFish

更像一个已经包装好的分析与模拟产品，擅长把“输入材料 -> 构图 -> 模拟 -> 报告”讲成完整产品故事。

  • 强项：需求编译、模拟世界构建、图谱可视化、报告交付。
  • 本质：更偏“产品壳”，不是通用的 agent runtime。
  • 边界：更适合“主题已知后的分析与推演”，不擅长机会判断前置门禁。
Horizontal Runtime

DeerFlow

更像一个 super agent harness，擅长组织 lead agent、skills、sandbox、memory、sub-agents 和 provider adapter。

  • 强项：执行层组织、技能装载、任务下发、sandbox 与多 provider 适配。
  • 本质：更偏“运行时基础设施”，不是垂直交付产品。
  • 边界：默认不以 Evidence 资产沉淀和 Opportunity 门禁为核心。
Control + Evidence Core

OpenFons

目标不是单纯做一个 agent 平台，而是把用户问题先编译成值得做的机会，再以受控采集和证据沉淀驱动报告交付。

  • 强项：`Intent Structuring -> Planning Swarm -> Opportunity Judge`。
  • 本质：以 `OpportunitySpec` 和 `EvidenceSet` 为真源的研究与交付系统。
  • 边界：v1 不应把模拟世界作为默认主链，而应把它视为可选高级分析模块。

2. 三条端到端流程对照

这里最关键的不是“它们都用了 AI”，而是“AI 在流程里承担的角色”到底不同。

MiroFish 真实主链

前半段的核心不是“先搜索”，而是“先把需求编译成一个可模拟的世界”。

用户输入 + 上传材料输入 `simulationRequirement`，同时提供文档种子。
Requirement CompilerLLM 结合材料生成实体类型、关系类型、分析摘要。
图谱构建把需求转成可执行的世界结构，而不是先转成外部搜索计划。
模拟配置生成时间配置、热点话题、叙事方向、初始帖子、Agent 活动配置。
模拟运行在模拟平台中形成动态演化结果。
报告阶段检索`insight_forge / panorama_search / quick_search` 在图谱内检索、拆问题、出报告。
可借点：把模糊需求编译成结构化执行上下文。不可照搬点：不能用单个大 prompt 直接决定全部下游事实。

DeerFlow 2.0 主链

更像“任务下达后，如何把 agent 执行层组织起来”。

任务输入用户或系统把任务交给 lead agent / harness。
Lead Agent / Planner分析任务、拆分步骤、决定是否拉起 sub-agents。
Skills / Tools通过内建技能、工具、文件系统和 memory 获取执行能力。
Sandbox / Provisioner在受控环境中执行代码或高风险步骤。
Provider Adapters模型路由、认证、能力矩阵与 API 兼容适配。
结果返回汇总执行结果，交回前端控制面或调用方。
可借点：runtime、skills、sub-agents、sandbox。不可照搬点：不能让 thread / memory 取代 OpenFons 的证据真源。

OpenFons 目标主链

重点不是“快执行”，而是“先判断值不值得做，再受控执行并沉淀资产”。

User Input自然语言问题、主题、目标受众、市场与上下文。
Intent Structuring把原始提问整理成结构化意图与候选方向。
Planning Swarm多角色受控分析：需求、竞争、变现、机会评分。
Opportunity Judge形成唯一外部前置契约 `OpportunitySpec`。
User Confirmation只有确认通过后，才允许把机会判断推进到任务编译层。
Task Compiler把 `OpportunitySpec` 编译成 `TaskSpec / WorkflowSpec`，明确执行边界与交付要求。
Controlled Collection / Worker Execution只允许在受控 workflow 中采集、分析和回传结果。
EvidenceSet -> ReportSpec -> Artifact沉淀证据资产，再交付页面、报告、长期 watch 主题。
这条链真正的护城河不是“能跑 agent”，而是“能把机会判断、证据门禁、资产沉淀做成可追溯主线”。

3. 四层架构映射

把三个项目都投到同一张分层图上，就能看清楚各自的空白区和强项区。

四层坐标系

OpenFons 四层架构坐标系，对应各层作用与要求。
层 作用 OpenFons 要求
Access Plane IM / Web / API 入口与用户交互 统一进入同一条编译主链
Control Plane Intent、Planning、Opportunity、Policy 必须以 `OpportunitySpec` 为唯一前置真源
Execution Plane skills、workers、tools、sandbox、providers 受控执行，不允许绕过控制面直接野跑
Data & Delivery Topic、Evidence、Artifact、Report Web 证据资产可审计、可回放、可沉淀

三项目映射结果

MiroFish、DeerFlow 与 OpenFons 在四层架构中的强弱映射。
项目 最强层 最弱层
MiroFish Requirement compilation + Delivery 通用 control plane 与可复用 execution plane
DeerFlow Execution plane Opportunity gate 与 evidence-first data plane

4. OpenFons 最佳实践融合架构

推荐不是“混合成大杂烩”，而是明确每一层该保留什么、借什么、包装什么。

推荐总原则

控制面保留 OpenFons 自己的判断逻辑，执行面借 DeerFlow 的 runtime 组织方式，交付面借 MiroFish 的可视化与报告表达。

Layer 01

Access Plane

IM、Web、API、外部系统入口。

User Input Control Web Ops Web Report Web
这里借 MiroFish 的不是“业务逻辑”，而是“让用户看见过程与结果”的产品表达。
Layer 02

Control Plane

这是 OpenFons 的核心，不应该外包给通用 planner。

Intent Structuring Planning Swarm Opportunity Judge OpportunitySpec Task Compiler Policy Gate
这层必须继续由 OpenFons 自己定义契约，不能被 DeerFlow 风格的 lead agent 吞掉。
Layer 03

Execution Plane

最适合借 DeerFlow 的组织方式，但输入必须来自显式契约。

Runtime Harness Skill Loader Tools Registry Sub-Agent Executor Sandbox / Provisioner Provider Adapters Controlled Collection
执行层不该直接把 thread 或 memory 当真源，它只负责受控执行和结果回传。
Layer 04

Data & Delivery

这是 OpenFons 与普通 agent harness 最不一样的地方。

Source Registry Topic TopicRun Evidence EvidenceSet ReportSpec Artifact Narrative Report UX
这层要把“采完就丢”的信息，变成长期可沉淀、可二次消费、可审计的资产。

5. 该借什么，不该借什么

融合不是平均主义，关键是把各自最有价值的部分放到正确层位。

推荐借鉴

OpenFons 推荐借鉴的能力来源与对应内容。
来源 内容
From DeerFlow skills / tools / sub-agents / sandbox / provider adapter / harness 组织方式
From MiroFish 需求编译感、过程可视化、图谱与报告型交付体验
Keep OpenFons Opportunity 门禁、Evidence 资产化、ReportSpec 契约、可回放真源

明确避免

OpenFons 在借鉴 MiroFish 与 DeerFlow 时应明确避免的做法。
动作 说明
Avoid 不要把 MiroFish 的模拟主链直接当成 OpenFons v1 主链。
Avoid 不要让 DeerFlow 风格的 lead agent 越过 `OpportunitySpec` 直接调度全系统。
Avoid 不要把 memory / thread 当成长期真源，真源必须落在 contracts 与 evidence artifacts。
Avoid 不要让单个 opaque prompt 同时负责“理解需求 + 规划收集 + 定义事实 + 生成结论”。

6. 推荐演进路线

先把主骨架打稳，再逐层吸收参考项目的优点，不要一上来追求全部能力同时到位。

现在

先把 OpenFons 的控制面和证据主链做稳。

  • 固定 `OpportunitySpec / TaskSpec / WorkflowSpec / ReportSpec`。
  • 固定 `Evidence / EvidenceSet / Artifact` 数据真源。
  • 先跑通受控采集、分析和报告交付闭环。

下一步

引入 DeerFlow 风格的执行平面，而不是整个产品壳。

  • 补 runtime harness、skill loader、tool registry。
  • 补 provider credential schema 与 capability matrix。
  • 补 sandbox / provisioner，支持更强的执行隔离。

更后面

把 MiroFish 风格的推演能力做成高级分析模块，而不是默认路径。

  • Requirement Compiler / Scenario Compiler。
  • 高级世界模型、角色推演、场景实验室。
  • 图谱、时间线、知识地图与更强的报告型交互。
最终建议可以压缩成一句话： OpenFons 应该用 DeerFlow 的方式组织执行，用 MiroFish 的方式包装交付，但绝不能放弃自己以 Opportunity 与 Evidence 为核心的控制层。
