# 利基选题标准表 v1 案例评分

> 项目名：OpenFons
> 文档定位：`利基选题标准表-v1.md` 的正式案例打分记录
> 评审日期：2026-03-27
> 状态说明：这是第一阶段的 `pre-launch scoring baseline`，用于帮助团队把标准表真正用起来，不代表最终上线结果已经验证

## 1. 文档目的

本文件用于把已经确认的 `利基选题标准表 v1` 应用到两个第一阶段盈利方向上：

1. `OpenClaw 部署 SEO 主题`
2. `AI 编程与 Agent 时代模型采购、路由、成本与地区选择`

它的目标不是重复标准表定义，而是回答更实际的 4 个问题：

1. 用这张表实际打分时，应该怎么填
2. 这两个题为什么都能过总门禁
3. 为什么 `OpenClaw` 更偏内容机会，而 `AI procurement` 同时具备较强产品机会
4. 在第一阶段各自最合理的动作是什么

## 2. 使用说明

本文件中的分数都遵守 [利基选题标准表-v1.md](./利基选题标准表-v1.md) 的规则：

1. 先看 `总门禁表`
2. 仅当总门禁非 `No-Go` 时，再看 `内容机会子表` 和 `产品机会子表`
3. 最后按决策映射规则落到第一阶段动作

需要再次强调：

1. 这是一份 `pre-launch baseline`
2. 评分结论服务于第一阶段排期与验证，不是永久裁决
3. 第一阶段默认外部交付仍以 `ReportSpec + report-web` 为主
4. 产品机会结果先保留在 `productOpportunityHints / expansionCandidates`

## 3. 案例一：OpenClaw 部署 SEO 主题

### 3.1 基础记录

| 字段 | 内容 |
| --- | --- |
| `Topic` | OpenClaw 部署 SEO 主题 |
| `Seed Question` | 围绕 OpenClaw 的本地部署、Windows、Docker、VPS、托管等方向获取流量并形成可持续更新的报告页 |
| `Audience` | 普通用户、开发者、小团队 |
| `Geo / Language` | 候选市场暂定 `US/en` 与 `CN/zh-CN` |
| `Assumptions` | 目前仍以公开报告页验证为第一阶段默认路径；市场尚未最终锁定到单一国家/语言 |
| `Confidence Level` | `Medium-High` |
| `Actual Outcome Snapshot` | 尚未进入以 Search Console 和真实转化数据回填的正式上线验证阶段，当前为上线前基线评分 |

### 3.2 总门禁评分

| 维度 | 分数 | 理由 | 关键证据 |
| --- | --- | --- | --- |
| `Problem` | 3 | 用户问题具体，真实在问“哪种部署方式更适合我” | 官方安装文档明确存在多条安装路径与平台差异 |
| `Intent` | 3 | 这是明显的部署选择与执行意图，不是泛资讯浏览 | 用户问题直接落在部署、托管、VPS、系统选择 |
| `Audience` | 2 | 人群已大体明确，但还未完全细分到更窄人群切面 | 已知至少可区分普通用户、开发者、小团队 |
| `Market Context` | 2 | 英文市场与中文市场都有机会，但当前仍处于候选市场阶段 | 现阶段仍是 `US/en` 与 `CN/zh-CN` 双候选 |
| `Evidence` | 3 | 存在官方文档、安装指引、平台差异、社区问题等多类证据 | 官方支持 `macOS / Linux / Windows`，并建议 Windows 优先 `WSL2` |
| `Gap` | 2 | 官方文档能回答“怎么装”，不能完整回答“哪种方式更适合谁” | 官方安装页更偏说明，不是决策树和比较矩阵 |
| `Monetization` | 3 | 变现路径非常清楚 | `Hosting / VPS affiliate`、托管 lead、部署咨询都存在 |
| `Updateability` | 2 | 可持续更新，但节奏低于价格/采购类题 | 版本更新、部署争议和安全讨论会带来更新空间 |

总分：`20 / 24`

红线项检查：

1. `Intent` 非 0
2. `Evidence` 非 0
3. `Gap` 非 0
4. `Monetization` 非 0

上限规则检查：

1. `Audience` 虽未满分，但已经有基本收敛，不触发压到 `Hold`
2. `Market Context` 虽未满分，但已有候选市场，不触发压到 `Hold`

总门禁结论：`Go`

### 3.3 内容机会评分

| 维度 | 分数 | 理由 | 说明 |
| --- | --- | --- | --- |
| `Query / Distribution Fit` | 3 | 部署、Windows、Docker、VPS、托管等 query 都较明确 | 易形成 query cluster 和页面归属 |
| `Angle Clarity` | 3 | 可拆成总对比、普通用户最佳方案、平台安装页、托管比较页 | 页面间可通过 thesis 与 slug 控制 cannibalization |
| `Evidence Depth` | 3 | 足以支撑矩阵、决策树、风险说明、步骤与误区 | 官方文档 + 社区痛点 + 平台差异足够形成深内容 |
| `Update Loop` | 2 | 有更新空间，但不像价格和采购决策那样高频 | 更适合阶段性更新 |
| `Content Gap` | 2 | 当前空位明确，但竞争并非完全空白 | 官方和教程站存在，但“选择建议”仍缺 |
| `Content Monetization Fit` | 3 | 内容页可直接承接 affiliate、lead、咨询 | 与用户意图高度匹配 |

总分：`16 / 18`

内容结论：`Content-Go`

### 3.4 产品机会评分

| 维度 | 分数 | 理由 | 说明 |
| --- | --- | --- | --- |
| `Repetition` | 1 | 部署选择不是高频重复动作 | 多数用户一旦完成部署，不会反复执行 |
| `Parameterizability` | 1 | 可做基础决策器，但输入参数有限 | 最多形成轻量选择器 |
| `Structured Output` | 1 | 可以输出推荐路径，但结果不算强结构化 | 更像建议型输出 |
| `Willingness To Pay` | 1 | 存在服务付费可能，但不强 | 更偏服务费，不是标准化订阅意愿 |
| `Subscription Fit` | 0 | 不适合长期订阅 | 用户完成部署后持续留存价值弱 |
| `Data / Workflow Moat` | 0 | 很难形成数据壁垒或工作流嵌入 | 易被复制与替代 |

总分：`4 / 18`

产品结论：`Product-No-Go`

### 3.5 决策结论

结论组合：

`Total Go + Content-Go + Product-No-Go`

第一阶段建议动作：

1. 明确进入 `report-web` 优先路线
2. 第一批优先方向维持：
   `OpenClaw deployment options`
   `best OpenClaw setup for beginners`
   `OpenClaw Windows install WSL2`
3. 不优先单独立项做工具或订阅产品

### 3.6 主要风险

1. 如果没有继续补 `Demand Inputs / Competition Inputs`，容易高估总方向价值
2. 如果页面归属不清，`deployment options`、`beginners`、`self-hosted vs managed` 会互相抢词
3. 如果商业来源处理不严，容易把 VPS 厂商页面当成最终结论依据

### 3.7 复盘重点

上线后应重点检查：

1. `OpenClaw deployment options`
   是否拿到顶层查询与内部链接中枢作用
2. `best OpenClaw setup for beginners`
   是否拿到更高质量的转化意图流量
3. `OpenClaw Windows install WSL2`
   是否拿到平台痛点流量

## 4. 案例二：AI 编程与 Agent 时代模型采购、路由、成本与地区选择

### 4.1 基础记录

| 字段 | 内容 |
| --- | --- |
| `Topic` | AI 编程与 Agent 时代模型采购、路由、成本与地区选择 |
| `Seed Question` | 用户想知道 AI coding / agent 场景下模型到底应该怎么买、是否要直连或通过中转、不同国家和语言下怎么做采购判断 |
| `Audience` | 个人开发者、小团队、技术负责人、采购角色 |
| `Geo / Language` | 候选市场暂定 `US/en`、`EU/en`、`IN/en`、`JP/ja`、`BR/pt-BR`、`CN/zh-CN` |
| `Assumptions` | 第一阶段仍以报告页验证为主，但该方向的产品潜力需要同步保留 |
| `Confidence Level` | `High` |
| `Actual Outcome Snapshot` | 尚未完成标准化上线后的流量与订阅验证，当前为上线前基线评分 |

### 4.2 总门禁评分

| 维度 | 分数 | 理由 | 关键证据 |
| --- | --- | --- | --- |
| `Problem` | 3 | 用户问题高度具体，集中在采购、路由、总拥有成本与地区选择 | 用户原始问题不是泛 AI 内容，而是高价值选择问题 |
| `Intent` | 3 | 这是典型采购与高价值决策意图 | 明显围绕“买哪家、怎么买、要不要通过中转” |
| `Audience` | 3 | 目标人群清晰且多角色可细分 | 个人开发者、小团队、采购、技术负责人都能清楚对应 |
| `Market Context` | 2 | 多市场语境明确存在，但仍需进一步收敛具体首发市场 | 不同国家、语言、支付和合规条件明显影响采购路径 |
| `Evidence` | 3 | 有官方定价页、官方地区可用性、平台页、社区讨论等多层证据 | 公开真源丰富，且可持续更新 |
| `Gap` | 3 | 官方页面能给价格，不能完整回答“如何采购与如何组合购买” | 采购判断与路由决策正是空位所在 |
| `Monetization` | 3 | 广告、咨询、工具订阅、数据订阅都存在 | 变现路径多且与意图高度匹配 |
| `Updateability` | 3 | 价格、路由策略、地区可用性、工具成本都会持续变化 | 天然适合更新日志与追踪 |

总分：`23 / 24`

红线项检查：

1. `Intent` 非 0
2. `Evidence` 非 0
3. `Gap` 非 0
4. `Monetization` 非 0

上限规则检查：

1. `Audience` 已高度收敛
2. `Market Context` 虽非满分，但已具备足够市场结构判断，不触发压到 `Hold`

总门禁结论：`Go`

### 4.3 内容机会评分

| 维度 | 分数 | 理由 | 说明 |
| --- | --- | --- | --- |
| `Query / Distribution Fit` | 3 | 采购、便宜但够用、直连 vs 中转、国家选择等 query 很明确 | 易形成 pillar + 决策页 + 国家页结构 |
| `Angle Clarity` | 3 | 可以稳定拆出主报告页、决策页、平台页、工具页、国家页 | 页面边界与 query ownership 都可设计 |
| `Evidence Depth` | 3 | 非常适合做矩阵、决策树、成本分解、证据附录 | 不只是写内容，而是能形成权威结构 |
| `Update Loop` | 3 | 天然适合持续更新 | 价格、地区、路由和产品策略变化频繁 |
| `Content Gap` | 2 | 竞争强，但“采购判断”这层仍然存在明确空位 | 大站和官方页多，仍需慎重切角度 |
| `Content Monetization Fit` | 3 | 内容页可直接承接广告、咨询、lead、订阅 | 与用户意图高度一致 |

总分：`17 / 18`

内容结论：`Content-Go`

### 4.4 产品机会评分

| 维度 | 分数 | 理由 | 说明 |
| --- | --- | --- | --- |
| `Repetition` | 3 | 团队会反复比较采购、路由、成本和地区差异 | 不是一次性主题 |
| `Parameterizability` | 3 | 预算、国家、语言、场景、工具栈都可以参数化 | 很适合比较器和决策器 |
| `Structured Output` | 3 | 可输出矩阵、推荐路径、追踪结果、比较结果 | 高结构化天然适合工具化 |
| `Willingness To Pay` | 2 | 省时、降本、减少误购有较强付费动机 | 尤其对小团队和采购角色明显 |
| `Subscription Fit` | 2 | 价格追踪、变化提醒、数据库查询可形成持续订阅 | 但需要先验证真实回访与留存 |
| `Data / Workflow Moat` | 2 | 长期积累后可形成一定数据壁垒和工作流嵌入 | 目前尚处早期，仍需真实验证 |

总分：`15 / 18`

产品结论：`Product-Go`

### 4.5 决策结论

结论组合：

`Total Go + Content-Go + Product-Go`

第一阶段建议动作：

1. 先走 `report-web` 验证路线
2. 第一批优先方向维持：
   `AI coding model procurement options`
   `best cheap model for coding agents`
   `direct API vs OpenRouter`
3. 同时把比较器、追踪器、采购决策工具保留为 `productOpportunityHints / expansionCandidates`

### 4.6 主要风险

1. 竞争显著高于 `OpenClaw`，需要更精细的角度切入与证据表达
2. 如果不做统一价格口径，页面很容易退化成静态价目表
3. 如果首发市场不收敛，容易把页面做成“全球大而泛总表”
4. 产品机会虽然高，但如果过早单独立工具，可能破坏第一阶段报告页验证边界

### 4.7 复盘重点

上线后应重点检查：

1. `AI coding model procurement options`
   是否能承担 pillar page 与总入口作用
2. `best cheap model for coding agents`
   是否拿到低成本强决策 query
3. `direct API vs OpenRouter`
   是否拿到高争议、高转化比较意图
4. 追踪或比较类模块
   是否真的出现回访、收藏、订阅或咨询信号

## 5. 两案例横向比较

### 5.1 相同点

1. 两个题都通过了总门禁，说明都值得进入第一阶段排期
2. 两个题的内容机会都很强，说明都适合先走 `report-web` 路线
3. 两个题都需要“窄而深”的页面切角，而不是大词泛写

### 5.2 不同点

1. `OpenClaw`
   更像“高意图内容题”，商业化主要依赖 affiliate、lead、咨询
2. `AI procurement`
   同时是“高意图内容题 + 高潜产品题”，后续更值得延展比较器、追踪器和订阅

### 5.3 当前最稳的排序判断

如果只从“第一阶段报告页验证”角度看：

1. `OpenClaw`
   更容易快速落地和验证内容变现路径
2. `AI procurement`
   更有上限，但竞争更高，对方法论要求更强

如果从“长期内容 + 产品一体化潜力”角度看：

1. `AI procurement`
   长期上限更高
2. `OpenClaw`
   更适合先拿内容验证，而不是优先发展工具产品

## 6. 当前结论

基于 `利基选题标准表 v1` 的第一次正式应用，可以先得出下面 4 个结论：

1. 这张标准表能够区分“内容强但产品弱”和“内容强且产品也强”
2. `OpenClaw` 适合当作第一阶段“报告页验证 + affiliate / lead”样例
3. `AI procurement` 适合当作第一阶段“报告页验证 + 产品潜力记录”样例
4. 下一步如果继续推进，最有价值的动作不是再空谈方法论，而是把这两份打分结果继续转成更细的 page-level 评分和上线优先级
