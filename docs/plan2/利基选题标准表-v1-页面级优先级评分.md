## 1. 文档目的

前面的标准表已经回答了：

1. 一个主题值不值得进入排期
2. 这个主题更偏内容机会还是产品机会
3. 第一阶段应该采取什么总体动作

但它还没有直接回答：

`一个主题下的具体页面，到底先发哪几页`

本文件就是补这一层。

它把：

`topic-level gate`

继续推进为：

`page-level launch priority`

## 2. 使用前提

这份页面级评分只适用于已经满足下面条件的主题：

1. 主题层总门禁不是 `No-Go`
2. 主题层内容机会至少达到 `Content-Go` 或接近 `Content-Go`
3. 团队已经拿到最基本的 query、证据和页面方向假设

如果一个主题在上一层仍是 `Total Hold + Content-Hold`，不建议先做页面级精排。

## 3. 页面级评分为什么单独做

同一个高价值主题下，不同页面的优先级可能差异很大。

例如：

1. 有的页面适合做总入口和 pillar
2. 有的页面适合承接高意图决策流量
3. 有的页面虽然能做，但更适合第二批或第三批
4. 有的页面理论上不错，但会和其它页面互相 cannibalize

所以页面级评分的目标不是“证明所有候选页都值得发”，而是：

1. 明确哪些页面先发
2. 明确哪些页面后发
3. 明确哪些页面暂时不发

## 4. 页面级评分规则

### 4.1 评分方式

页面级评分继续使用 `0-3` 四档，避免伪精确：

1. `0`
   当前不成立，或不适合上线
2. `1`
   有一定价值，但优先级较低
3. `2`
   基本成立，适合排入后续批次
4. `3`
   强成立，适合优先上线

总分满分 `18`。

### 4.2 评分维度

页面级评分采用 6 个维度：

| 维度 | 判断问题 |
| --- | --- |
| `Intent Match` | 这个页面是否承接了一个清晰而高价值的页面意图 |
| `Query Clarity` | 是否存在一组可归属、可防止 cannibalization 的核心 query |
| `Evidence Readiness` | 当前是否已经有足够证据支撑上线一页权威内容 |
| `Monetization Fit` | 这个页面是否能自然承接广告、affiliate、lead、咨询或订阅 |
| `Portfolio Role` | 这个页面在站点里是否承担清晰角色，例如 pillar、决策页、比较页、痛点页 |
| `Update Potential` | 这个页面是否适合长期更新，还是更像一次性页面 |

### 4.3 评分锚点

| 维度 | 0 | 1 | 2 | 3 |
| --- | --- | --- | --- | --- |
| `Intent Match` | 页面意图模糊或只是泛资讯 | 有意图，但价值较弱 | 意图明确，具备实际价值 | 高意图、高价值，明显可转化 |
| `Query Clarity` | query 边界模糊，易抢词 | query 存在，但归属不稳 | query 基本清晰 | query 很清晰，页面 ownership 明确 |
| `Evidence Readiness` | 当前证据不足以上线 | 只能写浅页 | 已能支撑较稳页面 | 足以支撑权威页、矩阵、决策树或附录 |
| `Monetization Fit` | 没有自然变现路径 | 变现路径很弱 | 至少有一条较清晰路径 | 变现路径明显且与页面意图高度匹配 |
| `Portfolio Role` | 页面角色不清晰 | 有角色，但可被替代 | 角色明确 | 是站点结构中的关键节点 |
| `Update Potential` | 基本不需要更新 | 更新价值较弱 | 有阶段性更新空间 | 天然适合长期更新与回访 |

### 4.4 页面级红线

下面 3 项作为页面级红线：

1. `Intent Match`
2. `Query Clarity`
3. `Evidence Readiness`

如果这 3 项里任意 2 项得 `0`，则该页默认进入：

`Page-No-Go`

### 4.5 页面优先级结论

| 结论 | 条件 | 含义 |
| --- | --- | --- |
| `P0` | 总分 `>= 14` 且无红线 | 第一批上线候选 |
| `P1` | 总分 `10-13` 且无红线 | 第二批上线候选 |
| `P2` | 总分 `6-9` 且无红线 | 可做，但不应优先 |
| `Page-No-Go` | 触发红线，或总分 `< 6` | 暂不进入页面排期 |

## 5. 批次设计原则

### 5.1 第一批页面的理想组合

对于大多数主题，第一批最稳的组合不是连发 3 篇同类型页面，而是：

1. `1 篇 pillar / 总入口页`
2. `1 篇强决策页或高转化页`
3. `1 篇高痛点页、争议页或明显场景页`

### 5.2 第二批页面的角色

第二批页面更适合承担：

1. 补内链结构
2. 承接更细分 query
3. 放大商业路径
4. 扩展垂直场景

### 5.3 第三批页面的角色

第三批页面更适合承担：

1. 长尾覆盖
2. FAQ 化补充
3. 风险/边界/附加说明页
4. 等待更多市场验证后再上的方向

### 6.2 为什么第一批不是直接发所有 `P0`

虽然这里有 6 个页面达到 `P0`，但第一阶段不建议一口气把它们全部上线。

原因是：

1. `P0` 代表“适合优先考虑”，不等于“必须同时发布”
2. 页面之间仍需避免互相抢词
3. 第一批更需要验证结构是否有效，而不是追求数量

### 6.3 第一批建议上线页

第一批建议只发这 3 页：

1. `OpenClaw deployment options`
   角色：`pillar / 总入口页`
2. `best OpenClaw setup for beginners`
   角色：`强决策页`
3. `OpenClaw Windows install WSL2`
   角色：`高痛点场景页`

推荐原因：

1. 三页组合覆盖“总览 + 决策 + 具体痛点”
2. 页面角色清晰，不容易高度重叠
3. 最容易同时验证内容权威感、query 承接和初步商业路径

### 6.4 第二批建议上线页

第二批建议：

1. `OpenClaw Docker setup`
2. `best VPS for OpenClaw`
3. `OpenClaw self-hosted vs managed`

这批页面的主要任务是：

1. 扩展更细的部署意图
2. 放大商业转化路径
3. 增强 pillar 页的内链结构

### 6.5 第三批建议上线页

第三批建议：

1. `OpenClaw install on Ubuntu`
2. `OpenClaw safe deployment`
3. `OpenClaw system requirements`
4. `OpenClaw install on macOS`

这批页面更适合作为：

1. 长尾补充
2. FAQ 与边界说明
3. 上线后根据真实 query 再决定最终顺序

### 6.6 OpenClaw 当前最稳判断

`OpenClaw` 的页面级优先级说明它最适合拿来做：

1. `第一阶段 report-web 内容验证`
2. `affiliate / lead 路径验证`
3. `页面结构与 query ownership 验证`

它不适合优先验证工具站订阅。

### 7.2 为什么第一批只发 3 页

这个主题虽然 `P0` 页面更多，但更不能一开始就铺太开。

原因有 4 个：

1. 竞争比 `OpenClaw` 高很多
2. 页面边界更容易互相重叠
3. 如果首发市场还没最终锁定，国家页和多语言页不应过早上线
4. `tracker` 和工具页虽然有价值，但更依赖后续数据积累

### 7.3 第一批建议上线页

第一批建议只发这 3 页：

1. `AI coding model procurement options`
   角色：`pillar / 总入口页`
2. `best cheap model for coding agents`
   角色：`强商业决策页`
3. `direct API vs OpenRouter`
   角色：`高争议比较页`

推荐原因：

1. 三页组合最能代表“采购判断”这个核心 thesis
2. 最容易同时承接顶层 query、低成本意图和强比较意图
3. 最适合先验证这个主题到底能不能建立权威感和转化

### 7.4 第二批建议上线页

第二批建议：

1. `hidden costs of ai coding models`
2. `single vs multi provider llm stack`
3. `OpenRouter vs Requesty vs Together API`
4. `best model for Cursor Cline Roo Code`

这批页面的任务是：

1. 补强中层决策结构
2. 强化比较矩阵与工具栈场景
3. 为后续产品化机会积累更细的使用场景数据

### 7.5 第三批建议上线页

第三批建议：

1. `llm pricing change tracker`
2. `best AI model provider in [country]`
3. `best multilingual model for coding teams`

原因是：

1. `tracker` 更依赖持续数据更新能力
2. `country pages` 需要先锁定真正首发市场
3. `multilingual` 很容易做得过泛，应该等真实 query 再细化

### 7.6 AI procurement 当前最稳判断

这个主题的页面级优先级说明它最适合拿来做：

1. `第一阶段 report-web 内容验证`
2. `高商业意图决策页验证`
3. `后续产品候选积累`

它既是强内容题，也是强潜在产品题。

## 8. 两主题页面级优先级对比

### 8.1 OpenClaw 更适合验证什么

`OpenClaw` 更适合验证：

1. `内容结构与 query ownership`
2. `affiliate / lead 路径`
3. `部署类主题的决策页是否能跑通`

### 8.2 AI procurement 更适合验证什么

`AI procurement` 更适合验证：

1. `采购判断型内容是否能建立权威感`
2. `高意图比较页是否能承接更强商业流量`
3. `内容页是否真能向工具、追踪、订阅延伸`

### 8.3 当前更稳的综合排期判断

如果从第一阶段整体排期看，当前更稳的顺序仍然是：

1. `OpenClaw`
   更适合先验证快速内容变现与内容结构
2. `AI procurement`
   更适合在方法论更稳后继续放大上限

但如果从长期上限看：

1. `AI procurement`
   长期内容 + 产品一体化上限更高
2. `OpenClaw`
   更像第一阶段验证型样例

## 9. 当前结论

到页面级这一层，当前可以先稳定地得出下面 5 个结论：

1. 两个主题都不该一次性铺开所有高分页
2. 第一批都应采用 `pillar + 决策页 + 痛点/争议页` 组合
3. `OpenClaw` 更偏内容验证
4. `AI procurement` 更偏内容验证 + 产品潜力积累
5. 下一步如果继续推进，最值钱的动作已经不是再抽象讨论，而是把第一批页面继续做成 `page-level ReportSpec / launch brief`
