# 北美利基内容采集、分析与内容变现路线

> 项目名：OpenFons
> 文档定位：`docs/plan2` 的阶段型规划文档
> 当前状态：在既有 source-native 平台规划上，进一步收口“采集 -> 证据 -> 分析 -> 内容 -> 变现”主线
> 适用范围：北美市场的 niche 利基主题研究、报告页交付、视频讲解与持续内容分发

> 2026-03-28 实验治理说明：本文中的平台 P0 / P1 / P2、Demand Validation / Evidence Feasibility 等规划判断，保留为北美内容路线的历史收口语境，不直接代表当前 dual experiment 的 winner 选择契约。若与当前实验冲突，统一以 hard-gate-first 和 artifact-bound evidence 为准。

## 1. 重新收口后的目标

本轮补充文档明确一件事：

OpenFons 的底层本质仍然是多来源采集与信息搜集系统，但平台价值不止于“抓到数据”，而是要把采集结果清洗为证据，再经过人工 + AI 的深度分析，最终生产为可分发、可复用、可变现的内容资产。

当前收口后的主线应表述为：

`北美 niche 主题发现 -> 意图澄清与需求验证 -> 核心搜索词 / 关键词簇 -> 多源采集 -> 证据清洗 -> 人工 + AI 深度分析 -> Next.js 报告页 -> 视频 / 文章 / 博客派生 -> SEO / YouTube / 其他收益`

## 2. 核心原则

### 2.1 采集仍是底层，不是终点

1. 平台底层仍以 crawler / adapter / browser automation / media download 为基础能力。
2. 采集层的职责是稳定地获得公开信息、媒体素材、评论、时间线和上下文。
3. 采集结果必须沉淀为可复核的证据，而不是直接跳到“自动生成结论”。

### 2.2 分析必须坚持人工 + AI 协同

1. AI 负责聚类、提炼、比对、生成候选结论和辅助写作。
2. 人工负责判断研究问题、修正误读、挑选证据、确定叙事和最终发布。
3. 任何面向 SEO 或 YouTube 变现的内容，都不应是无人工判断的大批量模板化输出。

### 2.3 第一优先交付物是网页型“投行式报告”

1. 最重要的内容产物不是普通博客，而是结构化、可索引、可复核、适合长期沉淀的报告页。
2. 报告页应具备类似投行 PPT 的叙事节奏，但交付形式优先选择 Next.js 网页，而不是静态 PDF。
3. 视频、文章、博客、newsletter、知识卡片等，属于从报告页进一步派生的内容层。

### 2.4 北美 niche 市场优先

1. 主题发现、关键词体系、平台优先级和变现逻辑都优先围绕北美市场设计。
2. 优先选择英文互联网中可持续产出利基主题、用户讨论和消费信号的平台。
3. 不以“覆盖所有平台”为初始目标，而以“找到高价值主题并稳定产出内容资产”为目标。

### 2.5 用户原话只是 seed，不是最终页面题目

1. 用户说出的需求是研究起点，不应直接拿来当页面标题、slug 或唯一主关键词。
2. 真正进入公开发布流程前，必须先判断这个主题是否适合做成可索引、可更新、可获得流量的页面。
3. 一个值得做成页面的主题，至少应同时满足：有明确用户问题、有持续搜索或讨论需求、有足够公开证据、能形成独立角度、后续可更新。
4. 如果需求本身更适合做内部简报、私有交付或一次性研究，就不应为了 SEO 勉强做成公开页面。

## 3. 端到端生产链路

### 3.1 主链路

建议将主链路明确为：

`niche discovery -> intent clarification -> demand validation -> keyword cluster -> source collection -> normalization -> evidence set -> human + AI analysis -> report spec -> Next.js report -> video/article derivatives -> distribution -> monetization`

### 3.2 分层职责

1. `Discovery`
   发现值得研究和发布的北美 niche 主题、产品、品牌、趋势、事件或讨论群体。
2. `Collection`
   从多个来源抓取帖子、评论、视频、商品评价、媒体附件和时间线。
3. `Normalization`
   将不同来源的原始结果映射到统一字段和统一证据模型。
4. `Evidence`
   建立主题级证据集，为后续分析、审计和内容引用提供真源。
5. `Analysis`
   由人工 + AI 对证据做聚类、对比、归因、评分、叙事和结论构建。
6. `Report Rendering`
   将分析结论渲染为 Next.js 报告页，作为主要公开交付。
7. `Derivative Content`
   从报告页反向生成文章、博客、视频脚本、讲解视频和社媒分发素材。
8. `Monetization`
   通过 SEO 流量、YouTube 播放量、广告、赞助、联盟、线索与高阶付费内容获取收益。

### 3.3 SEO 报告模式下的前置判断

当入口是自然语言问题，并且目标交付是可公开索引的 Next.js 报告页时，建议先做四步前置判断，再进入采集：

1. `Intent Clarification`
   先问清楚这是谁的问题、面向哪个国家和语言、是教程型、对比型、选型型、价格型还是风险型问题。
2. `Demand Validation`
   用 Google Trends、真实搜索结果页、社区讨论、媒体报道和视频平台讨论，判断这个主题是否存在持续需求，而不是只看一个词是否“热”。
3. `Keyword Cluster & Angle`
   从用户原话中衍生出核心搜索词、次级关键词、同义词和切入角度，决定页面究竟回答哪个问题。
4. `Evidence Feasibility`
   判断是否有足够公开证据支撑一篇独立页面。如果没有足够证据，就先沉淀为内部 brief，而不是急着公开发布。

### 3.4 以“OpenClaw 部署方案对比”为例

当用户只说一句“OpenClaw 部署方案对比”时，系统真正要确认的不只是“去抓 OpenClaw”，而是：

1. 目标受众是谁，是普通用户、开发者，还是企业技术团队。
2. 目标地区和语言是什么，是中文讨论，还是北美英文搜索流量。
3. 核心意图是哪一类，是“普通用户如何部署”“自托管 vs 托管”“最低成本方案”“最安全的部署方式”还是“VPS 选型”。
4. 页面最终要押注的不是原句本身，而是编译后的核心搜索词和页面 thesis。

只有这些问题明确后，后续的采集来源、证据结构、页面标题和报告结构才会稳定。

## 4. 平台优先级与专用采集建议

### 4.1 优先级收口

建议按以下顺序建设北美市场的来源能力：

| 优先级 | 平台 | 主要价值 | 推荐定位 |
| --- | --- | --- | --- |
| P0 | YouTube | 视频内容、评论、频道趋势、字幕、讲解素材 | 核心内容源 + 核心分发渠道 |
| P0 | Reddit | 利基社区讨论、痛点、观点冲突、评论树 | 核心研究源 |
| P0 | Hacker News | 科技与创业人群的高质量讨论 | 核心研究源 |
| P1 | Amazon | 商品评价、需求反馈、竞品对比、痛点语言 | 核心商业研究源 |
| P1 | X | 实时讨论、传播链、观点扩散 | 热点信号源 |
| P1 | TikTok | 爆款内容、短视频话题、评论反馈 | 趋势信号源 + 素材源 |
| P2 | Facebook | 公共主页、社区讨论、品牌反馈 | 补充研究源 |
| P2 | V2EX | 中文技术社区与跨境话题补充 | 补充源，不作为北美主源 |

| 跨平台媒体 | [`mikf/gallery-dl`](https://github.com/mikf/gallery-dl) | 图像、媒体附件 | 适合作为图像和媒体备份通道 |

### 4.3 重点判断

1. 最值得优先打透的是 `YouTube + Reddit + Hacker News + Amazon`。
2. `X` 和 `TikTok` 很重要，但应视为高波动、高风控成本来源。
3. `Facebook` 和 `V2EX` 更适合作为补充，而不是第一阶段核心来源。
4. 不是所有平台都要同深度建设，应先围绕最有内容变现价值的主题密集区建设。

### 4.4 当前验证快照（2026-03-25）

截至当前实验室验证，平台状态可以先粗分为四类：

1. 已有明确正向信号
   `YouTube` 的 `yt-dlp` 元数据抓取和评论抓取已经跑通；`Hacker News` 官方 API、`V2EX` JSON API 与 `amazon-review-scraper` 也已返回真实数据。
2. 可导入但需要更强前置条件
   `ytcog`、`praw`、`TikTokApi`、`twscrape` 都能安装，但真实抓取依赖 cookie、账号、会话、代理或令牌。
3. 当前环境下存在明显阻断
   `Reddit` 公开 JSON 端点重复请求返回 `403`，`PRAW` 无凭据返回 `401`，`TikTokApi` 创建公开用户会话超时。
4. 能导入但匿名有效性偏弱
   `facebook-scraper` 当前匿名公共页抓取返回的有效字段较弱，因此只能作为补充备选，不应高估。
5. 国内新媒体样板工程已验证可启动
   `MediaCrawler` 已在实验室完成 `uv sync`、浏览器安装、CLI 帮助、SQLite 初始化与 WebUI 健康检查，说明它适合作为国内平台采集架构参考；但真实抓取仍依赖登录态，且许可证不宜直接进入商业主线。
6. 浏览器桥接底座已出现明确正向信号
   `pinchtab` 官方 Docker 镜像已在本机健康运行，但默认带 token 鉴权与域名白名单；`camoufox` 已在宿主机真实启动并成功打开页面。

这组结果说明，第一阶段最重要的不只是“装上了哪些工具”，而是尽快区分哪些来源已经能稳定进入 `adapter -> evidence` 链路，哪些来源需要额外的账号与风控基础设施。

## 5. 多爬虫协调架构

### 5.1 不做单一万能爬虫

建议明确放弃“一个 crawler 统管所有平台”的思路，改为“多 crawler / 多 adapter / 多执行策略协同”。

### 5.2 建议的 worker 分工

1. `api-worker`
   适用于 Reddit、Hacker News、部分 Amazon、部分 YouTube 场景。
2. `demand-research-worker`
   适用于 Google Trends、搜索结果页、媒体与社区讨论汇总，用于判断搜索意图和页面切入角度。
3. `http-worker`
   适用于普通 HTML 页面、静态详情页、搜索结果页和列表页。
4. `browser-worker`
   适用于动态页面、登录态页面、瀑布流、滚动加载和 JS 渲染页面。
5. `media-worker`
   适用于视频、音频、字幕、缩略图和媒体附件下载。
6. `comment-worker`
   适用于评论树、回复层级、评论展开和情绪信号抓取。
7. `normalize-worker`
   适用于统一字段映射、清洗、去重、语言识别、实体提取。
8. `analysis-worker`
   适用于聚类、主题归纳、观点分布、证据打标签。
9. `artifact-worker`
   适用于报告页、文章、脚本、视频分镜和导出产物生成。
10. `cn-social-worker`
   适用于小红书、抖音、快手、B站、微博、贴吧、知乎等国内平台，建议借鉴 `MediaCrawler` 的登录态保存、平台拆分与 WebUI/CLI 结构，但实现保持自研。

### 5.3 调度策略

每个平台适配器都应遵守统一调度顺序：

1. `API-first`
   有稳定官方接口或成熟封装时优先走 API。
2. `Demand-first`
   对外公开报告页先判断需求强度、关键词簇和页面角度，再决定是否值得进入重采集。
3. `HTTP-second`
   当页面稳定且无需浏览器渲染时优先走轻量抓取。
4. `Browser-fallback`
   当页面强依赖 JS、登录态或滚动加载时再进入 Playwright。
5. `Anti-detect-fallback`
   只有在常规 Playwright 无法稳定获取结果时，才进入 Camoufox 等高成本兜底组件。

### 5.4 Camoufox 的定位

`Camoufox` 不应成为默认浏览器底座，而应定位为：

1. 高阻力页面的受控兜底能力
2. 只在特定 source、特定任务、特定代理策略下启用
3. 由平台层显式记录启用原因、成本、成功率与风险

推荐角色是：

`Playwright 默认执行 -> Pinchtab/Browser Bridge -> Camoufox 特种兜底`

而不是：

`所有浏览器任务都走 Camoufox`

## 6. 开源底座收敛建议

### 6.1 建议当前主线采用

| 层 | 建议采用 | 当前作用 |
| --- | --- | --- |
| 编排 | `Airflow` | 当前阶段的周期任务与确定性闭环编排 |
| 通用采集 SDK | [`apify/crawlee`](https://github.com/apify/crawlee) | 统一 HTTP / Browser / Queue 模式 |
| 浏览器自动化 | `Playwright` | 动态页面与受控自动化底座 |
| 视频下载 | `yt-dlp` | 多平台视频、字幕、媒体主下载器 |
| 队列与状态 | `Valkey` | 去重、节流、冷却、运行态计数 |
| 真源存储 | `PostgreSQL` | 任务、证据、产物和元数据真源 |
| 原始对象存储 | `SeaweedFS` | raw HTML、JSON、截图、媒体对象 |
| 站内搜索 / 检索 | `OpenSearch` | 后续报告与证据检索 |
| 报告站 | `Next.js` | 公开网页报告、SEO 落地页、内容分发载体 |

### 6.2 建议按需启用

| 层 | 按需组件 | 启用条件 |
| --- | --- | --- |
| 浏览器反检测 | `Camoufox` | 常规 browser worker 成功率不足时 |
| 浏览器桥接服务 | `Pinchtab` | 需要把浏览器能力以独立 Docker 服务方式暴露给 worker 或 agent 时 |
| 爬虫运维面 | `Crawlab` | worker 数量增多、运行态管理变复杂时 |
| 网页提取 | `Firecrawl` | 需要统一的 scrape/search/extract 接口时 |
| 长任务编排 | `Temporal` | 进入更复杂的人机协同和可回放流程时 |
| 图像/媒体补充 | `gallery-dl` | 多图片、多素材、多站点附件抓取时 |

### 6.3 不建议的做法

1. 不要在第一阶段同时接完所有平台。
2. 不要把 `Camoufox` 作为所有浏览器任务的统一入口。
3. 不要把参考仓库直接嵌进主仓充当“伪实现”。
4. 不要在证据模型还没稳定前就大规模铺内容生成。
5. 不要忽略 `pinchtab` 默认 token 鉴权和 allowedDomains 这类安全边界，避免把 Docker 化浏览器直接暴露成无保护的公共入口。

## 7. Next.js 报告站设计

### 7.1 为什么是 Next.js

Next.js 适合这条路线的原因在于：

1. 适合输出可索引的 HTML 页面，而不是只产出 PDF。
2. 适合静态生成、增量更新和主题化内容站。
3. 官方支持 `metadata`、`sitemap`、`robots` 等 SEO 基础能力。
4. 适合同时承载报告页、专题页、目录页、视频落地页和文章页。

### 7.2 报告站建议信息架构

建议在现有双后台之外，新增一个面向外部内容分发的 `report-web`：

```text
apps/
  control-web/
  ops-web/
  report-web/
```

`report-web` 建议承载以下路由：

1. `/reports/[slug]`
   单篇核心报告页
2. `/topics/[slug]`
   长期追踪主题页
3. `/markets/[slug]`
   某一垂直市场或 niche 聚合页
4. `/companies/[slug]`
   品牌、产品或竞品档案页
5. `/videos/[slug]`
   对应讲解视频落地页
6. `/sources/[platform]/[id]`
   可公开的证据引用页或来源说明页

### 7.3 “投行式报告页”建议结构

一篇主报告页不应只是博客长文，而应更接近结构化 deck：

1. 开场结论：一句话 thesis 和核心判断
2. Executive Summary：3 到 5 条最重要发现
3. Market Snapshot：市场规模、趋势、信号摘要
4. Competitive Landscape：竞品对比表、品牌定位图、价格带和功能带
5. Voice of Customer：来自评论、论坛、视频评论的观点聚类
6. Timeline / Momentum：时间线、热度变化、传播节点
7. Risks / Unknowns：结论边界、证据缺口和未证实假设
8. Evidence Appendix：关键来源、截图、链接、引用摘要
9. Update Log：版本、更新时间和主题持续跟踪记录

### 7.4 渲染链建议

报告页生成不应直接从 raw data 到最终页面，而应至少经过：

`EvidenceSet -> Analysis Brief -> Human Review -> ReportSpec -> Next.js Renderer`

这样才能保证：

1. 结论不是裸模型幻觉
2. 页面结构可复用
3. 视频、博客、newsletter 都能从同一份 `ReportSpec` 派生

## 8. 视频生产链

### 8.1 视频不是独立体系，而是报告页的派生产物

建议把视频生产明确为：

`ReportSpec -> VideoOutline -> Script -> Storyboard -> Voiceover / Editing -> YouTube Publish`

### 8.2 视频生产建议流程

1. 先完成网页报告页
2. 从报告页提取 5 到 10 分钟讲解结构
3. 生成脚本初稿、镜头卡和图表切换点
4. 由人工审阅论点、删去空话和模板化表述
5. 再进入旁白、配图、剪辑、字幕和发布

### 8.3 视频变现约束

1. 视频内容必须有清晰叙事和独立价值，不能只是网页朗读。
2. 必须加入人工选择、人工判断和人工编排，避免模板化批量生成。
3. 视频与网页之间应相互导流，但内容不应只是机械镜像。

## 9. SEO 与内容变现边界

### 9.1 SEO 原则

1. 公开站点要服务真实用户检索意图，而不是只做程序化收录。
2. 每篇报告页都应有独立主题、独立证据、独立结论和独立图表。
3. 不应批量生成只换关键词、不换观点的“伪研究页”。
4. 不应把用户原话直接当最终标题，必须先经过需求验证、关键词聚类和切入角度选择。
5. Google Trends 更适合用来判断相对热度、地域差异和话题变化，不等于绝对流量预估；正式发布后还应回到 Search Console 看真实曝光、点击和查询词。

### 9.2 YouTube 原则

1. 平台路线应避免“模板化、大批量、低差异”的 AI 视频。
2. 视频必须体现人工选题、证据筛选、观点组织和讲解价值。
3. 收益能力来自持续的高质量选题和讲解，而不是纯流水线堆量。

### 9.3 当前推荐的收益组合

1. Google SEO 流量
2. YouTube 播放和频道收益
3. 行业 newsletter / 博客导流
4. 高价值主题报告合集或会员化内容
5. 后续再考虑赞助、联盟、咨询线索和企业版能力

## 10. 工程路线建议

### 10.1 Phase 0：跑通最小闭环

1. 优先接入 `YouTube / Reddit / Hacker News / Amazon`
2. 先做 `intent clarification -> demand validation -> keyword cluster` 这一层
3. 跑通 `discovery -> collect -> normalize -> evidence -> report`
4. 先以人工驱动的视频制作流程验证内容产出质量
5. 先做少量高质量主题页，而不是大量低质量页面

### 10.2 Phase 1：形成稳定内容生产线

1. 加入 `X` 与 `TikTok` 的趋势信号采集
2. 建立主题库、报告模板库、图表模板库
3. 把 `ReportSpec` 扩展到文章、博客、newsletter
4. 开始做主题级长期 watch 和周更内容节奏

### 10.3 Phase 2：扩大来源和分发

1. 根据收益表现决定是否补 `Facebook` 和更多论坛源
2. 强化视频生产、频道矩阵和系列化栏目
3. 构建更成熟的 SEO 内链、专题页、聚合页和更新策略
4. 再评估是否需要更复杂的编排和运维体系

## 11. 当前最值得坚持的判断

1. OpenFons 不是“内容自动生成器”，而是“采集驱动的研究内容工厂”。
2. 最核心的资产不是单次文章，而是 `Topic / Evidence / ReportSpec` 这套可复用中间层。
3. 最强的壁垒不是单一爬虫，而是“北美 niche 主题选择 + 多源证据整合 + 人工 + AI 分析 + 报告与视频双重交付”。
4. 如果这条路线成立，OpenFons 不仅能做研究平台，也能做内容资产平台和收益型媒体基础设施。
