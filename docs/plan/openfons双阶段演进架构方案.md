# OpenFons 双阶段演进架构方案
> 文档定位：历史演进说明 / 决策背景
> 当前正式架构：[`../sot/开放源平台当前正式架构说明.md`](../sot/开放源平台当前正式架构说明.md)
> 状态：保留历史背景，不作为当前正式 SoT

## 1. 这份文档为什么保留
这份文档保留的目的，不是继续充当当前正式架构说明，而是记录 OpenFons 早期为什么会提出“双阶段演进”这条路线。

它回答的是：
1. 当时为什么会把平台拆成两个阶段来思考。
2. 当时为什么会考虑 Python、FastAPI、Next.js 等候选主线。
3. 后来哪些判断被保留，哪些判断被正式收口。

因此，本文应被当作历史演进与决策背景来阅读，而不是今天的正式 SoT。

## 2. 当时提出“双阶段演进”的背景
早期提出双阶段，核心是想同时覆盖两类目标：

1. 第一阶段：先验证外部流量、SEO 内容交付与最小变现闭环。
2. 第二阶段：在证据资产和采集能力沉淀后，再扩展到更深的分析、长期 watch、趋势判断与更复杂的研究交付。

这种拆分当时有其合理性：
1. 它避免一开始就把平台做成过重的全家桶。
2. 它允许先用更短的链路验证“是否值得继续投入”。
3. 它提醒团队区分“今天必须落地的最小闭环”和“未来可能演进的更大平台能力”。

这个阶段化思路本身是有价值的，到今天仍然保留。

## 3. 当时考虑过的候选路线
在早期设想中，我们曾认真考虑过以下候选路线：

1. 以 `Python + FastAPI + Next.js` 作为主工程主线。
2. 更早引入更大的分析平台、工作流引擎和统一平台化能力。
3. 更早把 SEO 报告生成、深度分析、模拟预测等目标放进一张大蓝图里统一表达。

这些候选路线在当时成立，主要因为：
1. Python 在采集、OCR、数据处理、模型接入生态上天然有吸引力。
2. Next.js 在面向公开页面与 SEO 场景上看起来顺手。
3. 统一大蓝图在产品叙事上更完整。

但这些候选路线不等于后来的正式落地路线。

## 4. 后来真正保留了什么
虽然具体技术路线发生了收口，但早期方案里有几条原则被保留下来，并继续影响当前主线：

1. 平台不是单纯抓取工具，而是 evidence-first 的研究与交付系统。
2. 必须把阶段方案和当前真相源分开，不能让历史方案冒充现状。
3. 必须优先跑通最小闭环，而不是提前铺满未来空壳。
4. 必须把正式契约、正式模型和正式执行边界立住，再谈更大规模扩展。
5. 必须保留从第一阶段走向第二阶段的演进空间，但不能把演进空间误写成当前实现。

换句话说，双阶段思路被保留了，旧主线本身没有被保留。

## 5. 后来正式收口了什么
经过后续讨论、工作树实现与 `repo-skeleton-v1` 合并验证，当前正式收口结果已经明确：

| 主题 | 早期候选假设 | 当前正式结论 |
| --- | --- | --- |
| 主工程语言 | Python 作为主工程主线 | TypeScript / Node.js 作为主工程主线 |
| 前端基线 | Next.js 候选主线 | React + Vite + TanStack Router |
| 第一批服务形态 | 更大范围平台化起步 | 先落 `control-web + control-api + report-web` 最小闭环 |
| 共享层 | 边做边扩展 | 先立 `contracts / domain-models / shared` |
| 基础设施 | 早期考虑更重方案 | 先用 PowerShell + Docker Compose 最小基线 |
| Python 角色 | 主线实现语言 | 辅助语言，仅在后续适配和局部 worker 场景中按需启用 |

这意味着：
1. 双阶段目标还在，但第一阶段的正式工程实现已经明确转向 TypeScript 主线。
2. 旧文档里出现的 Python/FastAPI/Next.js 主线，不应再被当作当前事实引用。

## 6. 今天应该如何理解“双阶段”
今天再看“双阶段”，更合适的理解方式是：

1. 第一阶段仍然是最小可运行、可验证、可交付的闭环建设期。
2. 第二阶段仍然代表长期分析、长期 watch、 richer worker、更多交付形态等演进方向。
3. 但第二阶段不再意味着可以把未来所有能力一次性写进当前正式架构里。
4. 第二阶段的具体形态，必须以第一阶段真实落地后的反馈为依据，而不是继续沿早期文档直接展开。

因此，双阶段今天是“演进框架”，不是“当前实现清单”。

当前文档与正式 SoT 的边界如下：

1. 本文记录历史背景、早期候选路线和收口过程。

3. 任何与当前开发、目录边界、技术主线、正式契约相关的判断，都应以 SoT 文档为准。
4. 任何试图理解“为什么会有双阶段概念、为什么早期讨论过另一条路线”的阅读，都应回到本文。

## 8. 当前结论
公允地看，双阶段演进这件事本身没有错，错的是让历史方案长期充当当前现状。

因此当前最合理的处理方式是：
1. 保留本文，作为历史演进说明。
2. 把当前正式架构单独收口到 SoT 文档。
3. 以后继续允许阶段性方案存在，但不再允许 `plan` 文档与 `sot` 文档混用。
