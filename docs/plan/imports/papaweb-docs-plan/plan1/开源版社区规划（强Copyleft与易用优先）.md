# 开源版社区规划（强Copyleft与易用优先）

> 版本：v1.2  
> 日期：2026-03-12  
> 适用：`papaweb` 开源社区版（可自部署）  
> 文档角色：产品化分轨规划（开源版）  
> 基线引用：`docs/plan/plan0/平台级采集与分析架构方案（开源可商用版）.md`
> 冲突裁决：与 SoT 不一致时，以 SoT 为准。

---

## 1. 定位与目标

开源版目标不是“功能最全”，而是“最快上手 + 易贡献 + 可运行”：

1. 让社区用户一键跑起三层采集主链路。  
2. 以可视化 UI 和较低学习成本换取更高采用率。  
3. 允许使用更强约束许可证组件（如 AGPL）来提升即用体验。  

---

## 2. 许可证策略（开源版）

1. 建议采用双许可证策略：社区版 `AGPL-3.0` + 商业版单独商业许可。  
2. 若采用双许可证，必须采用 `CLA`（或同等版权归集协议）；`DCO` 仅可作为补充签署流程。  
3. 允许引入 AGPL 组件作为默认体验栈（例如 Grafana、Metabase OSS）。  
4. 所有第三方组件必须在仓库内提供 `SBOM + License 清单`。  
5. 文档中明确“网络服务部署、二次分发、修改后发布源码”的义务说明。  

---

## 3. 统一编排架构（开源版）

```text
Dagster（社区编排/UI，易上手）
  -> Crawlab（执行网关）
    -> HTTP Worker / Browser Worker（Pinchtab）
      -> Valkey（默认，队列与去重）+ Kafka（按阈值启用）
        -> SeaweedFS（Raw）
        -> PostgreSQL（状态与元数据）
        -> ClickHouse（分析）
          -> dbt + Great Expectations（质量）
            -> Metabase/Grafana（社区版可视化）
```

说明：开源版默认编排器可使用 `Dagster`；平台收费版保持 `Airflow` 作为统一编排真源。  
若开源版实例需要对接平台版，统一通过任务模型与 API 契约对齐，不引入双编排真源。

执行约束（与 SoT 对齐）：
1. `PostgreSQL` 为任务状态真源，`ClickHouse` 仅作分析副本。  
2. `publish_worker` 是唯一发布入口，禁止脚本直写生产服务表。  
3. `Kafka` 默认为关闭，仅在吞吐/积压/多消费者场景达阈值时启用。

## 4. 组件清单（开源版）

| 层级 | 组件 | 许可证 | 说明 |
| --- | --- | --- | --- |
| 编排与主 UI | Dagster | Apache-2.0 | 交互体验好，适合社区快速上手 |
| 执行管理 | Crawlab | BSD-3-Clause | 与平台版保持执行层兼容 |

| 可视化 | Grafana OSS | AGPLv3 | 指标与告警可视化成熟 |

## 5. 三层采集逻辑落地（开源版）

与平台版一致，保持同构：

1. API 优先层。  
2. 网页抓取层。  
3. 种子兜底层。  

统一任务模型不变：

`crawl_job -> fetch_task -> normalize_task -> quality_task -> publish_task`

默认发布阈值（分源可覆盖）：
1. `coverage >= 0.85`  
2. `confidence >= 0.80`  
3. `freshness_hours <= 48`  

---

## 6. 与平台版的差异化边界

开源版包含：

1. 单租户默认部署模板。  
2. 基础策略配置、基础质量门禁。  
3. 社区可维护的插件接口与贡献指南。  

平台版独占（不在开源版默认提供）：

1. 多租户与细粒度权限隔离。  
2. 企业级审计、成本治理与 SLA 看板。  
3. 高级编排能力（跨区域容灾、灰度发布、商业连接器）。  

---

## 7. 90天落地路线

### 0-30 天

1. 发布开源版最小可运行包（Docker Compose）。  
2. 上线三层采集最小 DAG 和样例站点。  
3. 提供一键验收脚本与示例数据。  

### 31-60 天

1. 完成插件化接口（source adapter / parser adapter）。  
2. 开放社区贡献流程（Issue 模板、PR 规范、发布节奏）。  
3. 发布许可证与合规 FAQ。  

### 61-90 天

1. 达成“3个真实站点 + 1个API源”的稳定跑通案例。  
2. 建立社区版本发布节奏（月度小版本）。  
3. 与平台版形成同构升级路径（配置可迁移、任务模型不变，`publish_worker` 门禁一致）。  

---

## 8. 风险与控制

1. AGPL 组件接入前，必须明确部署与二次分发义务。  
2. 双许可证模式下必须执行 `CLA`（或同等版权归集协议），否则平台版商用权利链风险高。  
3. 开源版与平台版保持“同构不等权”，避免双线代码完全分叉。  
4. 严禁把历史脚本重新挂回生产编排主链路。
