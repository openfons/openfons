# 平台升级方案总览

> 日期：2026-03-19
> 范围：基于当前 `papaweb` 仓库真实代码状态整理
> 目标：把现有“最小采集闭环”升级为“用户可操作的后台平台”

## 1. 核心结论

当前仓库已经有平台底座，但还不是完整的平台后台。

现状更接近：

- 后端已有最小采集链路
- 已有任务、事件、来源站点、质量评分、发布结果等基础表
- 已有一个很薄的查询与触发 API
- 没有真正的统一前端入口
- 没有用户级任务中心
- 没有权限、审计、模板、调度、清洗规则配置这些平台层能力

所以后续升级方向应定义为：

`平台 = 前端骨架 + UI 组件 + 重表格组件 + 后端任务能力`

而不是只补几个页面，或只补几个触发接口。

## 3. 升级目标

第一目标不是重写现有 worker，而是把现有能力平台化，让用户可以：

- 在后台下达任务
- 查看运行过程
- 取消和重试任务
- 配置来源站点
- 配置三层策略
- 选择质量策略
- 后续逐步接入清洗规则配置
- 查看结果、证据、导出记录和审计日志

## 4. 关键判断

- 现有 `/api/v1/*` 只能算底层接口，不足以支撑平台后台
- 三层策略已经存在于 CLI 脚本，但还没有平台资源模型
- 清洗逻辑已经存在于 CLI 脚本，但当前不应直接改业务逻辑
- 前端尚未真正启动，因此这次升级本质是“补控制面”
- 后端最缺的是一级 `run` 对象，以及围绕 `run` 的控制与查询接口

## 5. 推荐总方向

- 前端骨架：`TanStack Router/Start`
- 后台 UI：`Ant Design`
- 重数据页面：`VTable`
- 平台正式接口：`/api/v2/*`
- 保留现有 `/api/v1/*` 作为兼容链路
- 三层策略从脚本定义迁移为平台配置
- 清洗规则后续平台化，但当前不改清洗实现
