# 实施阶段与验收标准

## 1. 总体实施顺序

本次升级建议按四个阶段推进，不建议一次性全铺开。

## 2. 第一阶段：任务中心最小闭环

目标：

- 让用户能正式下任务
- 让用户能看到 run
- 让用户能取消和重试

主要内容：

- 新增 `pipeline_runs`
- 新增 `runs / jobs / tasks / events / artifacts` API
- 保留 `/api/v1/pipeline/seed-run` 作为底层兼容入口
- 前端先做“新建任务页 + 运行中心页 + 运行详情页”

阶段结果：

- 平台第一次拥有真正的控制面

## 3. 第二阶段：配置中心

目标：

- 让用户能配置平台，而不是只会点触发

主要内容：

- `source-sites` 完整 CRUD
- `strategy-profiles`
- `task-templates`
- `schedules`
- `quality-policies`

阶段结果：

- 平台第一次具备“配置能力”

## 4. 第三阶段：清洗与实体治理

目标：

- 把 CLI 清洗能力逐步迁到平台
- 让前端用户配置清洗规则

主要内容：

- `clean_rule_sets`
- 清洗预览 API
- `clean / entity-resolve` 平台阶段
- 工具实体证据链

建议未来主链路升级为：

`seed/discovery -> fetch -> normalize -> quality -> clean/entity-resolve -> publish`

## 5. 第四阶段：平台治理能力

目标：

- 从“内部工具”升级为“正式平台”

主要内容：

- `auth`
- `rbac`
- `audit_logs`
- `user_preferences`
- `saved_views`
- `export_jobs`
- `system ops`

## 6. P0 最小闭环验收标准

P0 完成后，至少应满足：

- 用户可以在后台新建任务
- 用户可以查看 run 列表和 run 详情
- 用户可以取消 run
- 用户可以重试失败 task
- 用户可以查看任务事件和基础运行明细

如果这些能力还不具备，说明“任务中心最小闭环”还没有完成。

## 7. 平台 1.0 验收标准

平台 1.0 完成后，至少应满足：

- 用户可以维护来源站点
- 用户可以选择三层策略 profile
- 用户可以选择质量策略
- 用户可以看到工具结果和任务明细
- 用户可以导出结果
- 平台有最基本的角色权限
- 平台有用户审计日志

如果这些能力还不具备，就还不能称为“平台级后台”。

## 9. 实施原则

### 9.1 优先做控制面

不要在第一阶段就重写所有 worker。

现阶段更重要的是：

- 把已有能力收拢成平台入口
- 让用户可以操作
- 让用户可以看到过程和结果

### 9.2 先最小闭环，再扩能力

先把“下任务 -> 看运行 -> 查明细 -> 重试/取消”闭环跑通，再扩展模板、调度、清洗、权限。

### 9.3 兼容旧链路

新平台能力上线时，不要一上来就打断现有 `/api/v1/*` 与 worker 流程。

### 9.4 清洗逻辑先不改

当前已确认：

- 现有清洗逻辑先保留
- 后续把“配置能力”开放给前端用户

所以这一阶段先做规则集承载和预览能力，不直接重写业务逻辑。
