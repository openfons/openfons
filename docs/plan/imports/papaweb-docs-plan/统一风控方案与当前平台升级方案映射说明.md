## 6. 这说明当前文档路线发生了什么变化

严格说，当前文档不是旧方案的逐章续写，而是发生了一次路线重心切换：

### 6.1 旧路线更偏“先修地基”

核心排序是：

1. 冻结真源
2. 清目录语义
3. 重建 Topic / Evidence / Artifact
4. 再做控制面与交付层

### 6.2 新路线更偏“先补平台控制面”

核心排序是：

1. 建 `pipeline_runs`
2. 做 `/api/v2/*`
3. 做前端后台
4. 再补配置中心、清洗治理、权限与运维

### 6.3 两条路线的关系

新路线不是错的，但它默认了一个前提：

- 当前底座已足够稳定，可以先包一层平台控制面

而旧路线更谨慎，它认为：

- 如果不先把真源、目录边界、`gold_tool_entities` 定位、`source-sites` 口径理顺，平台控制面后面很容易搭在摇晃地基上

## 7. 当前真正“丢掉”的关键内容

如果只沿用现在的 `平台升级方案`，而不把旧文档里的关键信息补回来，最容易丢失的是下面这些内容：

1. `Topic / TopicRun / Evidence / EvidenceSet / Artifact` 作为长期研究平台核心模型
2. `TaskSpec / WorkflowSpec / ReportSpec` 作为编译层契约
3. `Source Registry` 单一真源及 `source-sites.v1.json` 的冻结语义
4. `gold_tool_entities` 只是兼容视图的正式定位
5. “先测试 discovery，再搬目录”的风险控制顺序
6. 运行时 / 资产区 / placeholder / 上游镜像的明确边界
7. 长期 watch、交付冻结、门户消费这些中长期能力的落点
8. “先修真源和迁移顺序，再做控制面”的总体约束

## 8. 现在应该怎么用这些文档

建议不要在“旧文档”和“新文档”之间二选一，而是按下面方式使用：

如果目标是继续推进当前仓库的前端后台、控制面、`/api/v2/*`、任务中心最小闭环，那么当前主文档应是：

- `docs/plan/平台升级方案/00-总览.md`

### 8.2 中长期架构约束，必须回看旧文档

如果目标是避免平台越做越乱，那么下面两份恢复出来的旧文档必须重新纳入参考：

- `docs/plan/统一风控对抗平台架构方案.md`

- 真源规则
- 证据模型
- 迁移顺序
- 目录语义
- discovery 测试基线

## 9. 推荐下一步

最稳妥的做法不是把旧文档重新当成唯一主线，而是做两步：

1. 先把恢复出来的两份文档正式重新纳入仓库，至少保留为可追溯参考。
2. 再开一个单独 docs PR，把旧文档里真正不能丢的内容并入当前 `平台升级方案`：
   - 真源规则
   - `gold_tool_entities` 降级定位
   - `Topic / Evidence / Artifact` 模型
   - `worker-discovery` 测试优先级
   - 运行时与资产区边界

这样做的好处是：

- 不会打断当前平台控制面的推进
- 也不会把旧文档里最重要的架构约束继续丢掉
