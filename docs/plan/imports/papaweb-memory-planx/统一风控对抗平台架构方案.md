# 统一风控对抗平台总架构书

> 面向 IM / Web / API 的长期研究、监控、报告与情报消费平台

## 1. 文档定位

这份文档是 `papaweb` 的总架构书，不是讨论纪要，也不是某一阶段的任务清单。

### 1.1 关联实现文档

- [统一风控对抗平台 Phase 0 实现级设计](plan0/统一风控对抗平台-Phase0实现级设计.md)
  负责把本书中的 `Phase 0` 进一步拆成可执行范围、目录改造、测试基线和 PR 顺序

它只回答 5 个问题：

1. 我们到底要做什么系统
2. 当前仓库真实处于什么位置
3. 目标系统的核心对象和核心边界是什么
4. 哪些是必须坚持的架构约束
5. 应该按什么顺序落地

这份文档不负责：

- 逐字段 API 细节
- 单个 Worker 的实现说明
- 单个平台适配器的抓取策略细节
- 外部参考项目的评测记录

上述内容应拆到实现级设计文档、ADR、Runbook 或专项方案中。

## 2. 北极星目标

`papaweb` 的目标不是“做一个更强的爬虫脚本集合”，而是构建一个：

- 能把自然语言需求编译为结构化任务的控制层
- 能稳定调用多种采集执行能力的执行层
- 能沉淀证据、主题、历史交付的证据层
- 能产出 IM / CSV / JSON / PDF / Markdown / Dashboard 的交付层
- 能支撑长期 watch、周期更新、组织内部持续消费的研究平台

一句话概括：

`自然语言输入 -> 结构化任务 -> 确定性执行 -> 证据沉淀 -> 多形态交付 -> 长期消费`

## 3. 愿景是否合理

这份愿景本身是合理的，而且值得作为长期目标继续推进。

合理的原因：

- 当前仓库已经有一条稳定的确定性 Worker 主链路，说明“模型负责理解，Worker 负责执行”的方向是成立的
- 当前仓库已经出现源站配置、发现规则、Pinchtab fallback、字段级 evidence 雏形，说明平台化不是凭空设想
- 当前业务诉求已经明显超出“抓公开网页”本身，正在走向“长期研究、可交付、可追溯、可运营”

但必须明确：

- 这是目标态，不是现状描述
- 当前仓库还没有真正的 `Topic / Evidence / Artifact` 真源
- 当前仓库还不具备真正的多租户、长期 watch、历史交付冻结能力

因此，这份架构书必须采用“三层写法”：

- `现状层`：今天已经有什么
- `目标层`：最终要变成什么
- `迁移层`：从现状如何过渡到目标

如果不这样写，文档会看起来“很完整”，但团队落地时会不知道哪些已经存在、哪些只是愿景。

## 4. 非目标

为了避免方向漂移，先明确不做什么：

- 不做“大模型直接驱动浏览器”的执行架构
- 不做“一个万能超级 Agent”统管所有任务
- 不做“单一爬虫引擎一统天下”的技术路线
- 不把报告模板反向当作抓取逻辑来源
- 不把 `gold_tool_entities` 继续硬撑成未来的证据真源
- 不把上游镜像目录、占位应用目录误当作正式运行时

## 5. 当前仓库基线

### 5.1 当前真实主链路

当前仓库真实主链路应描述为：

```text
query-api
  -> worker-discovery（当 sourceSites 配置存在时）
  -> worker-fetch-http
  -> worker-normalize
  -> worker-quality
  -> publish_worker
```

这条链路已经具备：

- 运行批次创建与查询
- sitemap / start_urls / discovery_patterns 驱动的发现
- HTTP 抓取与 Pinchtab fallback
- 标题、摘要、canonical URL 标准化
- 质量评分与质量门禁
- 结果发布与 CSV 导出

### 5.2 当前已经存在但容易被忽略的事实

当前仓库并不是从零开始，至少已经存在下面几类“隐式前身”：

- `apps/crawler/config/source-sites.v1.json`
  已承担部分 Source Registry 职责
- `worker-discovery`
  已承担部分 source discovery 和 route preparation 职责
- `apps/crawler` 原始候选中的 `evidence`
  已体现“证据项”雏形
- `apps/crawler`
  当前更像迁移期 CLI 采集工具与证据前身区，不是当前平台主链路服务
- `packages/shared/src/platform-fetch.mjs`
  已体现多通道 fetch 和 fallback 逻辑

### 5.3 当前最重要的限制

当前架构还受以下限制：

- 发布真源仍然是 `gold_tool_entities`
  它更像“最新结果视图”，不是历史冻结交付物
- 运行态仍然主要围绕 `source_id / canonical_url / run_id`
  不足以支撑真正的 `topic / topic_run / tenant / ACL`
- `apps/web`、`apps/api` 仍是 placeholder
  不能被当作现成产品前后端
- `worker-fetch-browser`、`web-console`
  当前更像上游资产区，不是成熟运行时

## 6. 核心架构判断

### 6.1 系统应分成四个平面

总架构不建议继续按“很多层抽象名词”堆叠，而应按四个平面来写：

1. `Access Plane`
   IM / Web / API / 外部系统接入
2. `Control Plane`
   Intent Router、Task Compiler、Policy Gate、Scheduler
3. `Execution Plane`
   Discovery、HTTP、Browser、Media、Adapter、风控对抗
4. `Data & Delivery Plane`
   Topic、Evidence、Artifact、Renderer、Portal

这样写的好处是：

- 团队容易定位一个模块属于哪一层职责
- 不会把“控制逻辑”“抓取逻辑”“数据真源”“展示层”混写
- 更适合从当前仓库分阶段迁移

### 6.2 大模型只负责编译，不负责执行

大模型的职责是：

- 理解用户问题
- 识别意图
- 选择 Agent Profile
- 生成 `TaskSpec / WorkflowSpec / ReportSpec`

大模型不负责：

- 驱动浏览器
- 直接下发页面操作
- 代替 Worker 做抓取细节决策

这条约束必须写死，否则后面所有边界都会糊掉。

### 6.3 未来系统必须以 Topic / Evidence / Artifact 为核心

这是整份架构书最关键的一条。

未来平台的核心对象不应再是“某条抓取结果”，而应是：

- `Topic`
  用户真正关心的研究对象
- `TopicRun`
  某次围绕 Topic 的执行批次
- `Evidence`
  可被引用、可被审计、可被回跳的证据项
- `EvidenceSet`
  某次交付采用的证据集合
- `Artifact`
  一次被正式生成并冻结的交付物

只要这条不成立，后面的：

- PDF
- CSV
- Watch
- Portal
- ACL
- 历史复现

都只会停在概念层。

### 6.4 当前 `gold_tool_entities` 的正确定位

在迁移期内，`gold_tool_entities` 的正确定位应该是：

- 最新结果视图
- 兼容导出视图
- 旧链路对外查询视图

它不应再被描述为：

- 最终证据真源
- 长期 watch 真源
- 报告冻结真源
- 多租户 Artifact Store

### 6.5 Source Registry 必须只有一个真源

未来一定要有 `source-registry`，但不能新造一套再和现有配置并行。

正确做法是：

- 先收编现有 `source-sites.v1.json + worker-discovery`
- 再逐步抽成正式 `source-registry`
- 最终只保留一套源站真源

否则后面一定会出现：

- 同一个站点两套配置
- 不同 Worker 解释口径不同
- 调试和回放困难

## 7. 目标态总架构

```text
Access Plane
  IM / Web / API / External Tools
        |
        v
Control Plane
  Intent Router
  Agent Profile Selector
  Task Compiler
  Policy Gate
  Scheduler / Watch Planner
        |
        v
Execution Plane
  Source Registry
  worker-discovery
  worker-fetch-http
  worker-fetch-browser-crawlee
  worker-media-fetch
  Platform Adapters
  Pinchtab / Camoufox / MediaCrawler bridge
        |
        v
Data & Delivery Plane
  RawCapture
  NormalizedItem
  Topic
  TopicRun
  Evidence
  EvidenceSet
  Artifact
  Template Center
  Document IR
  Renderers
  query-api / report-api / visualization-api / intel-portal
```

其中：

- `风控对抗` 是 Execution Plane 的一个核心子域
- `模板、报告、门户` 建立在 Evidence / Artifact 真源之上
- `长期 watch` 建立在 Topic / TopicRun 真源之上

## 8. 核心对象模型

### 8.1 TaskSpec

`TaskSpec` 描述“这次任务是什么”。

最少必须包含：

- `taskId`
- `runId`
- `tenantId / orgId`
- `intent`
- `profile`
- `topic`
- `mode`
- `sources`
- `operations`
- `schedule`
- `idempotencyKey`
- `compilerVersion`

### 8.2 WorkflowSpec

`WorkflowSpec` 描述“这次任务怎么跑”。

最少必须包含：

- `policyDecisionId`
- `policyVersion`
- `sourceRegistryVersion`
- `sourceRouting`
- `fetchPlan`
- `accessControl`
- `executionSnapshot`

### 8.3 ReportSpec

`ReportSpec` 描述“这次任务怎么交付”。

最少必须包含：

- `templateMode`
- `templateId`
- `templateVersionId`
- `templateScope`
- `topicRunId`
- `evidenceSetId`
- `artifacts`
- `artifactAcl`
- `documentIrVersion`

### 8.4 Topic / TopicRun

`Topic` 负责长期研究对象本身。

至少应包含：

- `topicId`
- `tenantId / orgId`
- `subject`
- `aliases`
- `scope`
- `visibility`
- `owner`

`TopicRun` 负责一次围绕 Topic 的执行批次。

至少应包含：

- `topicRunId`
- `topicId`
- `runId`
- `mode`
- `startedAt / endedAt`
- `status`
- `policyVersion`
- `sourceRegistryVersion`

### 8.5 Evidence / EvidenceSet / Artifact

`Evidence` 是最小可引用单位。

至少应包含：

- `evidenceId`
- `tenantId / orgId`
- `topicId / topicRunId`
- `platform`
- `itemType`
- `url`
- `capturedAt`
- `contentSummary`
- `rawRef`
- `sourceMeta`

`EvidenceSet` 是一次交付采用的证据集合。

至少应包含：

- `evidenceSetId`
- `topicRunId`
- `selectionRule`
- `evidenceIds`
- `createdAt`

`Artifact` 是正式冻结的交付物。

至少应包含：

- `artifactId`
- `topicRunId`
- `evidenceSetId`
- `templateVersionId`
- `documentIrHash`
- `rendererName`
- `rendererVersion`
- `generatedAt`
- `acl`

## 9. 真源规则

为了防止系统越做越乱，必须写清楚真源：

- `Source Registry`
  是源站与采集通道真源
- `TaskSpec / WorkflowSpec / ReportSpec`
  是任务编译真源
- `Topic / TopicRun / Evidence / Artifact`
  是研究、证据与交付真源
- `gold_tool_entities`
  只是迁移期兼容视图，不是最终真源

## 10. 代码与目录原则

如果允许较大重构，建议未来按以下原则整理仓库：

### 10.1 运行时代码

- `services/`
  所有正式运行的后端服务与 Worker
- `apps/`
  正式前端应用
- `packages/`
  契约、共享库、策略库、适配器库

### 10.2 资产区

把所有上游镜像、实验快照、外部参考代码统一搬到单独资产区，例如：

- `third_party/`
- `upstreams/`
- `labs/`

不要继续把上游镜像挂在正式服务路径下面。

原因很简单：

- 否则团队会误把资产区当运行时
- 文档会持续失真
- CI、测试、依赖管理都会混乱

## 11. 实施顺序

如果你们不怕重构，正确顺序不是先做门户，也不是先做 PDF，而是：

### Phase 0：冻结现状与拆分资产区

- 冻结当前运行链路和真源说明
- 把上游镜像、placeholder、运行时代码彻底分开
- 补 `worker-discovery` 与路由测试基线

### Phase 1：重建核心数据模型

- 建立 `Topic / TopicRun / Evidence / EvidenceSet / Artifact`
- 明确 `gold_tool_entities` 的兼容定位
- 设计从旧链路到新模型的迁移关系

### Phase 2：重建 Source Registry

- 收编现有 `source-sites.v1.json`
- 统一 discovery、routing、fetch policy 语义
- 消灭双轨源站配置

### Phase 3：补 Control Plane

- Intent Router
- Task Compiler
- Policy Gate
- Agent Profile Selector

### Phase 4：补多引擎执行与 watch

- HTTP
- Browser
- Media
- Adapter
- Watch Job
- Incremental run

### Phase 5：补交付层

- Template Center
- Document IR
- IM / CSV / JSON Renderer
- PDF / Markdown Renderer

### Phase 6：补门户与运营层

- report-api
- visualization-api
- intel-portal
- Dashboard / Watch Center / Evidence Explorer

## 12. 这份架构书的验收标准

一份合格的总架构书，不是“概念很多”，而是要满足以下标准：

- 任何人看完都能区分现状、目标态、迁移态
- 任何核心模块都能归到四个平面之一
- 任何新服务都能回答它的真源依赖是什么
- 任何交付物都能回答它依赖哪个 `EvidenceSet`
- 任何长期任务都能回答它绑定哪个 `Topic`
- 任何团队成员都不会再把资产区误当成运行时

## 13. 当前阶段的结论

对 `papaweb` 来说，最应该写进总架构书的，不是“我们将来想支持多少花哨能力”，而是以下三件事：

1. 我们要从“抓取结果平台”升级为“Topic / Evidence / Artifact 平台”
2. 我们要把“自然语言输入”稳定编译成结构化任务，而不是让模型直接控制执行
3. 我们要先修数据真源和迁移顺序，再做报告中心和情报门户

这三件事写清楚了，后面的实现级设计、服务拆分、数据库设计、前端门户设计才有真正的落点。
