## 术语表

| 术语 | 英文 | 说明 |
|------|------|------|
| **主题** | Topic | 用户定义的研究主题，如"某品牌舆情监控" |
| **运行** | TopicRun | 主题的一次具体执行，包含采集时间、策略版本等信息 |
| **证据** | Evidence | 采集到的单条内容及其分析结果，平台最小真源单元 |
| **证据集** | EvidenceSet | 经过筛选的证据集合，用于生成报告 |
| **交付物** | Artifact | 最终生成的报告、图表、数据导出等成果 |
| **方面级情感** | Aspect-Based Sentiment | 针对特定方面（如质量、价格）的情感判断 |
| **Agent** | Agent | 具有自主决策能力的智能体，可分解任务、调用工具 |
| **Worker** | Worker | 执行具体任务的确定性组件，输入输出明确 |
| **Skill** | Skill | 可复用的能力模块，如"微博采集技能"、"情感分析技能" |
| **Workflow** | Workflow | 由多个步骤组成的工作流程，可编排、可回放 |

## 1. 项目背景与目标

### 1.1 背景

当前互联网信息分散在多个平台（社交媒体、电商平台、新闻站点等），用户需要一种能力：
- **统一采集**：从多个平台自动采集相关信息
- **智能整理**：利用 AI 对海量信息进行摘要、分类、情感分析
- **按需生成**：根据用户需求生成报告、图表、数据导出等多种交付物

**参考项目**：
- **Google NotebookLM**：Source-Process-Generate 工作流（https://notebooklm.google.com）
- **BettaFish**：多平台采集与情感分析（https://github.com/666ghj/BettaFish）
- **DeerFlow（字节）**：Multi-Agent 协作与 Skill 系统设计（https://github.com/bytedance/deer-flow）

### 1.2 核心目标

构建一个"全平台数据采集 + AI 智能整理 + 按需内容生成"的研究平台：

```
用户输入主题/需求 → AI 分解任务 → 并行采集分析 → 生成定制化交付物
```

**核心创新**：融合 Multi-Agent 的灵活性和 Worker 的确定性，既保证智能又保证可控。

### 1.3 典型应用场景

| 场景 | 采集源 | 分析维度 | 交付物 |
|------|--------|----------|--------|
| 品牌舆情监控 | 微博、小红书、抖音、新闻 | 情感趋势、热点话题、传播路径 | 舆情日报、危机预警 |
| 商品评价分析 | 淘宝、京东、亚马逊 | 方面级情感（质量/价格/服务） | 评价分析报告、改进建议 |
| 竞品情报研究 | 多平台综合 | 产品特性、用户反馈、市场定位 | 竞品对比报告 |
| 热点事件追踪 | 社交媒体 + 新闻 | 时间线、关键节点、舆论走向 | 事件复盘报告 |

## 2. 架构设计哲学

### 2.1 为什么融合 Multi-Agent 和 Worker？

| 架构模式 | 优点 | 缺点 | 适用场景 |
|----------|------|------|----------|
| **纯 Multi-Agent** | 灵活、智能、自适应 | 不可预测、难审计、成本高 | 探索性研究 |
| **纯 Worker 流水线** | 确定性、可审计、低成本 | 僵化、无法处理复杂逻辑 | 固定流程任务 |
| **融合架构（我们）** | 灵活 + 确定性 | 设计复杂度高 | 生产级平台 |

**我们的设计原则**：

```
┌─────────────────────────────────────────────────────────────┐
│  上层：Multi-Agent（灵活智能）                               │
│  ├── 任务理解、分解、决策                                     │
│  └── 处理不确定性、异常情况                                   │
└─────────────────────────────────────────────────────────────┘
                              ↓ 生成确定性计划
┌─────────────────────────────────────────────────────────────┐
│  下层：Worker + Workflow（确定性执行）                       │
│  ├── 按既定计划执行                                          │
│  ├── 每步可审计、可回放                                       │
│  └── 失败可重试、可追踪                                       │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 核心设计原则

| 原则 | 说明 | 价值 |
|------|------|------|
| **Agent 决策 + Worker 执行** | Agent 负责任务分解和决策，Worker 负责确定性执行 | 既有灵活性，又有可控性 |
| **Skill 化设计** | 将能力封装为可复用的 Skill | 模块化、可扩展、易维护 |
| **真源驱动** | Topic/Evidence/Artifact 三级真源模型 | 数据可追溯、可审计、可复现 |
| **Workflow 显性化** | 所有执行流程显性化为可编排的 Workflow | 可审计、可回放、可优化 |

## 3. 核心数据模型

### 3.1 三级真源模型

```
Topic（研究主题）
  └── TopicRun（一次采集运行）
        └── Evidence（证据项） ← 最小真源单元
              ├── 原始内容（文本/媒体/元数据）
              ├── AI 分析结果（情感/主题/实体）
              └── 采集上下文（时间、平台、策略版本）
                    └── EvidenceSet（证据集合）
                          └── Artifact（交付物）
```

### 3.2 Evidence 模型（核心）

Evidence 是平台的最小真源单元，包含：

```typescript
interface Evidence {
  // 基础标识
  evidenceId: string;
  topicId: string;
  topicRunId: string;
  
  // 来源信息
  platform: 'weibo' | 'xiaohongshu' | 'douyin' | 'taobao' | 'jd' | 'news';
  itemType: 'post' | 'video' | 'comment' | 'product_review';
  url: string;
  capturedAt: Date;
  
  // 原始内容
  rawContent: {
    title?: string;
    body: string;
    author: string;
    publishTime: Date;
    mediaUrls?: string[];  // 图片/视频
    interactions: {
      likes: number;
      comments: number;
      shares: number;
    };
    // 评论列表（如果是帖子/视频）
    comments?: Comment[];
  };
  
  // AI 分析结果
  analysis: {
    // 情感分析（核心）
    sentiment: {
      postLevel: SentimentResult;           // 帖子级情感
      commentLevel?: SentimentResult[];     // 评论级情感
      aspectLevel?: AspectSentiment[];      // 方面级情感（商品评价）
      stats: SentimentStats;                // 情感统计
    };
    
    // 主题提取
    topics: string[];
    
    // 实体识别
    entities: {
      name: string;
      type: 'person' | 'brand' | 'product' | 'location';
    }[];
    
    // 分析元数据
    modelVersion: string;
    analyzedAt: Date;
  };
  
  // 质量评分
  qualityScore: number;
  
  // 审计字段
  workflowId: string;      // 生成该 Evidence 的工作流实例
  policyVersion: string;   // 策略版本
}
```

### 3.3 情感分析模型设计

针对商品评价、帖子评论等场景，设计多层级情感分析：

```typescript
interface SentimentResult {
  score: number;        // -1.0 ~ 1.0，负面到正面
  label: 'positive' | 'neutral' | 'negative';
  confidence: number;   // 0.0 ~ 1.0
  model: string;        // 使用的模型标识
}

interface AspectSentiment {
  aspect: string;       // 方面名称：质量、价格、服务、物流等
  sentiment: SentimentResult;
  evidence: string;     // 支撑该判断的原文片段
}

interface SentimentStats {
  totalComments: number;
  positiveCount: number;
  neutralCount: number;
  negativeCount: number;
  positiveRatio: number;
  aspectDistribution: {  // 各方面情感分布
    aspect: string;
    positiveRatio: number;
  }[];
}
```

**情感分析模型选型**：

| 模型 | 适用场景 | 特点 |
|------|----------|------|
| BERT-LoRA 微调 | 中文社交媒体 | 针对微博、小红书等平台微调 |
| Qwen3 小参数 | 通用中文内容 | 轻量、快速 |
| 多语言模型 | 跨境平台 | 支持英文评价分析 |
| 方面级情感模型 | 商品评价 | 识别质量/价格/服务等多维度情感 |

## 4. 系统架构

### 4.1 四大平面架构

```
┌─────────────────────────────────────────────────────────────────┐
│                        Access Plane                              │
│  用户接入层：IM / Web / API / 外部系统集成                        │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│                      Control Plane                               │
│  控制层：意图理解、任务分解、策略决策、工作流编排                  │
│  （Multi-Agent 协作，生成确定性 Workflow）                        │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│                     Execution Plane                              │
│  执行层：Skill 调用、工作流执行、智能分析、质量管控、反爬对抗      │
│  （确定性执行，可审计、可回放，内置反爬对抗能力）                  │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│                   Data & Delivery Plane                          │
│  数据与交付层：主题管理、证据沉淀、报告生成、多形态交付            │
└─────────────────────────────────────────────────────────────────┘
```

**反爬对抗在四大平面中的位置**：
- **Control Plane**：制定反爬策略（频率、代理、浏览器选择）
- **Execution Plane**：执行反爬措施（代理轮换、浏览器指纹伪装、行为模拟）
- **Data Plane**：记录反爬日志，分析封禁模式，优化策略

### 4.2 Control Plane：Multi-Agent 协作层

**核心职责**：理解用户意图，分解任务，生成确定性 Workflow

```
└─────────────────────────────────────────────────────────────┘
    ↓
┌─────────────────────────────────────────────────────────────┐
│  Strategy Planning Agent（策略规划代理）                     │
│  ├── 采集策略                                                │
│  │   ├── 平台：tmall, jd                                     │
│  │   ├── 关键词：['iPhone 16', 'iPhone16']                   │
│  │   ├── 采集深度：最近 1000 条评价                           │
│  │   └── 反爬策略：频率控制、代理轮换                         │
│  ├── 分析策略                                                │
│  │   ├── 情感分析：启用方面级                                 │
│  │   ├── 方面定义：['质量', '价格', '服务', '物流']           │
│  │   └── 主题提取：识别用户关注焦点                           │
│  └── 交付策略                                                │
│      ├── 报告：评价分析报告                                   │
│      ├── 图表：情感分布图、方面级对比图                        │
│      └── 导出：原始评价 CSV                                   │
└─────────────────────────────────────────────────────────────┘
    ↓
┌─────────────────────────────────────────────────────────────┐
│  Policy Gate（策略门）                                       │
│  ├── 评估合规性：是否符合平台规则                             │
│  ├── 评估成本：预计 token 消耗、存储成本                      │
│  └── 评估风险：反爬风险、数据敏感度                           │
└─────────────────────────────────────────────────────────────┘
    ↓
```

**Skill 接口定义**：

```typescript
interface Skill {
  name: string;
  version: string;
  inputs: InputSchema[];
  outputs: OutputSchema[];
  
  // 执行 Skill
  execute(context: SkillContext): Promise<SkillOutput>;
  
  // 验证输入
  validate(inputs: any): ValidationResult;
  
  // 获取成本估算
  estimateCost(inputs: any): CostEstimate;
}

// Skill 示例：微博采集
const WeiboCollectionSkill: Skill = {
  name: 'weibo-collection',
  version: '1.0.0',
  inputs: [
    { name: 'keywords', type: 'string[]', required: true },
    { name: 'timeRange', type: 'dateRange', required: true },
    { name: 'limit', type: 'number', default: 100 }
  ],
  outputs: [
    { name: 'posts', type: 'Post[]' },
    { name: 'metadata', type: 'CollectionMetadata' }
  ],
  
  async execute(context) {
    // 1. 发现 URL
    const urls = await discoverUrls(context.inputs.keywords);
    // 2. 采集内容
    const posts = await fetchPosts(urls);
    // 3. 返回结果
    return { posts, metadata: { count: posts.length } };
  }
};
```

#### 4.3.3 反爬对抗技术栈

**反爬对抗是 Execution Plane 的核心能力之一**，采用分层防御策略：

```
┌─────────────────────────────────────────────────────────────┐
│  第一层：网络层                                              │
│  ├── 代理池轮换（HTTP/HTTPS/Socks5）                         │
│  ├── 住宅代理 + 数据中心代理混合                              │
│  └── 智能代理选择（根据平台选择最优代理）                      │
├─────────────────────────────────────────────────────────────┤
│  第二层：浏览器层                                            │
│  ├── CDP 模式（MediaCrawler）                               │
│  │   └── 控制用户真实 Chrome/Edge 浏览器                      │
│  ├── Camoufox（备选）                                        │
│  │   └── Firefox 内核，自动修改浏览器指纹                      │
│  └── Puppeteer-Stealth（备选）                               │
│      └── 隐藏自动化特征                                       │
├─────────────────────────────────────────────────────────────┤
│  第三层：行为模拟层                                          │
│  ├── 鼠标轨迹模拟（贝塞尔曲线）                               │
│  ├── 页面滚动行为（随机停留时间）                             │
│  ├── 点击热区模拟（避开机器人检测区域）                        │
│  └── 操作间隔随机化（正态分布随机延迟）                        │
├─────────────────────────────────────────────────────────────┤
│  第四层：策略层                                              │
│  ├── 请求频率动态调整（根据响应调整）                          │
│  ├── 熔断机制（连续失败自动暂停）                             │
│  ├── 封禁检测与自动切换（IP/账号切换）                         │
│  └── 多账号轮换（Cookie 池管理）                              │
└─────────────────────────────────────────────────────────────┘
```

**反爬工具链**：

| 工具 | 用途 | 依赖 | 适用场景 |
|------|------|------|----------|
| **MediaCrawler CDP** | 控制真实浏览器 | Playwright | 小红书、抖音等需要登录的平台 |
| **Camoufox** | 反检测浏览器 | Playwright（内置） | 强反爬平台（淘宝、京东评价） |
| **Pinchtab** | 浏览器指纹管理 | Chromium | 多账号隔离、指纹切换 |
| **Puppeteer-Stealth** | 隐藏自动化特征 | Puppeteer | 轻量级动态页面采集 |
| **Playwright** | 现代浏览器自动化 | - | 通用动态页面采集 |

**Camoufox 说明**：
- 基于 Firefox 内核的反检测浏览器
- **内部集成 Playwright**，提供类似 Playwright 的 API（`camoufox.sync_api`）
- 自动进行指纹注入和轮换（Canvas、WebGL、字体等）
- 安装：`pip install camoufox`（会自动安装 Playwright 依赖）
- 适用于强反爬平台（淘宝、京东评价页）

**Pinchtab 说明**：
- 专业的浏览器指纹管理和多账号隔离工具
- 支持为每个采集任务分配独立的浏览器环境
- 自动管理 Cookie、LocalStorage、Canvas 指纹等
- 适用于需要多账号轮换采集的场景（如微博、知乎）

#### 4.3.4 验证码处理方案

**三层验证码降级策略**：

```
┌─────────────────────────────────────────────────────────────┐
│  第一层：本地识别（优先）                                     │
│  ├── ddddocr：滑块验证码识别（OpenCV模板匹配）               │
```

**工作流引擎选型说明**：

我们选择 **Temporal** 作为工作流引擎，原因如下：

1. **生产级可靠性**：Exactly-Once 执行、故障自动恢复、状态持久化
2. **长时间运行支持**：原生支持天/周/月级别的任务（如 7×24 舆情监控）
3. **完善的生态**：多语言 SDK、丰富的文档、活跃的社区
4. **Docker 部署简单**：官方提供 Docker Compose 配置，一键启动

**Docker 部署方案**：

```yaml
# docker-compose.yml
version: '3'
services:
  temporal-server:
    image: temporalio/server:latest
    ports:
      - "7233:7233"
    environment:
      - DB=postgresql
      - DB_PORT=5432
      - POSTGRES_USER=temporal
      - POSTGRES_PWD=temporal
      - POSTGRES_SEEDS=postgresql
    depends_on:
      - postgresql

  postgresql:
    image: postgres:13
    environment:
      POSTGRES_USER: temporal
      POSTGRES_PASSWORD: temporal
      POSTGRES_DB: temporal
    volumes:
      - temporal-data:/var/lib/postgresql/data

  temporal-ui:
    image: temporalio/ui:latest
    ports:
      - "8080:8080"
    environment:
      - TEMPORAL_ADDRESS=temporal-server:7233

volumes:
  temporal-data:
```

**启动命令**：`docker-compose up -d`

**使用说明**：Temporal 的引入会增加一定的学习成本，但可以避免后期架构重构的风险。团队可以在开发过程中逐步熟悉 Temporal 的概念和 API。

## 9. 与参考项目的区别

| 参考项目 | 核心特点 | 我们的区别 | GitHub |
|----------|----------|-----------|--------|
| **BettaFish** | AI Agent 直接驱动采集 | AI 制定策略，Worker 确定性执行，更可审计 | https://github.com/666ghj/BettaFish |
| **NotebookLM** | 用户上传文档，AI 整理 | 主动采集多平台数据，专注舆情/情报场景 | https://notebooklm.google.com |
| **DeerFlow** | Multi-Agent 直接执行 | Agent 决策 + Worker 执行，确定性更强 | https://github.com/bytedance/deer-flow |
| **传统爬虫** | 单一平台、固定规则 | 多平台统一适配、AI 动态策略、Skill 化设计 | - |

**核心差异化**：
1. **确定性执行**：DeerFlow 的 Agent 直接执行，我们的 Agent 只生成 Workflow，Worker 确定性执行
2. **主动采集**：相比 NotebookLM 的用户上传，我们主动采集多平台数据
3. **垂直场景**：专注舆情/商品评价，情感分析更深入
4. **Skill 化**：借鉴 DeerFlow 的 Skill 设计，但更适合我们的 Worker 架构

## 附录 A：与 papaweb 的关系

本项目是 papaweb 的 **Greenfield 重启版本**，与原有项目的关系：

| 维度 | 原 papaweb | 新版本 |
|------|-----------|--------|
| **架构** | 渐进改造 | 彻底重启 |
| **主线分支** | `main` / `dev` | `greenfield/main` / `greenfield/dev` |
| **核心模型** | 待梳理 | Topic/Evidence/Artifact |
| **AI 角色** | 待明确 | Agent 决策 + Worker 执行 |
| **技能系统** | 无 | Skill 化设计 |
| **情感分析** | 无 | 核心功能 |

**迁移策略**：
- 旧代码在 legacy 分支保留，仅做维护
- 新功能全部在 Greenfield 主线开发
- 未来通过数据迁移工具合并历史数据
