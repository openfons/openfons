OpenFons 最佳实践架构图

OpenFons 采用四平面解耦架构，让用户问题沿着 8 个受控步骤穿过系统，最终沉淀成可审计、可回放、可交付的长期资产。

四平面 = 架构分层 Access / Control / Execution / Data & Delivery 8步 = 主链步骤 用户问题如何穿过这 4 层并变成可交付资产
Plane 01

Access Plane

只负责承接输入与启动编译，不在入口层偷偷做业务判断。

Step 1. User Input

自然语言、模板、界面化操作、API 请求统一进入系统。

借鉴边界 借 MiroFish 的地方是“让用户看懂过程和结果”，不是把它的模拟主链搬过来。
Plane 02

Control Plane

这是 OpenFons 的核心大脑，负责机会判断、确认门禁、任务编译与执行放行。

Step 2. Intent Structuring

把模糊需求整理成结构化意图、边界和候选方向。

Step 3. Planning Swarm

以前置深度研究方式并行分析需求、竞争、变现、地区/语言与证据要求，并沉淀可审计 discovery captures。

Step 4. Opportunity Judge

统一裁决，输出唯一前置真源 OpportunitySpec。

Step 5. User Confirmation

确认通过后，机会判断才能进入执行编译层。

Step 6. Compile Confirmed Task Boundary

在用户确认后编译出 TaskSpec / WorkflowSpec，并经过 Execution Budget / Policy Gate / Scheduler 放行；ReportSpec 需要在证据与 findings 下游生成。

必须保留 不要把 DeerFlow 风格的 lead agent 升格成这层主脑。执行层只能消费这里产出的契约与放行结果。
Plane 03

Execution Plane

这一层最适合借 DeerFlow，但借的是组织方式，不是业务控制权。

Step 7. Worker Execution

受控执行采集、分析、回传。当前已明确的 gateway 子项目是 `SearchGateway v1`，它先产生 `SearchRun / SearchResult / UpgradeCandidate`；这些候选结果后续再进入正式采集链路，落成 `SourceCapture / CollectionLog`。同时可继续引入 harness、skills、sandbox、sub-agents。

借鉴边界 借 DeerFlow 的 Runtime Harness / Skills Loader / Tools Registry / Sandbox / Sub-Agent。 但 memory / thread 不能当长期真源；当前第一个具体 gateway 子项目是 SearchGateway v1，它属于 discovery execution layer，不等于 SourceCapture 或 EvidenceSet，也不应把后续正式采集链过早写死成已实现服务名。
Plane 04

Data & Delivery Plane

这里决定 OpenFons 为什么不是普通 agent 平台，而是证据驱动的研究与交付系统。

Step 8. Collection to Evidence to Artifact

从 Topic / TopicRun / SourceCapture / CollectionLog 留痕开始，经过 Evidence / EvidenceSet 资格化与 finding 归纳后，再生成 ReportSpec 进入 Renderer / Portal，最终形成 Artifact。

借鉴边界 借 MiroFish 的是“报告型产品表达”和“过程可见性”，但事实与结论仍然要由证据真源驱动，不能把前置规划结果误当成最终报告事实。

架构定义

四平面负责解耦系统边界，8 步主链负责把用户问题编译成可执行、可审计、可交付的资产链路。

最佳实践

上游像 OpenFons，中游像 DeerFlow，下游像 MiroFish，但真源始终还是 OpenFons 自己的契约与证据层。

执行顺序

先稳 Control Plane，再补 Execution Plane，最后强化 Data & Delivery 的产品表达与交付体验。
