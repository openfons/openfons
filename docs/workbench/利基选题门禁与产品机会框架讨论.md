# 利基选题门禁与产品机会框架讨论

> 项目名：OpenFons
> 文档定位：围绕“niche 利基市场选题标准表、内容机会、产品机会与可扩展门禁”的讨论纪要
> 形成日期：2026-03-26
> 状态说明：这是当前最稳的 `v1` 讨论结论，用于指导第一阶段产品测试，不应被视为不可变终局

> 2026-03-28 实验治理说明：本文保留为 v1 讨论纪要。文中把 Authority / Distribution / Tool Potential / Subscription Potential / Geo/Localization 等维度放在“后续可能引入”的位置，但当前 dual experiment 已经将 Authority / Distribution / Compliance / Maintenance Cost 提升为现行 hard gates。本文继续作为演化背景，不再单独作为执行契约。

## 1. 文档目的

本文件用于单独沉淀本轮对话中形成的关键判断：

1. OpenFons 的机会判断不应只服务于“生成一篇 SEO 页面”，而应同时服务于“生成内容机会”和“生成产品机会”。
2. 利基市场的判断标准不应只围绕“大词热度”，而应围绕“窄而深的小意图是否值得长期经营”。
3. 当前形成的门禁不应被写成不可变制度，而应作为可扩展、可迭代的 `v1 opportunity gate`，先通过两个产品或更多真实产品测试逐步完善。

## 2. 本轮核心结论

### 2.1 项目真正要做的不是“找词写页”，而是“把用户问题编译成机会”

对 OpenFons 来说，更准确的表述应是：

`用户问题 -> 机会判断 -> 内容机会图 + 产品机会图 -> 用户确认 -> 受控采集 -> EvidenceSet -> ReportSpec -> Next.js 页面；同时在 OpportunitySpec 中保留产品机会字段，用于后续数据产品 / 工具站 / 订阅入口评估`

这里最重要的变化是：

1. 前置层不只回答“该不该写一篇 SEO 页”。
2. 前置层还要回答“这个问题是否值得进一步长成工具、订阅或持续更新产品”。

需要特别说明：

1. 当前第一阶段的默认外部交付主线仍是 `ReportSpec -> report-web / Next.js 报告页`。
2. 工具站、数据产品和订阅产品的机会，当前更适合作为 `OpportunitySpec` 的内部扩展字段、候选方向或评估输出保留。
3. 在没有单独 ADR / RFC 之前，不建议直接新增与 `TaskSpec / WorkflowSpec / ReportSpec` 平级的新外部契约名。

### 2.2 真正的利基机会，不是“大词做浅”，而是“小意图做透”

利基市场不应优先追求“流量最大”，而应优先追求：

1. 问题足够具体。
2. 意图足够高价值。
3. 现有结果不能完整解决用户的真实选择问题。
4. 有足够公开证据支撑独立结论。
5. 可以长期更新。
6. 可以明确变现。

也就是说：

`窄而深` 的本质不是缩小题目字面长度，而是把一个小而贵的小意图讲透、做透、持续更新透。

### 2.3 “投行式权威感”来自结构和证据，不来自页面皮相

如果目标是生成“像投行 PPT 一样有权威感”的 Next.js 网站，真正构成权威感的核心要素是：

1. 明确的 thesis
2. 决策树和比较矩阵
3. 证据附录
4. 更新时间线
5. 来源分层
6. 明确的适用人群和适用边界

Next.js 是优秀载体，但不是护城河本身。

## 3. 公允判断

### 3.1 `对象 × 决策动作 × 受众 × 约束条件 × 地区/语言` 很有用，但更适合做页面编译公式

这个公式适合当前案例，尤其适合：

1. 选型型主题
2. 比较型主题
3. 采购型主题
4. 部署型主题

例如：

1. `OpenClaw × 部署方式选择 × 普通用户 × 低成本稳定优先 × 英文`
2. `AI coding × 模型采购 × 小团队 × 便宜但够用 × 美国`

但它并不天然等于“所有 niche 市场的总框架”。它更适合作为：

`值得做之后，把页面编译成具体落地方向的页面级编译公式`

### 3.2 更通用的最佳实践，应先回到“问题、意图、证据、空位、变现、更新性”

更通用的 `v1` 利基判断框架，建议先看这 8 个维度：

1. `Problem`
   用户到底有什么具体问题。
2. `Intent`
   这是资讯意图、决策意图、执行意图，还是持续监控意图。
3. `Audience`
   谁最痛，谁最可能付费。
4. `Market Context`
   地区、语言、平台、行业环境是什么。
5. `Evidence`
   有没有足够公开证据把这个问题讲透。
6. `Gap`
   当前内容、分发、工作流或数据层里，真正缺的切入空位到底是什么。
7. `Monetization`
   这个问题最后能接什么钱。
8. `Updateability`
   它是不是一个能持续更新和经营的主题。

这 8 维更适合作为：

`通用利基判断框架`

而页面编译公式更适合作为：

`第二层：把值得做的机会编译成页面与产品方向`

## 4. 当前推荐的 `v1` 利基门禁

一个主题要进入公开页面、报告站、工具站或订阅产品的排期，建议至少先通过下面 6 条：

1. 有明确决策意图或高价值执行意图，而不只是泛资讯浏览。
2. 官方文档、现有内容或既有产品，不能完整回答或解决用户的真实问题。
3. 有足够公开证据可做交叉验证，不依赖单一商业来源。
4. 可以持续更新，不是一锤子买卖。
5. 有明确变现路径，哪怕很窄。
6. 存在明确的分发或切入空位：
   对报告页更常见的是 `SERP gap`；
   对工具站或订阅产品更常见的是 `workflow gap / channel gap / data gap`。

需要明确：

1. 这 6 条是当前最稳的 `v1 core gate`。
2. 它们不是永久固定制度。
3. 它们应通过真实案例回放不断修订。

## 5. 不应把当前门禁当成最终执行法规

这是本轮讨论里最需要被单独写清楚的一点。

当前形成的门禁，更合理的定义不是：

`最终固定执行门禁`

而是：

`可扩展、可回放、可迭代的 v1 opportunity gate`

原因有三点：

1. 当前项目仍处于第一阶段验证期，真正重要的是“先用真实案例看门禁是否有命中率”，而不是假设纸面规则已是终局。
2. 不同利基市场的成功因素并不完全相同，后续很可能需要引入新的维度，例如 `Authority`、`Distribution`、`Tool Potential`、`Subscription Potential`、`Geo/Localization`。
3. 如果一开始把当前门禁写死，后续团队会更容易把它当教条，而不是当作可学习系统。

因此，更稳的工程原则应写成：

1. 先用当前 `v1 gate` 跑通 2 个产品测试。
2. 再根据结果补充、删减或调权。
3. 允许后续扩大到更多产品测试，而不是只围绕当前两个案例。

## 6. 内容机会与产品机会要分开看

对于每一个用户问题，前置层不应只输出一张“内容机会图”，还应输出一张“产品机会图”。

### 6.1 内容机会图回答什么

1. 值得做哪些 query / keyword cluster
2. 值得做哪些报告页、比较页、决策页、追踪页
3. 哪些页面先发
4. 哪些页面应作为 pillar，哪些是 supporting pages

### 6.2 产品机会图回答什么

1. 这个题能不能继续长成工具
2. 能长成什么类型的工具
3. 是免费工具、付费工具，还是数据订阅
4. 适合一次性售卖、会员订阅，还是持续更新订阅

为了不和当前正式契约打架，当前阶段更稳的做法是：

1. 先把这些结果保留在 `OpportunitySpec` 的内部字段、附录或扩展候选对象里。
2. 先用于判断“是否值得继续产品化”，而不是马上并列引入新的正式交付契约。
3. 未来若要形成独立工具构建契约，应通过单独 ADR / RFC 收口。

## 7. AI 工具站与订阅潜力判断

用户问题经过前置分析后，未必只生成一个 SEO 页面，也可能生成一个小产品簇：

1. 一篇权威主报告页
2. 两到三篇支持页
3. 一个交互式比较器或决策工具
4. 一个追踪页或订阅页
5. 一个可收费的数据产品入口

### 7.1 什么时候一个题适合继续长成 AI 工具站

建议额外增加下面 5 条判断：

1. 这个问题会不会被用户反复执行。
2. 这个问题能不能参数化输入。
3. 这个问题能不能输出结构化结果。
4. 用户愿不愿意为省时间、降风险或持续更新付费。
5. 这个能力是否适合形成持续订阅，而不是一次性消费。

如果一题在这 5 条中满足 4 条或以上，就不应只被视为“内容题”，而应被视为：

`内容题 + 工具题 + 潜在订阅题`

### 7.2 因此 OpenFons 的长期产物不应只是一种站

更合理的长期产物层次可以是：

1. `Authority Site`
   权威感报告页，承接 SEO 和品牌信任。
2. `Data Product`
   可更新的数据页、对比页、追踪页。
3. `Workflow Tool`
   帮用户完成具体动作的轻工具或交互式工具。
4. `Subscription Product`
   持续更新、提醒、数据库、监控或推荐订阅。

## 8. 第一阶段测试建议

当前最合理的做法不是一次把整套方法写死，而是先拿真实产品测试。

### 8.1 当前建议的第一批测试对象

1. `OpenClaw 部署 SEO 主题`
   更适合验证“利基页面机会判断 + 报告页 + affiliate / lead”。
2. `AI coding / agent 模型采购、路由、成本与地区选择`
   更适合验证“高商业意图 + 采购决策 + 工具潜力 + 订阅潜力”。

这里也要保持边界感：

1. 这两个测试当前仍以“报告页主交付”作为第一阶段默认主线。
2. 工具潜力、订阅潜力和数据产品潜力应被记录、评估和排序。
3. 只有当测试结果证明这些潜力足够强时，才应继续进入独立产品化排期。

### 8.2 这两个测试的意义

它们不是为了证明这两个题就是最终大机会，而是为了验证：

1. 当前 `v1 gate` 是否能正确筛题
2. 是否能把用户问题编译成稳定的内容机会图
3. 是否能识别出工具潜力和订阅潜力
4. 哪些字段和维度仍然缺失

### 8.3 建议记录的回放结果

每做完一个测试，建议至少复盘：

1. 这个门禁有没有正确拦掉低价值题
2. 有没有误判高价值题
3. 真实流量是否符合预期
4. 真实转化是否出现
5. 工具潜力和订阅潜力是否被高估或低估
6. 是否需要新增、删除或重配某些门禁维度

## 9. 当前更稳的定义

因此，对本轮讨论形成的内容，更稳的统一表述应是：

1. 当前形成的是 `OpenFons 利基选题与产品机会门禁 v1`
2. 它是第一阶段的测试框架，不是最终法规
3. 它应在真实产品测试中持续迭代
4. 它同时服务于内容机会判断和产品机会判断

## 10. 当前最值得坚持的判断

1. OpenFons 的核心不只是“找关键词做 SEO 页面”，而是“把用户问题编译成内容机会和产品机会”。
2. `对象 × 决策动作 × 受众 × 约束条件 × 地区/语言` 适合做页面编译，不适合被误读为所有 niche 的总框架。
3. 更通用的框架应回到 `Problem / Intent / Audience / Market Context / Evidence / Gap / Monetization / Updateability`。
4. 当前门禁应保持可扩展、可回放、可调权，而不是一开始就写成不可变教条。
5. 通过两个产品或更多产品先做真实测试，再逐步完善门禁，是当前更稳的工程策略。
