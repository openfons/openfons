# OpenFons 利基选题标准表 v1 设计稿

> 日期：2026-03-26
> 适用范围：`docs/plan1/利基选题标准表-v1.md` 的正式落地前设计
> 状态：待用户审阅

## 1. 目标

为 OpenFons 第一阶段建立一套可执行、可复盘、可迭代的 `利基选题标准表 v1`，用于把用户问题统一判断为：

1. 值不值得进入排期
2. 更适合先做内容机会还是继续长成产品机会
3. 在第一阶段应落成什么动作

这份标准表不是最终法规，而是第一阶段的 `v1 opportunity gate` 操作化版本。

## 2. 非目标

这份标准表当前不承担下面这些职责：

1. 不直接替代 `OpportunitySpec`
2. 不直接生成最终 `TaskSpec / WorkflowSpec / ReportSpec`
3. 不把工具站机会直接升级为与 `ReportSpec` 平级的新外部契约
4. 不试图一次覆盖所有利基市场终局规则

## 3. 设计原则

### 3.1 先总门禁，后分流

先判断一个题是否值得进入排期，再判断它更适合先做内容还是继续长成产品。

### 3.2 内容机会与产品机会分开打分

同一个题可能：

1. 内容机会强，产品机会弱
2. 内容机会一般，产品机会强
3. 两者都强
4. 两者都弱

因此不能只靠一张混合总分表做判断。

### 3.3 Phase 1 默认仍是报告页优先

即使一个题具备产品潜力，第一阶段默认动作仍应优先落在：

`ReportSpec -> report-web / Next.js 报告页`

产品潜力应作为 `OpportunitySpec` 内部扩展字段和后续排期依据保留。

### 3.4 评分必须可解释

每个维度不只给分数，还必须给：

1. 判断说明
2. 证据来源
3. 主要不确定性
4. 下一步动作建议

## 4. 产物结构

正式文档建议落成一个主文档：

`docs/plan1/利基选题标准表-v1.md`

该主文档内部由 3 个部分构成：

1. `总门禁表`
   决定是否进入排期
2. `内容机会子表`
   决定是否先做报告页、比较页、决策页、追踪页
3. `产品机会子表`
   决定是否保留为工具、数据产品或订阅候选

这不是 3 份孤立文档，而是 1 份标准表文档里的 3 层结构。

## 5. 总门禁表设计

### 5.1 用途

总门禁表只回答一个问题：

`这个题是否值得进入 OpenFons 当前排期`

### 5.2 维度

总门禁表采用 8 个维度：

1. `Problem`
   用户问题是否具体而真实
2. `Intent`
   是否存在明确决策意图或高价值执行意图
3. `Audience`
   目标人群是否清晰，是否有痛点和付费可能
4. `Market Context`
   地区、语言、平台和行业环境是否可判断
5. `Evidence`
   是否有足够公开证据可交叉验证
6. `Gap`
   当前内容、分发、工作流或数据层里是否存在明确切入空位
7. `Monetization`
   是否存在明确变现路径
8. `Updateability`
   是否适合持续更新，而不是一次性主题

### 5.3 评分方式

每个维度使用 `0-3` 四档评分，避免伪精确：

1. `0`
   明显不成立，或几乎没有证据支撑
2. `1`
   弱成立，有初步信号但很不稳定
3. `2`
   基本成立，有可操作证据
4. `3`
   强成立，证据和判断都很清晰

最大分数为 `24`。

### 5.4 红线项

下面 4 项同时作为红线判断：

1. `Intent`
2. `Evidence`
3. `Gap`
4. `Monetization`

如果这 4 项中任意 2 项得 `0`，则总门禁默认直接进入 `No-Go`。

### 5.4.1 上限规则

为了避免“分数堆高，但目标市场仍然模糊”的误判，增加一个 `ceiling rule`：

1. 如果 `Audience` 仍然不清晰，只能停留在泛人群描述，则总门禁最高只能到 `Hold`
2. 如果 `Market Context` 中的地区、语言或平台仍然完全未收敛，则总门禁最高只能到 `Hold`
3. 如果 `Audience` 与 `Market Context` 都仍是纯假设，没有最基本的研究支撑，则不应进入 `Go`

### 5.5 总门禁结论

总门禁输出 3 档：

1. `Go`
   无红线触发，且总分 `>= 16`
2. `Hold`
   无致命红线，但总分在 `10-15`
3. `No-Go`
   触发红线，或总分 `< 10`

### 5.5.1 判定优先级

总门禁的判定优先级必须固定为：

`红线项 -> 上限规则 -> 总分区间`

也就是说，不能因为总分高，就绕过红线项或上限规则。

### 5.6 为什么用 0-3 而不是 1-5

推荐使用 `0-3`，原因如下：

1. 内部评审更强调快速判断，不需要过高颗粒度
2. `0` 可以明确表达“当前不成立”
3. 比 `1-5` 更不容易制造“看起来很精确”的错觉
4. 更适合第一阶段快速复盘和调权

## 6. 内容机会子表设计

### 6.1 用途

内容机会子表只回答：

`这个题是否值得先落成报告页、比较页、决策页、追踪页`

### 6.2 维度

内容机会子表采用 6 个维度：

1. `Query / Distribution Fit`
   是否存在可承接的搜索或分发需求
2. `Angle Clarity`
   是否能形成窄而深、可防止 cannibalization 的页面角度
3. `Evidence Depth`
   是否有足够深度支撑权威型页面
4. `Update Loop`
   是否适合用更新时间线、日志、追踪持续经营
5. `Content Gap`
   当前结果是否普遍浅、旧、抄表、缺比较或缺决策树
6. `Content Monetization Fit`
   是否适合广告、affiliate、lead、咨询或内容订阅

### 6.3 评分方式

同样采用 `0-3` 四档，总分 `18`。

### 6.4 内容结论

1. `Content-Go`
   总分 `>= 12`
2. `Content-Hold`
   总分 `7-11`
3. `Content-No-Go`
   总分 `<= 6`

### 6.5 Phase 1 动作映射

1. `总门禁 Go + Content-Go`
   进入 `report-web` 路线候选
2. `总门禁 Hold + Content-Go`
   先保留在观察池，不直接发布
3. `总门禁 Go + Content-Hold`
   不优先做公开页面，除非产品机会很强

## 7. 产品机会子表设计

### 7.1 用途

产品机会子表只回答：

`这个题是否值得继续长成工具、数据产品或订阅产品`

### 7.2 维度

产品机会子表采用 6 个维度：

1. `Repetition`
   用户是否会反复执行这个问题
2. `Parameterizability`
   问题是否可以参数化输入
3. `Structured Output`
   结果是否可以稳定输出为结构化对象
4. `Willingness To Pay`
   用户是否愿意为省时、降风险或持续更新付费
5. `Subscription Fit`
   是否适合形成持续订阅，而不是一次性消费
6. `Data / Workflow Moat`
   是否可能形成数据壁垒、工作流嵌入或长期留存

### 7.3 评分方式

同样采用 `0-3` 四档，总分 `18`。

### 7.4 产品结论

1. `Product-Go`
   总分 `>= 12`
2. `Product-Hold`
   总分 `7-11`
3. `Product-No-Go`
   总分 `<= 6`

### 7.5 Phase 1 动作映射

1. `总门禁 Go + Product-Go`
   保留为 `productOpportunityHints / expansionCandidates`
2. `总门禁 Go + Content-Go + Product-Go`
   先走报告页验证，同时记录后续产品化候选
3. `总门禁 Go + Content-Hold + Product-Go`
   不强推 SEO 页，进入产品探索池

## 8. 标准表使用流程

推荐固定成下面 5 步：

1. 输入用户问题
2. 填写总门禁表
3. 仅当总门禁非 `No-Go` 时，再填写内容机会子表与产品机会子表
4. 输出排期结论
5. 为后续复盘记录真实结果

### 8.1 决策映射矩阵

为了避免“子表结论看得懂，但不知道下一步做什么”，正式文档应固定下面这组动作映射：

1. `Total No-Go`
   直接归档，不进入当前排期
2. `Total Hold + 任一子表 Go`
   进入观察池，先补研究，不直接上线
3. `Total Go + Content-Go + Product-No-Go / Hold`
   走 `report-web` 优先路线
4. `Total Go + Content-Hold / No-Go + Product-Go`
   进入产品探索池，不强推公开报告页
5. `Total Go + Content-Go + Product-Go`
   先走报告页验证，同时记录工具 / 数据 / 订阅候选
6. `Total Go + Content-Hold + Product-Hold`
   不立即启动，先补证据或换角度后再评

## 9. 文档内必须包含的字段

正式文档里每个题建议至少记录这些字段：

1. `Topic`
2. `Seed Question`
3. `Audience`
4. `Geo / Language`
5. `Assumptions`
6. `Total Gate Score`
7. `Content Opportunity Score`
8. `Product Opportunity Score`
9. `Decision`
10. `Recommended Phase 1 Action`
11. `Key Evidence`
12. `Open Risks`
13. `Review Date`
14. `Replay Notes`
15. `Confidence Level`
16. `Actual Outcome Snapshot`

## 10. 必须包含的说明书内容

正式文档不应只有空表格，还必须包含说明书部分：

1. 为什么采用 `1 个总门禁 + 2 个子评分表`
2. 每个维度是什么意思
3. 每一档分数如何判断
4. 什么情况下进入 `Go / Hold / No-Go`
5. 分数如何映射到 Phase 1 动作
6. 如何在真实案例后复盘并调整维度或阈值

### 10.1 每个维度必须带评分锚点

为了避免不同人打分口径完全不同，正式文档中每个维度都必须附带：

1. `0 / 1 / 2 / 3` 的判定锚点
2. 至少 1 个“高分示例”
3. 至少 1 个“低分示例”

这不是为了追求教科书完整度，而是为了提高不同评审者之间的一致性。

## 11. 必须包含的示例

正式文档至少带 2 个样例评分：

1. `OpenClaw 部署 SEO 主题`
2. `AI coding / agent 模型采购、路由、成本与地区选择`

样例的作用不是证明分数绝对正确，而是演示：

1. 如何填写
2. 如何从总门禁走到内容/产品分流
3. 如何把结果转成第一阶段动作

## 12. 与现有文档的对齐关系

### 12.1 与 `利基选题门禁与产品机会框架讨论.md` 对齐

标准表是该文档的操作化版本，不改变其核心结论：

1. `v1 opportunity gate` 不是终局制度
2. 内容机会与产品机会要分开看
3. 更通用的框架应回到 `Problem / Intent / Audience / Market Context / Evidence / Gap / Monetization / Updateability`

### 12.2 与 `开放源平台技术团队说明.md` 对齐

标准表不新增外部契约，只服务于：

`Intent Structuring -> Planning Swarm -> Opportunity Judge -> OpportunitySpec`

产品机会结果仍保留在 `OpportunitySpec` 的内部扩展字段中。

## 13. 正式文档建议结构

`docs/plan1/利基选题标准表-v1.md` 建议采用下面顺序：

1. 文档目的与状态说明
2. 为什么采用三层结构
3. 使用方式
4. 总门禁表
5. 内容机会子表
6. 产品机会子表
7. 决策映射规则
8. 复盘机制
9. OpenClaw 示例
10. AI procurement 示例

## 14. 成功标准

如果正式文档完成后满足下面 5 条，就算这次设计成功：

1. 团队可以拿一条用户问题直接评分
2. 团队能明确区分内容机会和产品机会
3. 团队能把分数映射到 Phase 1 动作
4. 文档与现有 `plan1` 主线不打架
5. 后续可以基于真实案例继续调分、调权和增删维度
6. 同一题由两位评审基于同一批证据独立评分时，结论通常不会相差超过一档

## 15. 风险与控制

### 15.1 风险

1. 表格过复杂，导致团队懒得用
2. 分数过精细，导致虚假精确
3. 过早把产品机会写成强执行门禁
4. 不留复盘位，导致文档只会越写越僵

### 15.2 控制

1. 先用 `0-3` 简化评分
2. 先用 8 + 6 + 6 维，不继续膨胀
3. Phase 1 默认动作仍以报告页验证为主
4. 明确保留 `Replay Notes` 与后续迭代说明

## 16. 本设计的推荐结论

推荐按本设计直接落成 `docs/plan1/利基选题标准表-v1.md`，不再继续抽象扩展。

原因如下：

1. 结构已经足够覆盖当前两类盈利项目
2. 与现有 `v1 opportunity gate` 和 `OpportunitySpec` 边界一致
3. 同时保留了可复盘、可迭代空间
4. 足以支撑第一阶段真实产品测试
