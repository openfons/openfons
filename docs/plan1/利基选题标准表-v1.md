# 利基选题标准表 v1

> 项目名：OpenFons
> 文档定位：第一阶段可执行的内部评审表 + 说明书
> 形成日期：2026-03-26
> 状态说明：这是当前最稳的 `v1` 版本，用于真实产品测试与复盘，不应被误读为最终固定法规

## 1. 文档目的与状态

本文件用于把前面已经确认的 `v1 opportunity gate` 落成团队可直接使用的标准表。

它要回答 3 件事：

1. 一个题值不值得进入当前排期
2. 这个题更适合先做内容机会，还是继续长成产品机会
3. 在第一阶段应该采取什么动作

这份标准表是操作化工具，不是新外部契约。

它不直接替代：

1. `OpportunitySpec`
2. `TaskSpec`
3. `WorkflowSpec`
4. `ReportSpec`

它服务的仍然是前置编译链：

`Intent Structuring -> Planning Swarm -> Opportunity Judge -> OpportunitySpec`

## 2. 为什么采用“三层结构”

当前最佳实践不是只做一张混合总表，也不是把内容机会和产品机会完全拆成两套互不相关的系统。

更稳的结构是：

1. `总门禁表`
   先判断一个题是否值得进入当前排期
2. `内容机会子表`
   再判断它是否值得先落成报告页、比较页、决策页、追踪页
3. `产品机会子表`
   再判断它是否值得继续长成工具、数据产品或订阅产品

采用这个结构有 4 个原因：

1. 可以先拦掉低价值题，避免一上来争论“做内容还是做工具”
2. 可以明确区分“内容强但产品弱”和“内容一般但产品强”
3. 可以和当前 `OpportunitySpec` 的内部扩展字段保持一致，不引入新平级契约
4. 可以保留 Phase 1 以 `report-web / Next.js 报告页` 为默认主线的工程边界

## 3. 使用方式

推荐固定成下面 5 步：

1. 输入用户问题，补齐最基本的受众、地区、语言、平台假设
2. 先填写 `总门禁表`
3. 仅当总门禁不是 `No-Go` 时，再填写 `内容机会子表` 和 `产品机会子表`
4. 根据决策映射规则输出排期结论
5. 在上线或测试后补 `Replay Notes / Actual Outcome Snapshot`

这份标准表建议用于：

1. 选题评审
2. 方向优先级排序
3. 案例回放
4. 后续调分、调权和增删维度

## 4. 总门禁表

### 4.1 用途

总门禁表只回答一个问题：

`这个题是否值得进入 OpenFons 当前排期`

### 4.2 评分方式

总门禁共 8 个维度，每个维度使用 `0-3` 四档评分：

1. `0`
   明显不成立，或当前几乎没有证据支撑
2. `1`
   弱成立，有初步信号，但很不稳定
3. `2`
   基本成立，已有可操作证据
4. `3`
   强成立，证据和判断都很清晰

最大分数为 `24`。

### 4.3 评分表

| 维度 | 判断问题 | 0 | 1 | 2 | 3 |
| --- | --- | --- | --- | --- | --- |
| `Problem` | 用户问题是否具体而真实 | 题目过大、过空、只是热词 | 有模糊问题感，但无法准确复述真实问题 | 可以说清用户在问什么，但边界仍偏宽 | 问题具体、痛点明确、可直接复述为高价值小意图 |
| `Intent` | 是否存在明确决策意图或高价值执行意图 | 纯资讯浏览 | 有兴趣但无明确动作 | 存在初步选型、比较、执行意图 | 明显是采购、部署、决策、降本、风险控制等高价值意图 |
| `Audience` | 目标人群是否清晰 | 完全不知道给谁看 | 只有泛人群，例如“所有用户” | 人群大体明确，例如“小团队/普通用户/开发者” | 人群清晰且痛点明确，例如“美国小团队采购负责人” |
| `Market Context` | 地区、语言、平台、行业环境是否可判断 | 完全无市场语境 | 只有非常泛的市场假设 | 已初步收敛到候选市场或语言 | 已明确到可以指导页面或产品落地的市场语境 |
| `Evidence` | 是否有足够公开证据可交叉验证 | 几乎没有公开证据，或只能靠单一来源 | 有零散来源，但交叉验证困难 | 有多类公开来源，基本能交叉验证 | 有稳定官方源、社区源、对照源，可支撑独立判断 |
| `Gap` | 当前是否存在明确切入空位 | 完全看不到切口 | 有直觉，但说不清空位在哪 | 可以指出内容、分发、工作流或数据层的缺口 | 切入空位清晰，且与现有大站/官方页有明确差异 |
| `Monetization` | 是否存在明确变现路径 | 几乎无变现路径 | 有很弱的间接变现想象 | 至少存在 1 条清晰变现路径 | 存在多条清晰路径，例如广告、affiliate、咨询、工具订阅 |
| `Updateability` | 是否适合持续更新 | 一次性主题 | 后续更新价值很弱 | 具备阶段性更新价值 | 天然适合做更新日志、追踪页、持续经营 |

### 4.4 评分锚点说明

高分示例：

1. `AI coding model procurement options`
   采购决策强、证据多、变现明确、可持续更新，通常会在 `Intent / Evidence / Monetization / Updateability` 上拿高分
2. `OpenClaw Windows install WSL2`
   问题具体、意图明确、证据充分，因此 `Problem / Intent / Evidence` 常见高分

低分示例：

1. `AI 很火，现在做点 AI 内容`
   `Problem / Intent / Audience / Gap` 都极可能偏低
2. `全球所有工具大全`
   问题过泛、切口不清，通常很难拿到高分

### 4.5 红线项

下面 4 项同时作为红线判断：

1. `Intent`
2. `Evidence`
3. `Gap`
4. `Monetization`

如果这 4 项中任意 2 项得 `0`，则总门禁默认直接进入 `No-Go`。

### 4.6 上限规则

为了避免“分数堆高，但目标市场仍然模糊”的误判，增加下面 3 条上限规则：

1. 如果 `Audience` 仍然不清晰，只能停留在泛人群描述，则总门禁最高只能到 `Hold`
2. 如果 `Market Context` 中的地区、语言或平台仍然完全未收敛，则总门禁最高只能到 `Hold`
3. 如果 `Audience` 与 `Market Context` 都仍是纯假设，没有最基本的研究支撑，则不应进入 `Go`

### 4.7 判定优先级

总门禁判定必须固定为：

`红线项 -> 上限规则 -> 总分区间`

不能因为总分高，就绕过红线项或上限规则。

### 4.8 总门禁结论

1. `Go`
   无红线触发，且总分 `>= 16`
2. `Hold`
   无致命红线，但总分在 `10-15`，或被上限规则压到 `Hold`
3. `No-Go`
   触发红线，或总分 `< 10`

## 5. 内容机会子表

### 5.1 用途

内容机会子表只回答：

`这个题是否值得先落成报告页、比较页、决策页、追踪页`

### 5.2 评分方式

共 6 个维度，依然使用 `0-3` 四档评分，总分 `18`。

### 5.3 评分表

| 维度 | 判断问题 | 0 | 1 | 2 | 3 |
| --- | --- | --- | --- | --- | --- |
| `Query / Distribution Fit` | 是否存在可承接的搜索或分发需求 | 几乎没有对应 query 或分发需求 | 有模糊需求，但很难稳定承接 | 存在可承接 query 或分发入口 | 需求明确，且能清楚对应一组查询或分发入口 |
| `Angle Clarity` | 是否能形成窄而深的页面角度 | 只能写成大而泛的总览 | 有角度想法，但容易互相打架 | 已有较清晰页面角度 | 页面 thesis、边界、归属规则都很清晰 |
| `Evidence Depth` | 是否有足够深度支撑权威型页面 | 证据太弱，只能写浅内容 | 可写基础页，但深度不足 | 可以支撑较扎实的决策页或比较页 | 可以支撑投行式结构、矩阵、决策树和附录 |
| `Update Loop` | 是否适合持续更新 | 发完即过时 | 只能偶尔更新 | 有阶段性更新空间 | 天然适合更新时间线、价格/版本/变化追踪 |
| `Content Gap` | 当前结果是否浅、旧、抄表、缺比较 | 结果已被完全占满，几乎无差异空间 | 有空位直觉，但证据不强 | 可以指出现有内容的几个明确缺口 | 空位清晰，例如官方不回答选择问题、大站内容太浅或太泛 |
| `Content Monetization Fit` | 是否适合广告、affiliate、lead、咨询或内容订阅 | 几乎无内容变现路径 | 只有很弱的间接变现 | 至少存在 1 条较清晰内容变现路径 | 存在多条清晰内容变现路径，且与页面意图匹配 |

### 5.4 评分锚点说明

高分示例：

1. `best OpenClaw setup for beginners`
   query 清晰、页面角度清晰、affiliate/lead 明确，通常能在 `Query / Distribution Fit` 和 `Content Monetization Fit` 上拿高分
2. `direct API vs OpenRouter`
   非常适合比较页和决策页，`Angle Clarity / Evidence Depth / Content Gap` 容易拿高分

低分示例：

1. `所有 AI 模型大全`
   角度过泛，易 cannibalization，通常 `Angle Clarity` 很低
2. `一个完全没人搜也没人讨论的冷门功能页`
   `Query / Distribution Fit` 容易低分

### 5.5 内容结论

1. `Content-Go`
   总分 `>= 12`
2. `Content-Hold`
   总分 `7-11`
3. `Content-No-Go`
   总分 `<= 6`

## 6. 产品机会子表

### 6.1 用途

产品机会子表只回答：

`这个题是否值得继续长成工具、数据产品或订阅产品`

### 6.2 评分方式

共 6 个维度，依然使用 `0-3` 四档评分，总分 `18`。

### 6.3 评分表

| 维度 | 判断问题 | 0 | 1 | 2 | 3 |
| --- | --- | --- | --- | --- | --- |
| `Repetition` | 用户是否会反复执行这个问题 | 基本只做一次 | 偶尔重复 | 会周期性重复 | 高频重复，是典型可工具化问题 |
| `Parameterizability` | 是否可以参数化输入 | 几乎无法参数化 | 只能做很弱参数化 | 可做基础参数化输入 | 参数化非常自然，适合做比较器、决策器、追踪器 |
| `Structured Output` | 是否可以稳定输出为结构化结果 | 结果只能是长文叙述 | 只能输出半结构化结果 | 可输出基础结构化结果 | 可稳定输出表格、矩阵、推荐结果、监控结果 |
| `Willingness To Pay` | 用户是否愿意为省时、降风险或持续更新付费 | 基本不愿付费 | 有模糊付费可能 | 存在较清晰付费意愿 | 付费动机明显，例如采购、降本、团队决策、持续监控 |
| `Subscription Fit` | 是否适合形成持续订阅 | 只适合一次性消费 | 适合弱持续消费 | 可以做阶段性订阅 | 天然适合监控、提醒、数据库、持续更新订阅 |
| `Data / Workflow Moat` | 是否可能形成数据壁垒或工作流嵌入 | 没有壁垒，任何人都可复制 | 有弱壁垒 | 可以形成一定数据积累或流程嵌入 | 有明显的数据壁垒、工作流嵌入或长期留存潜力 |

### 6.4 评分锚点说明

高分示例：

1. `AI provider procurement comparator`
   可参数化、可输出结构化结果、用户愿付费，`Parameterizability / Structured Output / Willingness To Pay` 常见高分
2. `pricing change tracker`
   适合持续订阅和更新，`Repetition / Subscription Fit` 容易高分

低分示例：

1. `一次性的部署经验分享`
   往往 `Repetition / Subscription Fit / Data Moat` 偏低
2. `纯观点型文章`
   结果很难结构化，`Structured Output` 常见低分

### 6.5 产品结论

1. `Product-Go`
   总分 `>= 12`
2. `Product-Hold`
   总分 `7-11`
3. `Product-No-Go`
   总分 `<= 6`

## 7. 决策映射规则

为了避免“分数看得懂，但不知道下一步该做什么”，统一采用下面这组映射：

1. `Total No-Go`
   直接归档，不进入当前排期
2. `Total Hold + 任一子表 Go`
   进入观察池，先补研究，不直接上线
3. `Total Go + Content-Go + Product-No-Go / Product-Hold`
   走 `report-web` 优先路线
4. `Total Go + Content-Hold / Content-No-Go + Product-Go`
   进入产品探索池，不强推公开报告页
5. `Total Go + Content-Go + Product-Go`
   先走报告页验证，同时记录工具 / 数据 / 订阅候选
6. `Total Go + Content-Hold + Product-Hold`
   不立即启动，先补证据或换角度后再评

### 7.1 Phase 1 默认边界

需要反复强调：

1. 第一阶段默认外部交付仍以 `ReportSpec + report-web` 为主
2. `Product-Go` 不等于立刻单独立项做工具
3. 产品机会结果应先保留为 `productOpportunityHints / expansionCandidates`

## 8. 复盘机制

### 8.1 每次评审必须记录的字段

每个题建议至少记录这些字段：

| 字段 | 说明 |
| --- | --- |
| `Topic` | 当前题目 |
| `Seed Question` | 用户原始提问 |
| `Audience` | 目标人群 |
| `Geo / Language` | 地区与语言 |
| `Assumptions` | 当前假设 |
| `Total Gate Score` | 总门禁得分 |
| `Content Opportunity Score` | 内容机会得分 |
| `Product Opportunity Score` | 产品机会得分 |
| `Decision` | `Go / Hold / No-Go` 及组合结论 |
| `Recommended Phase 1 Action` | 当前建议动作 |
| `Key Evidence` | 核心证据来源 |
| `Open Risks` | 主要不确定性 |
| `Review Date` | 评审日期 |
| `Replay Notes` | 上线后复盘 |
| `Confidence Level` | 当前置信度 |
| `Actual Outcome Snapshot` | 真实结果快照 |

### 8.2 复盘时要看什么

每做完一个真实测试，至少回看 6 件事：

1. 门禁有没有正确拦掉低价值题
2. 有没有误判高价值题
3. 内容机会是否被高估或低估
4. 产品机会是否被高估或低估
5. 第一阶段动作映射是否合理
6. 哪些维度、阈值或锚点需要调整

### 8.3 一致性标准

如果同一题由两位评审基于同一批证据独立评分，结论通常不应相差超过一档。

如果经常出现更大偏差，说明：

1. 锚点不清晰
2. 字段定义不清晰
3. 证据要求不清晰

## 9. 示例一：OpenClaw 部署 SEO 主题

### 9.1 题目定义

题目：

`围绕 OpenClaw 部署方式选择、Windows / Docker / VPS / 托管差异，寻找适合第一阶段上线的高意图报告页方向`

### 9.2 总门禁评分摘要

| 维度 | 分数 | 简短说明 |
| --- | --- | --- |
| `Problem` | 3 | 用户问题具体，部署路径与选择成本真实存在 |
| `Intent` | 3 | 明显是部署选择与执行意图 |
| `Audience` | 2 | 普通用户、开发者、小团队已可区分，但仍可继续细化 |
| `Market Context` | 2 | 至少可落到英文市场与中文市场候选 |
| `Evidence` | 3 | 官方安装文档、社区讨论、平台差异证据较充分 |
| `Gap` | 2 | 官方说明存在，但“怎么选”与“哪条路更适合谁”仍有空位 |
| `Monetization` | 3 | affiliate、lead、咨询都较明确 |
| `Updateability` | 2 | 版本变化与部署争议会带来阶段性更新 |

总分：`20 / 24`

总门禁结论：`Go`

### 9.3 内容机会评分摘要

| 维度 | 分数 | 简短说明 |
| --- | --- | --- |
| `Query / Distribution Fit` | 3 | 部署、Windows、Docker、VPS 等 query 清晰 |
| `Angle Clarity` | 3 | 可明确拆成总对比、入门决策、平台教程、托管对比 |
| `Evidence Depth` | 3 | 足以支撑决策树、矩阵、风险说明 |
| `Update Loop` | 2 | 可更新，但更新频率低于价格/采购类主题 |
| `Content Gap` | 2 | 官方能回答安装，不完全回答选择问题 |
| `Content Monetization Fit` | 3 | affiliate、lead、部署咨询较清晰 |

总分：`16 / 18`

内容结论：`Content-Go`

### 9.4 产品机会评分摘要

| 维度 | 分数 | 简短说明 |
| --- | --- | --- |
| `Repetition` | 1 | 部署选择通常不是高频重复问题 |
| `Parameterizability` | 1 | 可做简单决策器，但参数化程度有限 |
| `Structured Output` | 1 | 可输出推荐路径，但结果复杂度一般 |
| `Willingness To Pay` | 1 | 少部分人愿为部署服务付费，但不强 |
| `Subscription Fit` | 0 | 很难做成长期订阅 |
| `Data / Workflow Moat` | 0 | 壁垒较弱，易被复制 |

总分：`4 / 18`

产品结论：`Product-No-Go`

### 9.5 第一阶段建议动作

结论组合：

`Total Go + Content-Go + Product-No-Go`

第一阶段建议动作：

1. 进入 `report-web` 优先路线
2. 先做 `OpenClaw deployment options`、`best OpenClaw setup for beginners`、`OpenClaw Windows install WSL2`
3. 不优先单独立工具产品

## 10. 示例二：AI 编程与 Agent 时代模型采购、路由、成本与地区选择

### 10.1 题目定义

题目：

`围绕 AI coding / agent 场景中的模型采购、路由、总拥有成本与地区差异，建立可持续更新的决策型内容与潜在工具候选`

### 10.2 总门禁评分摘要

| 维度 | 分数 | 简短说明 |
| --- | --- | --- |
| `Problem` | 3 | 采购、路由、降本、地区选择都是具体真实问题 |
| `Intent` | 3 | 明显是采购与高价值决策意图 |
| `Audience` | 3 | 个人开发者、小团队、采购与技术负责人都可细分 |
| `Market Context` | 2 | 国家、语言、支付、合规差异明显，但仍需继续细化 |
| `Evidence` | 3 | 官方定价页、地区说明、社区讨论、平台页都较丰富 |
| `Gap` | 3 | 官方能给价格，无法完整回答采购决策与路由取舍 |
| `Monetization` | 3 | 广告、咨询、工具订阅、数据订阅都较清晰 |
| `Updateability` | 3 | 价格、策略、平台、地区变化非常频繁 |

总分：`23 / 24`

总门禁结论：`Go`

### 10.3 内容机会评分摘要

| 维度 | 分数 | 简短说明 |
| --- | --- | --- |
| `Query / Distribution Fit` | 3 | 采购、便宜但够用、直连 vs 中转、国家选择等 query 明确 |

## 11. 当前边界与迭代原则

这份标准表当前必须保持 4 个边界：

1. 它是 `v1 opportunity gate` 的操作化版本，不是终局制度
2. 它服务于内容机会与产品机会判断，但不新增平级外部契约
3. 第一阶段默认外部交付仍以 `ReportSpec + report-web` 为主
4. 后续必须根据真实案例调分、调权、补锚点，而不是把当前版本写成不可变教条

当前最值得坚持的判断仍然是：

1. OpenFons 的核心不是“找词写页”，而是“把用户问题编译成机会”
2. 利基不是把大词写浅，而是把小而贵的小意图做透
3. 内容机会和产品机会必须分开看
4. 真正的权威感来自 thesis、比较矩阵、决策树、证据附录、更新时间线和边界说明
