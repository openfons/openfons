# 第一阶段采集工具兼容性验证

> 项目名：OpenFons
> 文档定位：第一阶段执行级补充文档
> 当前目标：把采集相关开源工具从“文档选型”推进到“真实安装与冒烟验证”
> 验证日期：2026-03-25

## 1. 为什么要做这件事

经过本轮讨论，第一阶段的重点需要进一步收口：

1. 先验证采集相关能力是否真的可用。
2. 先确认哪些开源工具可以直接复用，哪些只能借鉴。
3. 先把多平台采集的真实前置条件摸清楚，再决定主线依赖。

这意味着第一阶段不应再停留在“文档中写过这些工具”，而应进入“安装、导入、最小调用、记录问题、更新选型”的兼容性验证阶段。

## 2. 公允判断

“把当前规划文档里和外部检索到的相关工具都装起来试一遍”这个想法本身是对的，但应限制在采集主链相关工具，而不是把所有数据面、分析面、观测面和 UI 组件同时拉进第一阶段。

更合理的做法是：

1. 第一批优先验证采集直接相关工具。
2. 重服务组件与数据分析组件单独批次验证。
3. 用隔离验证目录和结果矩阵来沉淀结论，而不是直接视为主仓正式依赖。

## 3. 当前验证方式

当前仓库已新增一个隔离验证目录：

`labs/collector-compat`

其中包含：

1. Node 侧依赖清单与冒烟脚本
2. Python 侧依赖清单与冒烟脚本
3. 平台级来源冒烟脚本
4. 兼容性矩阵
5. 当前环境下的真实安装结果

### 4.2 已完成的真实运行级检查

1. `Playwright` 已安装 Chromium，并完成无头启动。
2. `yt-dlp-exec` 已成功返回 `yt-dlp` 版本号。
3. `ddgs` 已完成真实搜索查询并返回结果。
4. `gallery-dl` 已成功执行 CLI 版本检查。
5. `ddddocr` 已成功实例化 OCR 对象。
6. `twscrape` 已成功执行 CLI 并返回版本与 SQLite 运行时。
7. `camoufox` 已成功下载浏览器二进制、真实启动浏览器并打开 `https://example.com` 返回标题 `Example Domain`。
8. `HackerNews/API` 已成功返回 `topstories` 和 `item` 真实数据。
9. `V2EX` JSON API 已成功返回站点统计与热议主题。
10. `amazon-review-scraper` 已通过 `woot.com` 公开 AJAX 端点成功拿到真实 Amazon 评论摘要。
11. `MediaCrawler` 已在 `repos/MediaCrawler` 完成 `uv sync`、`uv run playwright install`、`uv run main.py --help`、`uv run main.py --init_db sqlite` 与 WebUI `/api/health` 验证。
12. `pinchtab` 已完成官方 Docker 镜像拉取与容器启动验证；容器内 `pinchtab health` 返回 `ok`，`pinchtab snap` 可返回快照。

### 4.3 已识别的重要兼容性细节

1. `yt-dlp-exec` 在 Windows 下安装时依赖 PATH 中存在 `python` 命令；当前机器使用 `uv` 管理的 Python，需要显式补 PATH。
2. `facebook-scraper` 在当前依赖组合下需要额外补 `lxml_html_clean`，否则会在导入阶段失败。
3. `firecrawl` 当前仅验证了 SDK 安装和导入，真实抓取仍需要进一步验证其服务前置条件和调用方式。
4. `twscrape`、`TikTokApi`、`praw`、`python-amazon-paapi` 的“真正可用”仍取决于账号、cookie、API key 或平台风控条件。
5. `python-amazon-simple-product-api` 虽然可安装和导入，但项目较老，是否适合作为当前主线还需要谨慎评估。
6. `MediaCrawler` 不是单包，而是完整的仓库级工程，覆盖 `xhs / dy / ks / bili / wb / tieba / zhihu` 等国内平台，更适合作为国内新媒体专用 worker 的结构参考。
7. `MediaCrawler` 的真实采集依赖登录态、Playwright / CDP 与用户浏览器环境；本轮验证刻意停在“安装 + 启动链路 + WebUI 健康检查”，未做带账号的真实抓取。
8. `MediaCrawler` 仓库 README、LICENSE 与代码头部都明确声明“非商业学习用途”，与 OpenFons 面向商业化内容变现的目标存在直接约束，因此不适合直接作为商业主线依赖。
9. `MediaCrawler` 的 `playwright install` 在当前环境里主 CDN 出现过 `400 GatewayExceptionResponse`，最终依靠备用镜像成功，说明浏览器准备过程对网络环境敏感。
10. `pinchtab` 当前官方 Docker 镜像默认带鉴权，宿主机直接请求 `/health` 会返回 `missing_token`；容器内调用正常，说明后续接入时需要把 token 管理纳入配置方案。
11. `pinchtab` 默认启用了基于 `security.idpi.allowedDomains` 的导航限制，直接访问 `https://example.com` 会被阻止，因此后续要么配置允许域名，要么按平台级策略放行。

### 4.4 第二批平台级真实抓取结果

1. `YouTube / yt-dlp` 已成功返回公开视频元数据。
2. `YouTube / youtube-comment-downloader` 已成功抓取公开视频评论。
3. `YouTube / ytcog` 在当前无 cookie 会话下搜索返回 `status=NOK`，说明它更依赖真实浏览器会话条件。
4. `Reddit` 公开 JSON 端点在当前环境下重复请求返回 `403`，说明即使不走官方 SDK，也可能被平台阻断。
5. `PRAW` 在无 Reddit 凭据时真实请求返回 `401`。
6. `TikTokApi` 在当前环境下创建公开用户会话超时，至少现阶段不能视为“开箱即用”。
7. `twscrape` 在没有账号池时会直接提示 `No active accounts`。
8. `facebook-scraper` 虽然完成了导入，但匿名公共页抓取返回的有效字段较弱。
9. `MediaCrawler` 在当前环境下完成了安装、CLI、数据库初始化与 WebUI 健康检查，但真实平台抓取仍卡在登录态与账号条件，因此更像“可落地的国内平台参考工程”，而不是“开箱即用的商业主线依赖”。

### 4.6 `pinchtab` / `camoufox` 补充结论

针对你特别关心的浏览器层组件，本轮补充验证结果如下：

1. `pinchtab` 支持 Docker，且本机已成功拉起官方镜像 `pinchtab/pinchtab:latest`。
2. 当前存在两个有效信号：你机器上已有一个健康运行中的 `papaweb-src-pinchtab`，本轮又新增了 `openfons-pinchtab-official` 官方容器。
3. `openfons-pinchtab-official` 在 `29867 -> 9867` 端口映射下状态为 `healthy`。
4. 容器内执行 `pinchtab health` 返回 `status=ok`，执行 `pinchtab snap` 能返回 `about:blank` 的可访问性快照，说明不是“假启动”。
5. 直接 `pinchtab nav https://example.com` 会被 `security.idpi.allowedDomains` 拦下，说明它默认安全策略较强，不是裸奔浏览器。
6. `camoufox` 暂未确认有同等成熟的官方运行时 Docker 路线；但在宿主机已真实启动成功，且实际打开页面返回标题。
7. 因为后续还要做账号登录、二维码扫码与持久会话，现阶段更推荐 `pinchtab` 走 Docker 服务化，`camoufox` 先走宿主机 Playwright 集成。

针对用户特别点名的 [`NanmiCoder/MediaCrawler`](https://github.com/NanmiCoder/MediaCrawler)，本轮在 `labs/collector-compat/repos/MediaCrawler`（验证时 HEAD 为 `e8b1868`）完成了仓库级验证：

1. `uv sync` 成功，依赖与虚拟环境可以正常解析。
2. `uv run playwright install` 最终成功，但主下载源曾返回 `400 GatewayExceptionResponse`，依赖备用镜像回退。
3. `uv run main.py --help` 可以正常输出 CLI 参数，证明主入口可启动。
4. `uv run main.py --init_db sqlite` 可以成功初始化 SQLite schema，证明主流程不只停留在帮助页。
5. `uv run uvicorn api.main:app` 后，`/api/health` 成功返回 `{"status":"ok"}`。
6. 仓库自带已构建的 WebUI 静态资源，可直接由 FastAPI 提供。
7. 当前仍未验证带登录态的真实抓取，因此对“小红书 / 抖音 / 快手 / B站 / 微博 / 贴吧 / 知乎”只能下“工程能装能起”的结论，不能下“采集稳定可商用”的结论。

## 5. 当前结论

截至 2026-03-25，OpenFons 第一阶段“先做采集工具兼容性验证”是合理且必要的，而且已经证明这条路线有现实价值，因为它在第一天就暴露出了文档里看不到的真实问题。

当前可以明确得出的判断是：

1. 浏览器自动化、视频下载、基础搜索、媒体下载、OCR 和部分平台 SDK 在当前环境下是可落地的。
2. `Hacker News` 和 `V2EX` 这类公开 JSON 源非常适合作为低成本 adapter 优先接入。
3. `Amazon` 方向需要拆成两条线：商品信息优先 API-first，书面评论可优先评估 `amazon-review-scraper` 这类轻量脚本路线。
4. `YouTube` 是当前最成熟的一条线；`Hacker News`、`V2EX` 和 `amazon-review-scraper` 也已经有正向信号。
5. `Reddit`、`X`、`TikTok`、`Facebook` 都存在更明显的账号、会话、风控或匿名访问限制。
6. 若进入多平台真实采集，账号、cookie、代理、风控和 API key 管理会成为更大的约束，而不是“能不能 pip install / npm install”。
7. `Camoufox` 应保留为高阻力站点兜底，不适合作为默认浏览器链路。
8. 第一阶段最值得先打透的仍是 `YouTube + Reddit + Hacker News + Amazon`，再逐步推进 `X + TikTok`。
9. `MediaCrawler` 对国内新媒体平台采集很关键，但它更适合做“国内平台适配器参考样板”和实验室基线，不适合未经改造直接进入 OpenFons 商业主线。
10. `pinchtab` 的 Docker 服务化路线已经可以进入候选底座，但必须把 token、allowedDomains 和实例安全策略当成一等配置项。
11. `camoufox` 现在可以确认“本机真实可用”，适合进入后续 `MediaCrawler + Camoufox` 的受控联调阶段。

## 6. 下一步建议

1. 补第二批验证：真实抓取级 smoke test，而不只是安装与导入。
2. 先围绕 `YouTube / Reddit / Hacker News / Amazon` 写最小 adapter。
3. 为 `cookie / account / api key / proxy` 单独设计受控配置方案。
4. 再决定哪些库进入主线依赖，哪些只留在实验室或策略层。
5. 如果要补国内新媒体链路，建议单列 `CN social` 轨道，自研 adapter，但参考 `MediaCrawler` 的平台组织、登录态保存与 WebUI/CLI 结构。
