# 开放源平台投资人说明

> 项目名：OpenFons
> 中文工作名：开放源平台
> 当前阶段：平台底座初步成型 / 外部执行条件待收口
> 状态说明：当前已完成一批关键平台底座与单案例交付闭环，但仍未形成完整商用产品，真实外部执行闭环仍受外部条件约束

## 1. 一句话说明

OpenFons 要做的不是一个更强的爬虫工具，而是一套面向企业研究、舆情监控、商品评价分析与情报生产的 source-native 平台：

`用户意图 -> 流量切入判断 -> 结构化任务 -> 确定性执行 -> 证据沉淀 -> 多形态交付`

目标态将同时支持自然语言、界面和 API 三种入口，但底层统一进入同一条任务编译与执行链路。

## 1.1 当前状态说明

为了避免误读，先明确两点：

1. 当前仓库不是已经商业化交付完成的成熟平台。
2. 但它也不再只是“只有蓝图的启动仓”，而是已经完成一批关键平台底座。
3. 下面文档中的很多能力描述，仍然属于目标态和产品路线，而不是“今天已经全部上线”。

## 2. 我们看到的问题

今天企业在做品牌监测、竞品分析、商品评价研究、热点事件跟踪时，普遍会遇到四类问题：

1. 信息源分散在微博、小红书、抖音、电商平台、新闻站点等多个渠道，人工收集成本高。
2. 传统爬虫工具只能“抓到数据”，不能把数据稳定转成可复用的研究资产。
3. 纯大模型工作流虽然灵活，但执行不可控、不可审计、成本波动大。
4. 很多报告没有证据链，难以复核，无法沉淀为长期可运营的情报资产。

这意味着市场真正缺的，不是一个新的采集脚本仓库，而是一套能把采集、分析、审计、交付连起来的平台。

## 3. OpenFons 的解决方案

OpenFons 的产品定义是：

> 用户通过自然语言、界面或 API 提交研究任务，系统先判断需求是否存在可执行、可交付、可获得流量或业务价值的切入角度，再将需求编译为结构化计划，调用 Skill 和 Worker 完成多源信息接入、采集执行、分析研判、证据沉淀与交付。

它的核心设计不是“让大模型直接做所有事情”，而是：

1. Agent 负责理解需求和编译计划。
2. Worker 负责确定性执行。
3. Evidence 负责作为最小证据真源。
4. Artifact 负责作为正式冻结的交付物。

这样既保留 AI 的灵活性，又保留企业级系统必须具备的可控性、可审计性和可复现性。

## 4. 目标态产品形态

OpenFons 不是单一形态产品，而是一个统一底座上的多入口平台。

### 4.1 用户入口

1. 自然语言入口：适合快速发起研究任务。
2. 控制面界面：适合像传统 SaaS 一样点击操作、查看运行、重试、取消。
3. API 入口：适合集成到企业现有系统。

### 4.2 平台核心

1. Control Plane：通过深度研究与机会判断先产出 `OpportunitySpec`，经确认后编译 `TaskSpec / WorkflowSpec`。
2. Execution Plane：调用 Skill 与 Worker 完成采集、分析和质量控制。
3. Data & Delivery Plane：沉淀 `Topic / TopicRun / SourceCapture / CollectionLog / Evidence / EvidenceSet`，再生成 `ReportSpec / Artifact`，输出报告、图表、导出文件、预警消息。

### 4.3 交付结果

1. 舆情日报与预警
2. 商品评价分析报告
3. 竞品对比研究报告
4. 面向公开分发的 Next.js 研究报告页与专题页
5. CSV / JSON / Markdown / PDF 导出
6. Dashboard 与长期 watch 结果

### 4.4 第一阶段外部交付与流量路径

第一阶段如果要尽快验证商业化与盈利，不应先做大而全 SaaS，而应先验证一条更短的外部交付闭环：

`用户问题 -> 深度研究 / 机会判断 -> User Confirmation -> 任务边界编译 -> 多源采集与留痕 -> EvidenceSet -> ReportSpec -> Next.js 报告页 -> SEO 流量验证`

这里有五个关键收口：

1. 用户的一句话只是研究 seed，不直接等于最终页面标题、slug 或主关键词。
2. 对外公开页面必须先判断是否存在真实搜索需求，而不是先写完报告再期待搜索流量自然出现。
3. 前置深度研究用于确定任务边界、证据要求和反证问题，不自动等于最终证据；公开结论必须回到 `EvidenceSet`。
4. 对应页面必须能回答明确用户问题，并具备足够公开证据、独立观点、独立图表和后续更新空间。
5. 第一阶段最先验证的不是“能生成多少页面”，而是“能否持续产出少量高质量、可索引、可复核、可更新的页面并获得真实点击与曝光”。
6. 发布后的成功判断应回到 Search Console、点击、曝光、查询词和页面停留等真实反馈，而不是只看模型生成质量。

## 4.5 当前已完成与正在建设

当前更接近下面这个状态：

1. 已完成：
   - 最小控制与交付入口
   - `AI procurement` 单案例 file-backed artifact delivery
   - `config-center` 的读、写、doctor、backup history 与 runbook 收口
   - `search-gateway`、browser runtime 与 crawler route resolution 的平台接入
2. 正在建设或待继续证明：
   - 更完整的 `Topic / TopicRun / SourceCapture / CollectionLog / Evidence / EvidenceSet / Artifact` 真机闭环
   - retrieval orchestration、长期 watch 与更强交付面
3. 尚未完整落地：
   - 成熟 SaaS 控制面
   - 成熟运维面
   - 长期 watch / 预警
   - 真实外部 runtime 完整验收

## 5. 为什么是现在

我们认为现在是这个项目的窗口期，原因有三点：

1. AI 让“意图输入”大幅简化，但企业依然需要稳定、审计和成本可控的执行系统。
2. 公开信息源越来越多，企业对品牌、市场、竞争和舆情的持续感知需求显著增长。
3. 市场上的很多产品要么偏“采集工具”，要么偏“聊天研究助手”，中间缺少一层“证据优先、可长期运营的平台化系统”。

OpenFons 试图补上的，正是这层中间市场。

## 6. 目标客户与典型场景

### 6.1 目标客户

1. 品牌与市场团队
2. 电商与产品团队
3. 咨询与研究机构
4. 企业战略、风险、公共事务团队
5. 有长期舆情、竞品、情报跟踪需求的中大型组织

### 6.2 核心场景

1. 品牌舆情监控：持续跟踪品牌声量、负面波动、热点话题。
2. 商品评价分析：提取质量、价格、服务、物流等方面级反馈。
3. 竞品情报研究：汇总多平台信息，形成可交付的竞品分析材料。
4. 热点事件跟踪：形成时间线、传播路径、情绪走向与预警。

## 7. 核心差异化

### 7.1 不是“采集工具”，而是“研究资产平台”

平台核心对象是 `Topic / TopicRun / SourceCapture / CollectionLog / Evidence / EvidenceSet / Artifact`，而不是一堆临时抓取结果。

### 7.2 不是“纯 Agent”，而是“Agent 决策 + Worker 执行”

大模型不直接驱动浏览器，不直接充当主执行层，避免高成本和低可控。

### 7.3 不是“只有聊天”，而是“聊天、界面、API 三入口统一”

无论入口来自哪里，最终都进入同一条编译链，不会出现一套给聊天、一套给后台的双轨系统。

### 7.4 不是“报告先行”，而是“证据先行”

所有交付物都需要回到 Evidence 证据集，适合严肃研究、审计与复核场景。

### 7.5 不是“一次性任务”，而是“长期 watch 与持续消费”

平台路线从第一天就围绕长期主题、周期运行、历史沉淀和门户化交付来规划。

## 8. 商业模式方向

当前更适合采用“开源底座 + 商业化能力”的组合路径。

### 8.1 开源层

1. 开源主仓和基础能力
2. 统一契约、基础 Worker、基础 Skill
3. 社区可运行版本和标准部署骨架

### 8.2 商业层

1. SaaS 托管版本
2. 私有化部署与企业支持
3. SEO 友好的研究报告站、专题页与行业模板
4. 高级控制面与运维能力
5. 高价值连接器、策略中心、审计与治理能力
6. 行业模板和报告交付能力

## 9. 护城河判断

OpenFons 的护城河不只来自某一个模型或某一个抓取器，而来自四层叠加：

1. 多源适配与执行策略积累
2. 稳定的任务编译中间态与平台契约
3. 证据真源与交付真源体系
4. 长期 watch、质量门禁、运维治理带来的平台经验

换句话说，真正难复制的不是“能抓”，而是“能长期稳定地抓、分析、交付并沉淀”。

## 10. 当前阶段与近期里程碑

### 10.1 当前阶段

当前仓库更准确地说，处于“平台底座初步成型，但真实外部执行闭环仍未完全收尾”的阶段。

当前更准确的说法是：

1. 已有明确产品方向与技术路线。
2. 已有最小控制面、交付面、artifact delivery 与 config-center 平台底座。
3. 当前最大 blocker 来自真实外部 runtime 条件，而不是平台核心代码完全缺失。
4. 尚未进入“完整商业产品已上线”的阶段。

### 10.2 第一阶段目标

1. 建立前置与交付契约：`OpportunitySpec / TaskSpec / WorkflowSpec / ReportSpec`
2. 建立核心数据模型：`Topic / TopicRun / SourceCapture / CollectionLog / Evidence / EvidenceSet / Artifact`
3. 跑通最小执行闭环：`seed -> discovery -> fetch-http -> normalize -> quality -> publish-evidence`
4. 跑通第一条对外盈利闭环：`用户问题 -> 深度研究 / 机会判断 -> User Confirmation -> 任务边界编译 -> EvidenceSet -> ReportSpec -> Next.js 报告页`
5. 同时落最小控制面和运维面

### 10.3 第二阶段目标

1. Skill 注册与调度
2. 更多来源适配器
3. 报告与交付能力
4. 长期 watch 与预警能力

## 11. 融资使用建议

如果进入融资阶段，本轮资金更适合投入以下方向：

1. 核心平台工程能力：契约、数据模型、执行链路、测试护栏
2. 关键来源适配和反爬对抗能力
3. 控制面与运维面产品化
4. 质量门禁、审计、Evidence 体系建设
5. 首批商业化场景验证与行业模板沉淀

## 12. 本项目当前最值得关注的判断

对投资人而言，OpenFons 的价值不在于“是否又做了一个采集器”，而在于它能否成为：

1. 企业研究任务的统一输入层
2. 多源公共信息的统一执行层
3. 可审计证据的统一沉淀层
4. 舆情、研究、情报交付的统一平台层

如果这个定位成立，它对应的不是单点工具市场，而是一个可以向研究平台、企业 SaaS 和行业解决方案延展的平台级机会。
