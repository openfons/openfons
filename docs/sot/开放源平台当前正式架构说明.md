## 2. 一句话定义
OpenFons 是一个面向研究、情报、舆情与内容交付场景的 source-native 平台：它先把真实用户问题编译为结构化 `OpportunitySpec` 与 `TaskSpec / WorkflowSpec`，再通过受控执行沉淀为可复核的 `EvidenceSet`，最后生成 `ReportSpec` 与可交付的 `Artifact`。

当前第一阶段（v1）的唯一北极星是：
`OpenFons v1 的唯一北极星：把真实用户问题稳定编译成可验证的 OpportunitySpec、EvidenceSet 和 ReportSpec，先交付可盈利的网页报告，并为视频、文章与工具/订阅机会保留统一真源。`

当前正式主线对应的最小闭环仍是：
`control-web -> control-api -> report-web`

它当前证明的是受限子集：
`OpportunityInput -> OpportunitySpec -> TaskSpec / WorkflowSpec -> ReportSpec -> report-web artifact shell`

它尚未正式证明完整的：
`TopicRun / Source Captures / Collection Logs -> EvidenceSet -> final Artifact`

当前主线还必须按以下方式理解：

1. `Planning Swarm` 在 v1 是前置深度研究 / 多 Agent 规划阶段，不是轻量搜索便民层。
2. 前置研究首先产出 `OpportunitySpec`、未知项、反证问题与证据要求，不自动等于最终证据或最终结论。
3. 规划阶段的搜索与发现必须留下可审计 discovery captures，优先以 `SourceCapture / CollectionLog` 兼容对象落盘。
4. `ReportSpec` 应基于 `Evidence / EvidenceSet` 与 findings 生成；`report-web` 当前证明的是报告壳，不代表完整 truth chain 已经成立。

## 3. 当前正式技术主线
截至 2026-03-28，OpenFons 当前正式主线如下：

1. 主工程语言：TypeScript / Node.js
2. Monorepo：pnpm workspace
3. 前端：React + Vite + TanStack Router
4. API 边缘服务：Hono
5. 契约与校验：Zod
6. 测试体系：Vitest
7. 本地开发与验证：PowerShell + Docker Compose

补充约束：
1. Python 目前不是主工程语言，只保留为后续采集适配、OCR、模型推理或辅助 worker 的候选辅助语言。
2. Rust 目前不是主线实现语言，只保留为未来性能热点或底层能力的候选方案。
3. 不应再把早期的 `Python + FastAPI + Next.js` 方案表述成当前正式实现。

## 4. 当前仓库结构与边界
当前仓库按以下一级目录分层：

```text
openfons/
  apps/
  services/
  packages/
  config/
  data/
  infra/
  docs/
  Memory/
  labs/
  tests/
  scripts/
```

当前正式边界如下：
1. `apps/`：面向人的应用壳。当前已落地 `control-web` 与 `report-web`。
2. `services/`：面向系统的服务。当前已落地 `control-api`。
3. `packages/`：共享契约、模型与基础能力。当前核心为 `contracts`、`domain-models`、`shared`。
4. `config/`：声明式配置与模板。当前使用 `sources`、`policies`、`templates`。
5. `tests/`：跨层测试。当前正式使用 `contract`、`integration`、`smoke`。
6. `scripts/`：仓库级开发与校验脚本。当前正式使用 `dev` 与 `check`。
7. `infra/`：本地开发与最小 Docker 基线。当前正式使用 `docker` 与 `local-dev`。
8. `docs/`、`Memory/`、`labs/`：是仓库中的一等资产，但不进入运行时依赖链。

明确约束：
1. `apps/*` 不直接导入 `services/*` 源码，只通过 API 与契约协作。
2. `docs/`、`Memory/`、`labs/` 不作为运行时依赖。
3. `data/` 不是长期证据仓，只用于 fixtures、samples、seeds 与临时开发辅助。

## 5. 当前已落地的最小闭环
截至当前主线，以下内容已经正式落地：

### 5.1 应用与服务
1. `apps/control-web`：用于创建并编译机会，证明控制面入口成立。
2. `services/control-api`：负责最小编译流程，提供健康检查、机会创建、编译与报告读取接口。
3. `apps/report-web`：负责渲染最小报告壳，证明交付面入口成立。

### 5.2 共享层
1. `packages/contracts`：当前控制面与交付面的正式契约来源。
2. `packages/domain-models`：当前领域对象与证据对象基础模型。
3. `packages/shared`：少量跨层共享基础能力。

### 5.3 基础配置与脚本
1. `config/sources/default.json`
2. `config/policies/default.json`
3. `config/templates/report-default.md`
4. `scripts/dev/start-all.ps1`
5. `scripts/check/verify.ps1`
6. `infra/docker/control-api.Dockerfile`
7. `infra/docker/compose.yaml`
8. `infra/local-dev/README.md`

### 5.4 当前已验证的闭环能力
当前主线已经验证：
1. 可以创建一个机会对象。
2. 可以把机会编译为任务、工作流和模板化报告壳。
3. 可以通过 `report-web` 读取并渲染报告壳。
4. 可以通过统一验证脚本完成 `lint + typecheck + test + build + docker compose config` 校验。

## 6. 当前真相源
当前正式真相源分为两层：

### 6.1 控制面与交付面真相源
以下对象是当前正式对外契约与编译真相源：
1. `OpportunityInput`
2. `OpportunitySpec`
3. `TaskSpec`
4. `WorkflowSpec`
5. `ReportSpec`
6. `CompilationResult`

当前约束：
1. 控制面的编译输出应围绕上述契约稳定演进。
2. 新能力接入时，优先扩展契约而不是临时绕过契约。
3. 面向下游系统时，以这些结构化对象为边界，而不是以自由文本为边界。
4. `OpportunitySpec` 是用户确认前的唯一前置外部契约；`ReportSpec` 是 evidence / findings 下游交付契约。

### 6.2 领域层真相源
以下对象是当前正式领域模型基础：
1. `Topic`
2. `TopicRun`
3. `SourceCapture`
4. `CollectionLog`
5. `Evidence`
6. `EvidenceSet`
7. `Artifact`

补充说明：当前主线尚未把完整 `TopicRun`、长期 watch、分析流水线全部落地，但相关扩展必须继续围绕证据优先与可复核对象来设计。

## 7. 当前明确不做什么
当前正式主线明确不做以下事情：

1. 不提前铺开大量 `worker-*` 服务目录并把空壳当实现。
2. 不把 `ops-web`、`k8s`、`terraform`、重基础设施方案提前当成第一阶段必须项。
3. 不把 Python/FastAPI 旧路线继续表述为当前正式主线。
4. 不把 `docs/`、`Memory/`、`labs/` 中的资料直接纳入运行时代码依赖链。
5. 不让大模型直接绕过控制面，成为默认执行层。
6. 不在没有明确契约的情况下，让 UI、脚本或实验代码直接驱动未受控执行。

## 8. 近中期演进原则
当前主线后续演进应遵循以下原则：

1. 继续沿 TypeScript / pnpm / `apps + services + packages` 主线扩展，而不是重新切回旧技术路线。
2. 先扩展契约、测试和最小服务边界，再扩展 worker、registry、ops 能力。
3. 只有在真实业务压力出现后，才引入更重的服务拆分、部署形态或性能语言。
4. Python 与 Rust 若被重新引入，必须承担边界清晰、可替换、可审计的局部职责，而不是重新成为主线。
5. 任何会改变正式主线的技术切换，都应通过单独 ADR / RFC 或同等级文档收口。
