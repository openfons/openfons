# 开放源平台技术团队说明

> 项目名：OpenFons
> 中文工作名：开放源平台
> 文档定位：面向内部技术团队的统一技术说明、架构边界说明与开源技术清单
> 状态说明：本文件区分“当前主线假设”“按需启用备选”“参考借鉴”，避免把目标态或历史方案误读为既定事实

## 1. 文档目的

本文件用于统一团队对以下问题的理解：

1. 我们到底在做什么系统
2. 哪些是当前已冻结的架构原则
3. 哪些能力是第一阶段必须落地的
4. 当前规划中明确采用、按需启用、参考借鉴了哪些开源技术

本文件优先基于当前仓库的根规划文档整理，不把临时脑暴或尚未进入主规划的探索项目写成既定事实。

## 1.1 当前工程阶段

当前仓库仍处于 Greenfield Bootstrap 阶段，意味着：

1. 当前最确定的是架构边界、真源边界、迁移边界和第一阶段范围。
2. 当前最不应该误读的是“目标态能力已经全部实现”。
3. 所有技术清单都必须区分：当前主线假设、按需启用备选、参考借鉴。

## 1.2 编排决策状态

当前规划资料里同时存在 `Airflow` 和 `Temporal` 两条历史来源，因此这里明确收口如下：

1. 对 OpenFons 当前主线假设，按现有 SoT 采用 `Airflow` 作为当前唯一编排真源。
2. `Temporal` 保留为历史方案中的强参考和后续演进备选，不作为当前 bootstrap 默认编排基线。
3. 如果未来要从 `Airflow` 切换到 `Temporal`，必须通过单独的 ADR / RFC 和迁移评审，不允许两套编排长期并行。

## 2. 项目定位

OpenFons 要做的不是一个更强的抓取脚本仓库，而是一个 source-native 平台：

`自然语言 / 界面 / API -> 意图澄清 / 流量判断 -> TaskSpec / WorkflowSpec -> Skill / Worker 执行 -> Evidence 沉淀 -> Artifact 交付`

目标场景包括：

1. 品牌舆情监控
2. 商品评价分析
3. 竞品情报研究
4. 热点事件追踪
5. 长期 watch 与周期性交付

## 3. 北极星与非目标

### 3.1 北极星

我们要建立的是：

1. 可把用户意图编译为结构化任务的控制层
2. 可稳定执行多源采集与分析的执行层
3. 可沉淀 Topic / Evidence / Artifact 的证据层
4. 可输出多种交付物的交付层

### 3.2 明确非目标

以下方向不应作为主架构：

1. 不做“大模型直接驱动浏览器”的默认执行架构
2. 不做“一个万能 Agent 统管所有任务”的黑盒系统
3. 不做“只靠脚本和临时工具拼起来”的生产链路
4. 不把 `gold_tool_entities` 之类的兼容视图当最终真源

## 4. 架构总原则

### 4.1 四大平面

1. `Access Plane`
   IM / Web / API / 外部系统入口
2. `Control Plane`
   Intent Router、Intent Structuring、Planning Swarm、Opportunity Forecast / Traffic-Fit Evaluator、Opportunity Judge、Task Compiler、Policy Gate、Scheduler
3. `Execution Plane`
   Source Registry、Skill、Worker、风控对抗与适配层
4. `Data & Delivery Plane`
   Topic、TopicRun、Evidence、EvidenceSet、Artifact、Renderer、Portal

### 4.1 v1 唯一北极星

`OpenFons v1 的唯一北极星：把真实用户问题稳定编译成可验证的 OpportunitySpec、EvidenceSet 和 ReportSpec，先交付可盈利的网页报告，并为视频、文章与工具/订阅机会保留统一真源。`

### 4.2 统一编译链

无论入口来自哪里：

1. Skill
2. Web 界面
3. API

最终都必须进入同一条链路：

`User Input -> Intent Structuring -> Planning Swarm -> Opportunity Judge -> OpportunitySpec -> User Confirmation -> Task Compiler -> TaskSpec -> WorkflowSpec -> Worker Execution -> EvidenceSet -> ReportSpec -> Artifact`

### 4.2.1 Opportunity Forecast Layer 与 v1 角色拆分

当入口任务的目标不是“立刻执行采集”，而是“先判断是否值得做成公开页面、研究任务或长期 watch 主题”时，控制面前段应额外补一层：

`Intent Structuring -> Planning Swarm -> Opportunity Judge -> OpportunitySpec -> User Confirmation`

这里的 `Planning Swarm` 不是模拟人格世界，也不是一组自由发挥的大模型，而是一组受控角色。

当前建议的 `v1` 最小拆分是：

1. `Intent Clarifier`
   把用户原话变成结构化意图、受众、市场和语言候选。
2. `Demand Analyst`
   判断真实需求、趋势速度和讨论强度。
3. `Competition Analyst`
   判断 SERP 格局、内容缺口和切入难度。
4. `Monetization Analyst`
   判断商业意图、广告、affiliate、咨询或 B2B 转化价值。
5. `Opportunity Judge`
   对前 4 个角色的输出做统一裁决，输出唯一的 `OpportunitySpec`。

这里需要明确两点：

1. 这不是声称“4 个角色永远是最终最佳实践”，而是当前最稳的 `v1` 最小合理拆分。
2. 如果后续真实案例证明某两个角色长期高度重叠，或缺少地区、语言、内容差距等独立判断维度，应允许合并、拆分或新增角色。

与之对应，`Opportunity Forecast Layer` 使用的信号面也不应被写成封闭列表。当前更合理的理解是：

1. 先定义 `v1 signal families`
2. 先用这些默认信号面跑通机会判断闭环
3. 再根据真实案例补充、删减或重配权重

也就是说，角色集合和信号集合都应被视为：

`当前最稳的 v1 假设，而不是终局静态制度`

也就是说，当前主线不是“先证明理论上绝对最优”，而是：

`先建立可回放、可评估的机会判断层 -> 再用真实案例迭代角色集合`

### 4.2.2 最佳实践：契约层、角色层、步骤层分离

为了避免命名混乱，控制面应显式区分 3 个层级：

1. `契约层`
   对下游暴露的结构化对象，使用名词，例如 `OpportunitySpec`、`TaskSpec`、`WorkflowSpec`、`ReportSpec`。
2. `角色层`
   负责判断和裁决的受控角色，使用角色名，例如 `Intent Clarifier`、`Demand Analyst`、`Competition Analyst`、`Monetization Analyst`、`Opportunity Judge`。
3. `步骤层`
   流程中的具体动作，使用动词短语，例如 `structure_intent`、`run_demand_analysis`、`run_competition_analysis`、`run_monetization_analysis`、`judge_opportunity`、`confirm_user_scope`。

边界规则如下：

1. `IntentSpec`、`DemandResearchBrief`、`OpportunityMap`、`ApprovalPayload` 不再作为与 `OpportunitySpec` 平级的外部契约；更适合作为 `OpportunitySpec` 的内部字段或子对象。
2. `clarify_intent`、`demand_research` 这类名称如果继续保留，只应用作 planning workflow 的内部步骤或遥测标签，不应用作对下游暴露的最终契约名。
3. 下游的 `Task Compiler`、调度层、审计回放和 UI 默认只消费一份前置规划结果：`OpportunitySpec`。

### 4.3 角色边界

1. Agent 负责理解、澄清、判断切入角度、拆解、编译与策略决策
2. Worker 负责确定性执行
3. Skill 是可复用能力单元
4. LLM 浏览器自动化只能作为受控兜底 Skill，不是默认执行层

## 5. 真源模型

### 5.1 控制面编译真源

1. `OpportunitySpec`：机会判断层对下游暴露的唯一外部契约
2. `TaskSpec`：这次任务是什么
3. `WorkflowSpec`：这次任务怎么跑
4. `ReportSpec`：这次任务怎么交付

其中 `IntentSpec`、`DemandResearchBrief`、`OpportunityMap`、`ApprovalPayload` 应作为 `OpportunitySpec` 的内部字段或子对象保留，而不是与其平级对外暴露。

### 5.1.1 面向 SEO 报告模式的补充编译字段

当任务目标是对外发布、获取搜索流量或验证内容变现时，`OpportunitySpec / TaskSpec / ReportSpec` 至少还应显式表达：

1. `audience / geo / language`
   面向谁、哪个地区、什么语言。
2. `searchIntent`
   这页到底回答什么搜索意图，是教程、对比、选型、价格、风险，还是最佳实践。
3. `keywordSeed`
   用户原始提问或初始主题词。
4. `keywordCluster`
   编译后得到的核心搜索词、次级关键词和同义变体。
5. `trafficFit`
   这个主题是否值得做成公开页面，依据包括需求强度、可持续性、竞争度、证据可得性和更新空间。
6. `angle`
   最终切入角度，例如“普通用户怎么部署”“自托管 vs 托管”“最低成本方案”“安全风险与代价”。
7. `evidenceRequirements`
   这类页面最低需要哪些来源和证据类型，避免“有词无证据”。
8. `productOpportunityHints` / `expansionCandidates`
   如果前置判断还识别出工具站、数据产品或订阅潜力，当前阶段应先把这些结果保留为 `OpportunitySpec` 的内部扩展字段，而不是直接新增平级外部契约。

这里要保持契约边界：

1. 第一阶段默认外部交付仍以 `ReportSpec` 和 `report-web` 为主。
2. 如果未来确实需要引入专门的工具交付契约，例如 `ToolSpec`，必须通过单独 ADR / RFC 收口，而不是在案例文档里先行扩散。

### 5.2 研究与交付真源

1. `Topic`
2. `TopicRun`
3. `SourceCapture`
4. `CollectionLog`
5. `Evidence`
6. `EvidenceSet`
7. `Artifact`

### 5.3 真源规则

1. `Source Registry` 是来源配置真源
2. `OpportunitySpec / TaskSpec / WorkflowSpec / ReportSpec` 是控制面编译真源
3. `Topic / TopicRun / SourceCapture / CollectionLog / Evidence / EvidenceSet / Artifact` 是研究与交付真源
4. 兼容视图只能作为迁移层，不得冒充最终真源

## 6. 第一阶段工程范围

### 6.1 必须优先落地

1. `packages/contracts`
2. `packages/domain-models`
3. 最小 shared 基础模块
4. 最小执行闭环
5. 最小控制面 `control-web`
6. 最小运维面 `ops-web`
7. 面向外部交付的 `report-web`

### 6.2 第一阶段执行闭环

建议保持克制，只跑通：

`seed -> discovery -> fetch-http -> normalize -> quality -> publish-evidence`

但如果本阶段要验证基于 SEO 的 Next.js 报告页变现，则控制面在进入 `seed` 之前还必须补一层前置编译：

`intent-structuring -> planning-swarm -> opportunity-judge -> compile-opportunity-spec -> user-confirmation -> task-compiler -> seed`

### 6.3 暂缓事项

第一阶段不应同时做大的浏览器链路、多媒体链路、复杂门户、大而全后台。

## 8. 当前主规划采用与按需扩展策略

### 8.1 当前主规划

当前主规划强调：

1. 先把契约和数据模型立住
2. 先跑通确定性执行闭环
3. 先保留控制面和运维面的最小可用界面
4. 先让 Skill、界面和 API 共用同一条编译层
5. 先把面向公开报告模式的流量判断层和关键词角度选择层立住
6. 先把“机会预测层”做成可评估的 `v1`，不要一开始就追求理论上最复杂的多 Agent 架构
7. 如果机会判断还识别出数据产品、工具站或订阅潜力，先记录为扩展候选，不要过早破坏第一阶段以报告页验证闭环为主的边界

### 8.2 按需启用

以下能力属于按条件启用，不应在第一阶段预埋复杂度：

1. Kafka
2. 更强检索与分析组件
3. 浏览器自动化主链路
4. 更复杂的门户与商业化边界能力

## 10. 开源技术使用约束

### 10.1 一律先区分“主规划”与“参考”

任何组件进入主线前，都必须先回答：

1. 它是第一阶段默认启用，还是按需启用
2. 它是正式组件，还是参考项目
3. 它是否会改变当前的真源边界和执行边界

### 10.2 一律坚持单一真源

1. 编排真源不能双轨并行
2. 来源配置真源不能双轨并行
3. 研究与交付真源不能退化成兼容视图

### 10.3 一律先契约后实现

1. 先定义 `TaskSpec / WorkflowSpec / ReportSpec`
2. 再定义 `Topic / TopicRun / Evidence / Artifact`
3. 再推进 Worker、Skill、UI 和 API

### 10.4 一律禁止“AI 直接统管执行”

LLM 只能做意图理解、任务分解、策略规划和中间态编译，不得直接成为默认浏览器执行主链。

## 11. 近期落地建议

### 11.1 技术优先级

1. 先把 `contracts` 和 `domain-models` 立住
2. 先做最小控制面与运维面骨架
3. 先跑通最小 Worker 闭环
4. 先补测试和可观测护栏
5. 先落地 `Intent Structuring -> Planning Swarm -> Opportunity Judge` 的 `v1` 闭环，再根据真实案例调整角色拆分

### 11.2 不建议的做法

1. 不要一开始就同时接入所有来源
2. 不要一开始就引入所有可选组件
3. 不要为了演示效果，把参考项目直接塞进主仓变成“伪实现”
4. 不要在没有中间态契约的情况下先堆界面
5. 不要把用户原话直接当公开页面标题或主关键词，必须先经过需求澄清和流量切入判断
6. 不要在没有真实案例反馈之前，把“四角色拆分”写成不可变教条
7. 不要把 `MiroFish` 这类“已知主题后的模拟与报告生成”逻辑，直接等同于 OpenFons 所需的“生成前机会判断层”
