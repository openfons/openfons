# Docs Index

`docs/` 是 OpenFons 当前文档体系的总入口。

如果你现在要快速进入项目，先记住这条主线：

1. `docs/plan1`
   第一阶段执行与验证材料
2. `docs/plan2`
   第二阶段架构、机会判断、案例与页面规划
3. `docs/sot`
   跨阶段长期真源
4. `docs/references`
   参考材料与原始输入
5. `docs/superpowers`
   设计稿、执行计划、迁移记录等过程文档

## 五层目录怎么理解

### `docs/plan1`

用来回答：
- 第一阶段到底验证了什么
- 哪些采集能力已经真实安装和跑通
- 当前执行边界在哪里

适合：
- 看第一阶段落地证据
- 查兼容性验证结论
- 回看阶段执行记录

推荐先读：
- `docs/plan1/README.md`
- `docs/plan1/第一阶段采集工具兼容性验证.md`

### `docs/plan2`

用来回答：
- 我们下一阶段的最佳实践架构是什么
- 为什么要借鉴 `MiroFish` 和 `DeerFlow`
- 机会门禁、评分体系、案例和页面规划怎么落地

适合：
- 做产品与架构讨论
- 做案例推演
- 做页面和机会规划

推荐先读：
- `docs/plan2/README.md`
- `docs/plan2/openfons-best-practice-architecture-2026-03-27.html`
- `docs/plan2/MiroFish与DeerFlow深度对比及对OpenFons的借鉴建议-2026-03-27.md`

### `docs/sot`

`sot` 是 Source of Truth。

这里的文档不属于单一阶段，而是跨阶段长期有效的总纲。阶段文档和总纲冲突时，以这里为准。

适合：
- 对齐技术真源
- 对齐对外叙事
- 判断阶段文档是否偏离主线

推荐先读：
- `docs/sot/README.md`
- `docs/sot/开放源平台技术团队说明.md`
- `docs/sot/开放源平台投资人说明.md`

### `docs/references`

这里放参考材料、原始输入、外部整理稿。

它们可以提供背景、证据和启发，但默认不当成“当前正式计划真源”。

适合：
- 回看原始材料
- 查外部输入来源
- 做辅助比对

推荐先读：
- `docs/references/README.md`
- `docs/references/raw-inputs/README.md`

### `docs/superpowers`

这里放的是工作过程文档：
- 设计稿
- 实施计划
- 迁移说明
- 执行辅助文档

它们很重要，但它们描述的是“当时怎么设计、怎么执行”，不等于当前正式归档位置。

适合：
- 回看方案演化过程
- 理解当时怎么落地
- 查计划与执行记录

推荐先读：
- `docs/superpowers/README.md`

## 新人最快阅读顺序

1. 先读 `docs/sot/开放源平台技术团队说明.md`
2. 再读 `docs/plan1/第一阶段采集工具兼容性验证.md`
3. 再读 `docs/plan2/openfons-best-practice-architecture-2026-03-27.html`
4. 再读 `docs/plan2/MiroFish与DeerFlow深度对比及对OpenFons的借鉴建议-2026-03-27.md`
5. 最后按任务需要进入 `docs/references` 或 `docs/superpowers`

## 关于其他目录

`docs/context` 和 `docs/plan` 当前不作为这轮正式五层主入口的一部分。

如果后续继续清理旧材料，可以再决定是否把它们并入现行结构，或明确归档定位。
