# 仓库落地建议与参考代码

## 1. 对本仓库的直接建议

### 1.1 前端目录

建议平台前端直接落在：

- `apps/web`

原因：

- 目录已经预留
- 语义清晰
- 适合承接统一后台入口

### 1.2 平台 API 目录

建议平台正式 API 继续落在：

- `services/query-api`

原因：

- 当前查询与触发接口已经在这里
- 便于在现有服务基础上逐步扩展 `/api/v2/*`

### 1.3 契约收敛位置

建议 DTO、接口契约、平台级类型统一收敛到：

- `packages/contracts`

原因：

- 避免前后端各自维护一套结构
- 方便前端直接复用类型

### 1.4 三层策略平台化原则

建议直接复用：

- `apps/crawler/src/three-layer-runner.mjs`

已有字段定义与语义，不建议重新发明一套名字相似但不兼容的字段。

### 1.5 清洗规则平台化原则

建议直接复用：

- `apps/crawler/src/clean.mjs`

已有规则语义，先以“规则集承载”和“预览能力”的方式平台化。

## 2. 不建议做的事

- 不要第一阶段就重写所有 worker
- 不要把前端长期暴露在底层执行参数上
- 不要把用户审计和机器事件混为一类
- 不要把 VTable 当整个平台框架
- 不要继续把三层策略和清洗能力长期停留在 CLI 侧
