# 前端平台方案

## 1. 前端总体方向

平台前端建议采用：

- 前端骨架：`TanStack Router/Start`
- 后台 UI：`Ant Design`
- 重数据页面：`VTable`

这三者不是互相替代，而是分工协作。

## 2. 为什么这样组合

### 2.1 TanStack Router/Start 负责什么

它主要负责整个平台的“骨架”和“页面组织”。

更通俗地说，它负责：

- 页面结构
- 菜单路由
- 新建任务到详情页的跳转
- 地址栏参数同步
- 页面级加载状态
- 页面级错误态

它适合做：

- 仪表盘
- 新建任务页
- 运行中心
- 运行详情页
- 来源站点管理页
- 审计日志页

它不负责重表格渲染性能，也不负责整套视觉组件。

### 2.2 Ant Design 负责什么

它主要负责后台常见交互组件。

更通俗地说，它帮你快速搭好：

- 表单
- 按钮
- 输入框
- 选择器
- 抽屉
- 弹窗
- Tabs
- Steps
- 标准后台布局

对于“用户下任务”“编辑来源站点”“配置策略”“查看详情页”这类页面，它很合适。

### 2.3 VTable 负责什么

VTable 不适合做整个平台骨架，但很适合做特别重的数据页。

例如：

- 任务明细大表
- 运行事件时间线矩阵
- 原始数据对比页
- 清洗预览页
- 导出预览页

也就是说：

- Router/Start 管页面骨架
- Ant Design 管后台组件
- VTable 管重表格性能

## 3. 为什么不直接全站都用 VTable

原因很简单：

- 你们要做的是后台管理平台，不是单一表格应用
- 平台需要菜单、路由、详情页、配置页、表单页、弹窗流程
- VTable 强在大表格，不强在全站应用骨架

所以最合理的用法是：

- 把 VTable 当重表格引擎
- 不把 VTable 当整个平台基座

## 5. 关键页面方案

### 5.1 新建任务页

目标：

- 让用户正式下达平台任务

核心功能：

- 选择来源站点
- 选择任务模板
- 选择三层策略
- 设定目标条数
- 设定 discovery / fetch 参数
- 选择质量策略
- 选择清洗规则
- 提交运行

### 5.2 运行中心页

核心功能：

- 查看 run 列表
- 按状态筛选
- 查看当前活跃 run
- 取消 run
- 克隆 run
- 重试 run

### 5.3 运行详情页

核心功能：

- 查看 summary
- 查看 job 进度
- 查看 task 状态分布
- 查看 pipeline 时间线
- 查看失败原因聚合
- 查看原始产物入口

### 5.4 来源站点页

核心功能：

- 编辑来源站点配置
- 校验配置
- 预览 discovery

### 5.5 三层策略页

核心功能：

- 维护 aggressive / conservative / fallback 组合
- 模拟某个站点的执行效果

### 5.6 清洗规则页

核心功能：

- 配置去重规则
- 预览清洗结果
- 对比规则变更前后差异

注意：

- 当前先规划，不先改现有清洗逻辑
- 后续以规则集方式平台化

### 5.7 工具数据页

核心功能：

- 搜索
- 筛选
- 查看实体详情
- 查看证据
- 查看历史版本

## 6. 前端页面和后端能力的关系

前端不是单独做页面，而是要和后端控制面一起落地。

对应关系如下：

- 新建任务页 对应 `runs`、`task-templates`、`strategy-profiles`
- 运行中心页 对应 `runs`、`jobs`、`tasks`
- 运行详情页 对应 `timeline`、`events`、`artifacts`
- 来源站点页 对应 `source-sites`
- 三层策略页 对应 `strategy-profiles`
- 清洗规则页 对应 `clean-rule-sets`
- 工具数据页 对应 `tools`、`evidence`、`history`

## 7. 第一阶段前端建议

第一阶段不要贪多，先把最核心的三页做出来：

- 新建任务页
- 运行中心页
- 运行详情页

这样能最快验证平台控制面是否真的成型。
