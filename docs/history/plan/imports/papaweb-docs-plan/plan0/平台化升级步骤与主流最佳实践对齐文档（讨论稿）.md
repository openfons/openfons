## 1. 二次 Review 结论（先看）

1. 方向确认：采用“多服务拆分”是主流，且与 Plan0 文档一致。  
2. 节奏确认：采用“小闭环先跑通，再扩展”是主流，适合当前阶段。  
3. 部署确认：Docker Compose 适合作为开发/联调/单机交付基线；面向高可用生产，需规划 Kubernetes 化。  
4. 迁移确认：优先走“渐进替换（Strangler Fig）”，避免一次性推翻重写。

## 2. 外部主流最佳实践对标（摘要）

### 2.1 Docker / Compose

1. 应用应按“多容器应用模型”组织，而不是单容器全包。  
2. 容器边界建议“一容器一关注点”（rule of thumb）。  
3. 生产应采用多 Compose 文件分层（基础 + 环境覆盖），并通过 `-f` 组合。  
4. 通过 `profiles` 按场景启停可选服务（如 Grafana）。  
5. 服务间应使用服务名发现，不依赖固定 IP。

### 2.2 n8n（对“小闭环再扩展”参考价值高）

1. `queue mode` 是官方推荐的高扩展模式。  
2. 拓扑是“主实例 + worker + Redis + DB”，worker 负责执行，主实例负责触发/协调。  
3. 可通过增加/减少 worker 做水平扩缩。  
4. 官方建议 queue 模式优先使用 Postgres；不建议 SQLite。  
5. Webhook 可独立扩展，且不建议把主进程放入 webhook 流量池。

### 2.3 Airflow

1. 官方明确：Docker Compose 示例是 quick-start，不是生产级方案。  
2. 生产部署官方主推 Helm + Kubernetes。  
3. 结论：当前可用 Compose 建立平台能力与流程边界，后续再迁移到 K8s。

### 2.4 Dify（对“多服务拆分”参考价值高）

1. 官方仓库 Compose 采用多服务拆分：`api / worker / worker_beat / web / plugin_daemon / redis ...`。  
2. 配置通过 `.env` + compose 组合管理。  
3. 结论：主流开源平台普遍采用“控制面/执行面/依赖面”分拆，不走单体容器。

### 2.5 渐进式现代化

1. Fowler 的 Strangler Fig 模式强调“逐步替换、持续交付、降低切换风险”。  
2. AWS 现代化指南也将其作为风险可控的迁移策略。  
3. 与你们当前“脚本退役 + Worker 接管”的路线一致。

## 3. 对 papaweb 的统一升级策略（建议冻结）

1. 架构形态：多服务拆分。  
2. 交付节奏：小闭环先通（`fetch -> normalize -> quality -> publish`），再扩能力。  
3. 编排边界：`Airflow` 为唯一编排真源。  
4. 发布边界：仅 `publish_worker` 允许写 Gold 发布面。  
5. 数据边界：Worker 不直写 ClickHouse。  
6. 迁移方式：Strangler Fig 渐进迁移，不做大爆炸式重构。  
7. 组件启用：Kafka/OpenSearch/更多分析组件按触发条件启用，不预埋复杂度。

## 4. 升级步骤（只讨论步骤，不在本稿执行）

## 阶段 0：决策冻结（D0-D3）

1. 冻结三条红线：单编排真源、单发布写口、Worker 不直写 CH。  
2. 冻结目录策略：`services/*` 与 workspace 调整的最终方案（二选一一次性落地）。  
3. 冻结最小闭环接口：输入输出 schema、重试策略、幂等键规则。

验收产物：
1. 1 份 ADR：架构红线与例外审批机制。  
2. 1 份 ADR：目录/工作区迁移策略。  
3. 1 份接口契约草案（contracts）。

## 阶段 1：P0 小闭环上线（D4-D20）

1. 以 `base + collect + orchestrator` 完成可运行链路。  
2. 串通 `crawl_jobs/crawl_tasks/url_state` 状态机。  
3. 串通 `publish_worker` 门禁前置链路。  
4. 建立最小观测：Prometheus + Alertmanager + 基础仪表板。

验收口径：
1. 闭环任务可重复执行，失败可重放。  
2. 每个任务有 trace_id / 证据链定位信息。  
3. 非脚本路径可独立完成端到端发布。

## 阶段 2：质量门禁产品化（D21-D45）

1. 将 `coverage/confidence/freshness` 做成可配置门禁。  
2. 阈值版本化写入 `policy_versions`。  
3. 降级策略自动化（告警 + 回退 + 限流）。

验收口径：
1. 门禁策略可灰度、可回滚、可审计。  
2. 未达标数据无法进入发布面。

## 阶段 3：脚本退役与双跑观察（D46-D75）

1. 仅保留脚本在“回填/对账/应急”标签下可触发。  
2. 进入 30 天双跑观察窗，持续比对数据一致性。  
3. 达标后移除生产调度中的脚本主路径入口。

验收口径：
1. 连续 30 天主任务由 Worker 链路执行。  
2. 重复发布率、失败率、门禁通过率满足阈值。

## 阶段 4：按触发条件扩展（D76+）

1. 满足阈值再启用 Kafka。  
2. 满足检索/分析业务再引入 OpenSearch/增强 CH。  
3. 进入 K8s 化评审，准备高可用部署。

## 5. 团队对齐清单（会议必须回答）

1. 我们是否接受 `services/*` 作为唯一目标目录并一次性迁移 workspace？  
2. `publish_worker` 的发布拒绝条件是否以质量门禁为硬门槛？  
3. Kafka 的启用阈值是否采用 Plan0 默认值，还是需要业务修订？  
4. 脚本退役观察窗是否固定 30 天，还是按来源分层？  
5. Compose 到 K8s 的迁移触发点由谁审批、何时启动？
