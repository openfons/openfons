# 平台级重构架构与目录方案

> 版本：v0.1  
> 日期：2026-03-12  
> 适用范围：`papaweb` Plan0 开源基线  
> 基线约束来源：`docs/plan/plan0/平台级采集与分析架构方案（开源可商用版）.md`

---

## 1. 目标与范围

本方案采用 Greenfield 思路重构，不依赖现有脚本目录形态，直接按平台化架构组织代码与基础设施。

目标：
1. 建立可长期维护的采集与治理平台结构，而不是脚本集合。
2. 在开源版内保持与平台基线一致的任务模型和执行边界。
3. 让新功能通过“新增模块”接入，而不是修改主流程骨架。

---

## 2. 架构硬约束（不可破坏）

1. 编排真源唯一：`Airflow` 是唯一调度决策源。  
2. 任务与状态真源：`PostgreSQL`（`crawl_jobs/crawl_tasks/url_state/seed_state`）。  
3. 发布写口唯一：仅 `publish_worker` 可写入 Gold 服务表。  
4. Worker 不直写 `ClickHouse`，分析层仅接批处理或 CDC。  
5. 开源版可替换控制台/观测界面，不得引入第二编排真源。

---

## 3. 目标架构（重构后）

```text
策略配置/编排层（唯一编排真源）
  └─ Airflow (DAG + 调度策略)
        ↓
执行网关层
  └─ Crawlab (执行入口、运行态、日志)
        ↓
采集执行层
  ├─ worker-fetch-http
  ├─ worker-fetch-browser (Pinchtab/Playwright)
  ├─ worker-normalize
  ├─ worker-quality
  ├─ publish_worker
  └─ worker-seed
        ↓
Frontier/去重层
  └─ Valkey (默认) + Kafka (条件启用)
        ↓
数据分层
  ├─ Raw: SeaweedFS (证据原文/截图/日志)
  ├─ Silver: PostgreSQL (结构化事实/任务状态)
  └─ Gold: Query API + 服务表
        ↓
质量与分析
  ├─ dbt + Great Expectations
  ├─ Metrics/Alerting (Prometheus + Alertmanager)
  └─ BI (Superset)
```

## 5. 模块职责边界

1. `services/orchestrator-airflow`：仅编排，不实现业务解析逻辑。  
2. `services/gateway-crawlab`：统一执行入口与运行态采集。  
3. `services/worker-*`：每类 Worker 单一职责，输入输出走统一 `contracts`，发布路径统一命名为 `publish_worker`。  
4. `services/query-api`：只读聚合与服务化查询，不承担采集写入。  
5. `packages/contracts`：跨服务共享的唯一协议源，版本化管理。

---

## 6. 开发与扩展规则

1. 新接一个来源：新增 `packages/adapters/<source>` + `config/source-sites/<source>.yaml`。  
2. 新增一个门禁规则：只改 `packages/policy-engine` + `config/policies`。  
3. 新增一个 Worker：新增独立 `services/worker-*`，不得塞进现有 Worker。  
4. 所有服务日志与指标必须接入 `sdk-observability`。  
5. 禁止把运行态数据写回仓库业务目录（例如 `apps/**/data` 形态）。

---

## 7. P0 最小可运行交付

1. `orchestrator-airflow` + `gateway-crawlab` + `worker-fetch-http` + `worker-normalize` + `worker-quality` + `publish_worker`。  
2. `contracts`、`policy-engine`、`query-api` 最小可用版本。  
3. 数据底座：`PostgreSQL + Valkey + SeaweedFS`。  
4. 可观测：`Prometheus + Alertmanager`。  
5. 验收口径：30 天内生产任务由 Worker 链路执行，脚本不进入主路径。

---

## 9. 仓库结构迁移说明（workspace 对齐）

当前仓库 workspace 为 `apps/*` 与 `packages/*`，本方案使用 `services/*` 属于目标形态。为避免包管理与 CI 失效，迁移必须二选一并一次性落地：
1. 路径迁移：将 `services/*` 全量落在 `apps/*` 下（例如 `apps/publish_worker`），文档中的 `services/*` 仅表示职责域。  
2. workspace 迁移：保留 `services/*`，并同步修改根 `package.json` 的 workspaces、CI 脚本与命令入口。  
3. 禁止半迁移：不得在同一阶段内长期并存 `apps/*` 与 `services/*` 双主目录作为生产路径。

---

## 8. 非目标（P0 不做）

1. 不做双编排体系并行。  
2. 不做 Worker 直写 ClickHouse。  
3. 不做多租户计费与企业私有化工具链。  
4. 不在 P0 引入“脚本主链路 + 平台链路”双轨长期共存。

---

## 10. 本轮评审补充（2026-03-13）

### 10.1 架构形态补充

1. 采用“多服务拆分”作为最终工程形态：控制面、执行面、数据面、观测面分离。  
2. 采用“小闭环先跑通再扩展”作为交付节奏：先最小链路，再扩组件。  
3. 采用渐进迁移（Strangler Fig）：优先接管主路径，再退役历史脚本路径。

### 10.2 Docker 部署补充

1. Compose 用于开发/联调/单机部署基线。  
2. 部署原则为一服务一容器，不采用全组件单容器。  
3. 推荐分层编排文件：  
   `compose.base.yml`、`compose.collect.yml`、`compose.orchestrator.yml`、`compose.observability.yml`。  
4. 生产阶段目标为 Kubernetes 化，Compose 不作为最终高可用形态。

### 10.3 执行策略补充

1. 优先确保 `services/worker-*` 最小闭环可运行。  
2. 优先固化 `packages/contracts` 与状态机，不先扩展外围组件。  
3. Kafka/OpenSearch/增强分析组件采用触发式启用策略。  
4. 继续执行第 9 节迁移约束：目录迁移必须二选一且一次性落地，禁止半迁移。

### 10.4 与基线文档的一致性说明

本节补充不改变第 2 节硬约束，仅用于明确本轮评审后的实施方法与部署节奏。
