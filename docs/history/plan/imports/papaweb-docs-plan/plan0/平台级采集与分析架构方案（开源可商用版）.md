# 平台级采集与分析架构方案（开源可商用版）

> 版本：v1.5  
> 日期：2026-03-12  
> 适用：`papaweb` 长期采集、清洗、分析、表格化展示（TanStack Table）  
> 原则：优先开源、可商用、可二次开发，兼顾长期可维护性
> 文档角色：**本文件是 plan0 合并后的唯一基线（SoT）**
> 冲突裁决：其他 `docs/plan` 文档与本文件不一致时，以本文件为准。
> 合并来源：原 `平台版开源可商用技术选型方案.md`、`平台版收费产品规划（宽松许可商用优先）.md`、`种子驱动采集方案与长期策略.md`

## 1. 背景与目标

当前前端展示（TanStack Table）已具备方向，核心瓶颈在后端数据底座：需要从多源（网站、YouTube、Reddit、X 等）持续获取数据，自动清洗、质量评估、结构化入库，并形成可分析、可追溯、可复用的长期系统。

目标不是“继续堆脚本”，而是建设“平台化采集与治理系统”。

---

## 2. 结论（先看这个）

1. 现有 `Crawlab + Pinchtab` 是正确基础设施，但生产链路必须平台化，不依赖 `collect/clean` 脚本。  
2. 未来要升级为“平台优先 + 适配层”架构：平台负责通用能力，自研只做业务差异化。  
3. 不建议现在推翻重写；建议分阶段演进，在现有代码上接入队列、存储分层、质量治理和分析层。  
4. 外部平台接入遵循“官方 API 优先，网页抓取兜底”的稳定与合规原则。  
5. 2026 基线采用 `Airflow` 作为唯一编排真源；`Temporal` 为后续演进备选，需通过单独 RFC 迁移评审后启用。  
6. 本 SoT 适用于平台收费版基线；开源版仅可在不改变统一任务模型与执行网关边界的前提下替换编排控制台/观测界面，不得引入第二编排真源接管调度决策。

## 3. 总体架构（目标形态）

```text
策略配置/编排层（唯一编排真源）
  └─ Airflow（当前唯一真源）
        ↓
执行网关层
  └─ Crawlab（执行入口、日志与运行态）
        ↓
采集执行层
  ├─ HTTP Workers（快速、低成本）
  └─ Browser Workers（Pinchtab/Playwright，动态页面）
        ↓
Frontier与去重层
  └─ Valkey（默认） + 可选 Kafka（高吞吐事件总线；Redis 仅历史兼容）
        ↓
数据分层
  ├─ Raw 原始层（SeaweedFS：HTML/JSON/截图/日志）
  ├─ Silver 清洗层（PostgreSQL/ClickHouse 结构化事实）
  └─ Gold 服务层（面向 TanStack Table 的查询表/API）
        ↓
质量与分析层
  ├─ dbt + Great Expectations（质量规则）
  ├─ 指标评分（coverage/confidence/freshness）
  └─ Superset/Grafana（运营与质量看板）
```

说明：上图为平台收费版基线；开源版仅可替换编排控制台/观测界面，不改变任务模型与执行链路边界，也不改变 Airflow 作为唯一编排真源。

### 3.1 真源与写入边界（必须）

1. 编排真源：`Airflow`（2026 基线只允许单编排真源）。  
2. 任务与状态真源：`PostgreSQL`（`crawl_jobs/crawl_tasks/url_state/seed_state`）。  
3. 原始证据真源：`SeaweedFS`（HTML/JSON/截图/日志，按对象键不可变留存）。  
4. 分析副本：`ClickHouse` 仅接收由 PostgreSQL/Raw 产出的批处理或 CDC 数据，Worker 不允许直写 ClickHouse。  
5. Gold 发布依赖 `publish_worker`，不得由脚本或 ad-hoc SQL 直接写入生产服务表。  

### 3.2 Kafka 启用条件（默认可不启用）

MVP 阶段默认仅 `Valkey`；满足任一条件再引入 `Kafka`：
1. 峰值任务吞吐持续超过 `5,000 task/min`（连续 30 分钟）。  
2. 队列积压超过 `1,000,000` 条且 Valkey 扩容后仍无法在 `60` 分钟内回落。  
3. 事件消费者达到 3 类以上（采集、质量、检索索引、审计回放）且出现跨服务解耦瓶颈。  
4. 需要保留可回放事件日志以满足审计或故障重放。

## 4. 关键问题解释：原始存储 vs PostgreSQL（元数据与事务）

这是两个不同职责的层，必须同时存在。

### 4.1 原始存储（Raw Storage）是什么

原始存储用于保存“抓回来的原始证据”，例如：
- 页面 HTML
- API 原始 JSON
- 截图/快照
- 原始响应头、trace 日志

推荐：`SeaweedFS`（对象存储，新建默认）。
`MinIO` 仅作为历史存量兼容评估对象，不作为新建默认选型。

特点：
- 数据量大、写入多、长期留存
- 基本不更新，偏“追加式”
- 面向审计、回放、复盘

### 4.2 PostgreSQL 元数据与事务层是什么

PostgreSQL 用于保存“结构化、可查询、可事务保证”的业务数据，例如：
- 任务表：`crawl_jobs`, `crawl_tasks`
- 状态表：`url_state`, `seed_state`
- 实体表：`tool_entities`, `tool_facts`, `tool_evidence`
- 规则配置：`source_sites`, `policy_versions`

特点：
- 强一致、支持事务（ACID）
- 适合频繁查询、更新、关联分析
- 适合做业务主数据和流程状态

### 4.3 两者区别（一句话）

- 原始存储：存“原材料和证据包”。  
- PostgreSQL：存“加工结果、索引和业务状态”。  

### 4.4 为什么不能二选一

1. 只有 PostgreSQL：大体量原始文件存取成本高，回放/审计不方便。  
2. 只有对象存储：无法高效做事务、关联查询、状态流转。  
3. 同时使用：既能追溯证据，又能高效服务业务查询和自动化流程。

## 5. 推荐技术选型（开源可商用优先）

| 层级 | 推荐组件 | 用途 | 许可证/商用说明 |
| --- | --- | --- | --- |
| 执行网关 | Crawlab（已接入） | 爬虫任务执行管理、运行与日志 | BSD-3-Clause，适合商用 |
| 动态渲染 | Pinchtab（已接入） | 浏览器渲染、模拟交互 | MIT，适合商用 |
| 编排 | Airflow（唯一真源；Temporal 仅迁移备选） | 跨任务 DAG、数据流水线编排 | Apache-2.0 / MIT |
| 队列去重 | Valkey（优先） | 队列、去重、冷却、计数 | BSD-3-Clause |
| 事件总线 | Kafka（按需启用） | 高吞吐事件流、回放与跨服务解耦 | Apache-2.0 |
| 对象存储 | SeaweedFS（默认） | Raw 原始文件/快照存储 | Apache-2.0 |
| 事务数据库 | PostgreSQL | 主数据、元数据、状态机 | PostgreSQL License |
| 分析引擎 | ClickHouse | 大规模聚合分析 | Apache-2.0 |
| 数据质量 | dbt-core + Great Expectations | 转换建模与质量校验 | Apache-2.0（引入时二次核验） |
| 搜索服务 | OpenSearch | 文本检索与筛选 | Apache-2.0 |
| 监控与告警（必选） | Prometheus + Alertmanager | 指标采集、阈值告警、容量监控 | Apache-2.0 |
| 链路追踪（推荐） | OpenTelemetry + Jaeger | 分布式链路追踪与瓶颈定位 | Apache-2.0 |
| 可视化（按版本选择） | Superset（平台默认）/Grafana（可选） | 运营、质量与运维看板 | Apache-2.0 / AGPLv3（Grafana） |

> 备注：涉及 AGPL 的组件（如 Grafana）需结合你们的分发模式和法务策略评估。`MinIO` 仅作为历史存量兼容对象，其维护状态、许可证与发布节奏需在选型评审时复核。

## 6. 与现有仓库的映射（平台优先，脚本退役）

现有基础：
- 调度与渲染：`infra/docker/docker-compose.yml`（Crawlab + Pinchtab）
- 平台目标：`Airflow（Temporal 备选） + Worker 服务 + Valkey/Kafka + Raw/Silver/Gold`
- 迁移脚本：`apps/crawler/src/collect.mjs`、`apps/crawler/src/clean.mjs`、`apps/crawler/src/review-stage-b.mjs`、`apps/crawler/src/seed-toolify.mjs`（仅迁移期使用）

升级方式：
1. 直接建设可队列驱动的 Worker（`fetch/normalize/quality/publish`），脚本不作为生产执行单元。  
2. 新增统一任务状态表（PostgreSQL）和原始对象落盘规范（SeaweedFS；MinIO 仅存量兼容评估）。  
3. 保持 Crawlab 可用，由 Airflow 作为唯一编排真源管理跨阶段流水线（Temporal 仅备选）。  
4. 脚本仅保留“历史回填、数据对账、故障应急”场景，完成迁移后下线。

## 7. 90 天落地路线图（建议）

## 阶段一（0-30 天）：底座加固
1. 引入 PostgreSQL + Valkey，建立 `crawl_jobs/url_state/source_sites`。  
2. 原始采集结果统一落对象存储，规范路径与命名。  
3. 落地 `fetch/normalize/quality` Worker，接入队列与任务状态机（不依赖脚本）。  

## 阶段二（31-60 天）：多源与质量
1. 接入 YouTube/Reddit/X 的合法数据通道（优先官方 API）。  
2. 建立 Silver 层标准字段与证据链关联。  
3. 引入质量规则（覆盖率、置信度、新鲜度、异常波动），并在 `source_sites` 完成阈值配置。  
4. 上线 `publish_worker`，打通质量门槛到发布门槛的自动化链路。  

## 阶段三（61-90 天）：分析与服务
1. 建立 Gold 层服务表，面向 TanStack Table 高性能查询。  
2. 接入 ClickHouse 做趋势、对比、分群分析。  
3. 建立运营看板与告警体系，形成自治巡检闭环。  
4. 启动并完成“脚本退役 30 天观察窗”。
5. 完成一次生产级灾备恢复演练，并达到本文件 8.2 设定的 RPO/RTO 目标。

### 7.1 质量门禁阈值（默认）

默认阈值（可按 `source_sites` 分源覆盖）：
1. `coverage_score >= 0.85`  
2. `confidence_score >= 0.80`  
3. `freshness_hours <= 48`（高活跃源默认 `<=24`，低活跃源默认 `<=168`）  

发布判定：
1. `publish_worker` 必须同时满足 3 个阈值才可发布。  
2. 连续 `60` 分钟发布通过率 `< 70%` 或重复发布率 `> 1%` 时，自动触发降级与告警。  
3. 阈值调整需版本化记录在 `policy_versions` 并可追溯。

### 7.2 指标口径字典（统一口径）

以下口径统一用于 3.2、7.1、12 的自动化阈值判定，默认统计源为 `publish_worker` 事件日志 + PostgreSQL 任务表：
1. `task_throughput_1m`：每分钟新入队任务数（按 1 分钟滚动桶统计 `crawl_tasks` 新增）。`5,000 task/min` 判定需连续 30 个桶同时超阈值。  
2. `publish_pass_rate_60m`：最近 60 分钟 `publish_pass_count / publish_attempt_count`。  
3. `duplicate_publish_rate_60m`：最近 60 分钟 `duplicate_publish_count / publish_attempt_count`。  
4. `failure_rate_6h_403_429`：最近 6 小时 `(http_403_count + http_429_count) / http_attempt_count`。  
5. 若分母为 `0`，该指标记为 `N/A` 且不触发自动降级，需人工复核并记录审计原因。

## 8. 合规与风险控制（必须）

1. 仅采公开数据，遵守 robots 与平台条款。  
2. 自动策略是“节流、退避、降级、切换来源”，不是绕过验证。  
3. 平台 API（YouTube/Reddit/X）优先官方通道，网页抓取只作为补充与兜底。  
4. 保留证据链和策略版本，确保可复核与可审计。  
5. API 密钥与账号凭据统一进入 Secret Manager，最长期限 `90` 天轮换。  
6. 任何来源若收到删除/下线请求，需在 `7` 天内完成状态标记、下游撤回与审计留痕。法务删除请求优先级高于默认留存策略。  
7. 删除执行范围至少覆盖：Raw 对象、检索索引、Gold 服务表及缓存副本；审计留痕仅保留合规最小字段与不可逆摘要。  

### 8.1 平台条款合规矩阵（默认模板）

| 来源类型 | 采集通道 | 存储边界 | Raw 默认留存 | 删除请求 SLA | 删除范围（最小） | 必留审计 |
| --- | --- | --- | --- | --- | --- | --- |
| 官方 API（YouTube/Reddit/X） | 官方 API Key/OAuth | 仅存条款允许字段与证据索引 | 180 天（或按平台更短上限） | 7 天 | Raw + 索引 + Gold + 缓存 | 请求参数摘要、响应摘要、策略版本 |
| 公开网页 | HTTP/Browser Worker | 不登录、不越权、不绕过付费墙 | 90 天 | 7 天 | Raw + 索引 + Gold + 缓存 | 抓取时间、URL、robots 判定、提取器版本 |
| 手工白名单种子 | Seed Worker | 仅采白名单域与允许路径 | 90 天 | 7 天 | Raw + 索引 + Gold + 缓存 | 白名单版本、审批人、任务链路 |

### 8.2 灾备目标（RPO/RTO）

1. `PostgreSQL`：`RPO <= 5 分钟`，`RTO <= 30 分钟`；启用 WAL 归档与每日全量备份。  
2. `SeaweedFS`：`RPO <= 15 分钟`，`RTO <= 60 分钟`；对象副本至少跨可用区部署。  
3. `Airflow` 元数据库与 DAG 配置：`RPO <= 15 分钟`，`RTO <= 30 分钟`。  
4. 演练要求：每月执行恢复演练、每季度执行故障切换演练；结果需沉淀审计记录并关联 `policy_versions`。

## 9. 脚本退役验收标准（必须量化）

1. 连续 30 天生产环境 `crawl_jobs` 全部由 Worker 链路执行（0 次脚本直调）。  
2. `fetch/normalize/quality/publish` 四类 Worker 覆盖率达到 100%。  
3. 脚本仅在“回填/对账/应急”任务标签下可触发，并具备审计日志。  
4. 迁移完成后将脚本从生产调度配置移除，仅保留归档与手动工具入口。

## 10. 一页式最终建议

你们的最佳路径是：
1. 保留现有 `Crawlab + Pinchtab` 基础设施。  
2. 立即补齐 `Valkey + PostgreSQL + 对象存储` 三件套。  
3. 直接建设 `Airflow（Temporal 备选） + Worker + ClickHouse + 质量治理` 的平台主链路，脚本仅作迁移工具。  

这条路径能同时满足：
- 开源可商用
- 可持续扩展
- 与当前代码兼容
- 面向收费产品的长期可运营能力

## 11. 平台版商业化边界（合并自平台版收费规划）

平台版保留商业差异化能力（建议）：
1. 多租户隔离与配额计费。  
2. 高级策略中心（自动限流、自动降级、动态阈值调参）。  
3. SLA 级观测（链路追踪、审计报表、成本看板）。  
4. 企业交付能力（私有化部署模板、升级通道、运维工具链）。  

平台版验收补充：
1. 编排统一率：生产任务 100% 由 Airflow 编排。  
2. Worker 覆盖率：`fetch/normalize/quality/publish` 达到 100%。  
3. 质量达标率：发布数据满足默认阈值。  
4. 运营可见性：任一任务在 5 分钟内可定位失败点与证据链。

## 12. 种子兜底子策略（合并摘要）

统一优先级（高 -> 低）：
1. 官方 API 通道。  
2. 常规网页采集（HTTP/Browser Worker）。  
3. 种子驱动兜底采集。  

进入种子兜底层（满足任一）：
1. `api_cost_ratio > 1.0`（连续 24 小时）或官方 API 不可用。  
2. `failure_rate >= 35%`（403/429，连续 6 小时）。  
3. 需要高成本账号/代理规模化且 `ROI < 1.2`（按周评估）。  

退出种子兜底层（全部满足）：
1. 官方 API 恢复且成本回到阈值内。  
2. 常规网页采集成功率 `>= 95%` 且连续 7 天达标。  
3. `coverage >= 0.85` 且 `freshness_hours <= 48`。  

种子策略只影响候选 URL 生成与刷新，不改变统一任务模型与发布门槛。

## 13. 0-180 天路线（统一版）

1. `0-30 天`：底座加固（PostgreSQL + Valkey + Raw 落盘 + 基础 Worker）。  
2. `31-60 天`：多源接入与质量门禁（含 `publish_worker` 全量接管）。

4. `90-180 天`：商用扩展（多租户、配额计费、SLA、企业交付工具链）。

## 14. 主流最佳实践对齐补充（2026-03-13 评审新增）

### 14.1 升级方法论（冻结）

1. 采用“多服务拆分 + 小闭环先跑通再扩展”的路线：  
   多服务拆分用于长期可维护与可扩展，小闭环优先用于控制初期交付风险。  
2. 采用渐进式迁移（Strangler Fig）：  
   优先让 Worker 主链路接管，再逐步退役脚本主链路，不做一次性重写。  
3. 继续执行 3.1 真源边界：  
   `Airflow` 单编排真源、`publish_worker` 单发布写口、Worker 不直写 `ClickHouse`。

### 14.2 部署策略（冻结）

1. 本地/联调基线：Docker Compose 分层部署。  
2. 容器边界原则：一服务一容器，不采用“全服务单容器”。  
3. 分层启动顺序：  
   `base(数据底座) -> collect(采集执行) -> orchestrator(编排) -> observability(观测)`。  
4. 生产目标：在 Compose 稳定后进入 Kubernetes 化，不以 Compose 作为最终高可用形态。

### 14.3 P0 执行优先级（补充）

1. 先打通最小闭环：`fetch -> normalize -> quality -> publish`。  
2. 先固化状态机与契约：`crawl_jobs/crawl_tasks/url_state/seed_state/policy_versions`。  
3. 先把质量门禁前置到发布：覆盖率/置信度/新鲜度不达标不发布。  
4. Kafka、OpenSearch、增强分析能力按阈值触发启用，不提前堆叠复杂度。

### 14.4 本轮双评审结论
