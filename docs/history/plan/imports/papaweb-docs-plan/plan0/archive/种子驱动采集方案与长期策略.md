# 种子驱动采集方案与长期策略（平台子策略）

> 版本：v2.0  
> 日期：2026-03-12  
> 文档角色：平台采集体系中的“兜底子策略”，不替代平台级基线  
> SoT：`docs/plan/plan0/平台级采集与分析架构方案（开源可商用版）.md`

---

## 1. 定位

本方案用于“官方 API 不覆盖、站点风控较高或成本受限”场景下的长期兜底采集。  
它是平台主链路中的一个策略分支，不是独立系统，也不是脚本主链路。

---

## 2. 优先级与边界

统一优先级（高 -> 低）：
1. 官方 API 通道（首选）  
2. 常规网页采集（HTTP/Browser Worker）  
3. 种子驱动兜底采集（本方案）

执行边界：
1. 所有任务必须进入平台任务模型：`crawl_job -> fetch_task -> normalize_task -> quality_task -> publish_task`。  
2. 所有执行都由 Worker 完成，通过队列调度，不允许脚本直连生产链路。  
3. 种子策略只影响“如何生成与刷新候选 URL”，不改变质量门槛与发布门槛。

---

## 3. 触发条件

进入种子兜底层的条件（满足任一即可）：
1. 官方 API 不可用或成本超预算阈值。  
2. 目标站点持续出现高失败率（如 403/429）且超过策略阈值。  
3. 需要账号/高成本代理才能规模化，ROI 不达标。  
4. 合规评估要求降频与保守抓取。

退出种子兜底层的条件（全部满足）：
1. 官方 API 恢复可用且成本回到阈值内。  
2. 常规网页采集成功率稳定达标（连续 7 天）。  
3. 数据覆盖率与新鲜度达到目标线。

---

## 4. 架构映射（平台内）

```text
Airflow/Temporal（编排）
  -> Crawlab（执行网关）
    -> Seed Worker（种子发现/刷新）
    -> Fetch Worker（详情抓取）
    -> Normalize Worker（结构化清洗）
    -> Quality Worker（质量门槛）
    -> Publish Worker（发布）
```

关键数据对象：
1. `seed_state`：种子状态、冷却时间、失败计数。  
2. `url_state`：URL 首次发现、最近抓取、状态迁移。  
3. `tool_evidence`：证据链（原始页面、来源、提取器信息）。

---

## 5. 核心流程

1. `Seed Worker` 从 sitemap、分类页、标签页、人工白名单发现候选 URL。  
2. 统一 URL 规范化（去跟踪参数、去锚点、域名统一）。  
3. 通过 `Valkey` + `PostgreSQL` 做去重、冷却、优先级。  
4. `Fetch Worker` 逐步抓取详情页，必要时调用 Browser 能力。  
5. `Normalize Worker` 输出 Silver 结构化记录并关联 Raw 证据。  
6. `Quality Worker` 按覆盖率/置信度/新鲜度打分并判定是否发布。  
7. `Publish Worker` 仅发布通过质量门槛的数据。

---

## 6. 去重与增量策略

去重键建议：
1. `source_id + normalized_url + fetch_window`（平台统一幂等键）。  
2. 对实体层可补充 `canonical_domain + normalized_title` 作为软去重键。

复抓策略建议：
1. 默认复抓窗口：7 天（活跃源）/30 天（低活跃源）。  
2. 连续失败进入退避：指数退避 + 冷却窗口。  
3. 达到失败上限触发自动降级或暂停，等待策略复核。

---

## 7. 合规与风险控制

1. 仅采公开可访问数据，严格遵守 robots 与站点条款。  
2. 不绕过付费墙、不非法登录、不规避平台安全机制。  
3. 保留策略版本、任务日志、证据链，确保可审计可复核。  
4. 对高风险站点默认降频，优先选择官方通道或放弃采集。

---

## 8. 监控指标

必须监控：
1. 种子新增率与失效率。  
2. 详情抓取成功率（按 source 分层）。  
3. 风控失败率（403/429/401）。  
4. 去重命中率与重复发布率。  
5. 质量达标率（coverage/confidence/freshness）。

告警建议：
1. 连续失败次数超过阈值。  
2. 单源成功率连续 N 小时低于阈值。  
3. 质量达标率异常下滑。

---

## 9. 落地阶段建议（与平台路线对齐）

阶段一（0-30 天）：
1. 上线 `Seed Worker` 与 `seed_state/url_state`。  
2. 建立统一 URL 规范化与去重键。  
3. 接入质量评分最小闭环。

阶段二（31-60 天）：
1. 按 source 做差异化刷新周期与退避策略。  
2. 引入优先级队列（高价值种子优先）。  
3. 完成指标看板与告警联动。

阶段三（61-90 天）：
1. 形成“API -> 网页 -> 种子兜底”自动策略切换。  
2. 完成策略复盘机制（周度阈值调参）。  
3. 将历史应急脚本彻底从生产调度移除。

---

## 10. 最终结论

种子驱动方案应作为平台体系中的“长期兜底能力”，其价值是稳定与可持续，而不是替代平台主链路。  
在你们的最终目标下，主链路始终是：**平台编排 + Worker 执行 + 质量门槛发布**。
