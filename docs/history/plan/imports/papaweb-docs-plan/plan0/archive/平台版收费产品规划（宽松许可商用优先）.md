# 平台版收费产品规划（宽松许可商用优先）

> 版本：v1.1  
> 日期：2026-03-12  
> 适用：`papaweb` 收费平台版（商用交付）  
> 文档角色：产品化分轨规划（平台版）  
> 基线引用：`docs/plan/plan0/平台级采集与分析架构方案（开源可商用版）.md`
> 冲突裁决：与 SoT 不一致时，以 SoT 为准。

---

## 1. 定位与目标

平台版用于对外收费交付，核心目标是：

1. 商业可持续：许可证风险可控、可长期迭代。  
2. 统一编排：所有采集流程在一个主 UI 可观测、可审计、可回放。  
3. 平台化执行：不依赖 `collect.mjs/clean.mjs` 作为生产主链路。  

---

## 2. 总体策略

1. 采用“宽松许可证优先”策略：MIT/BSD/Apache/PostgreSQL License 为主。  
2. 保留当前 `Crawlab + Pinchtab` 资产，定位为执行网关与浏览器能力层。  
3. 由 `Airflow` 作为唯一编排真源和统一操作入口（主 UI）。  
4. 所有业务能力沉淀为 Worker 与规则配置，而不是脚本。  

---

## 3. 统一编排架构（平台版）

```text
Airflow（统一编排/UI）
  -> Crawlab（执行网关、任务运行态、日志）
    -> HTTP Worker / Browser Worker（Pinchtab）
      -> Valkey（队列、去重、冷却、优先级）
        -> SeaweedFS（Raw 证据层）
        -> PostgreSQL（任务/状态/元数据）
        -> ClickHouse（分析）
        -> OpenSearch（检索）
          -> dbt + Great Expectations（建模与质量）
            -> API 服务 -> 前端产品页
```

---

## 4. 组件清单（平台版）

| 层级 | 组件 | 许可证 | 选择原因 |
| --- | --- | --- | --- |
| 编排与主 UI | Airflow | Apache-2.0 | 成熟 DAG、API 完整、可自部署 |
| 执行管理 | Crawlab | BSD-3-Clause | 现有资产可复用，适合多语言爬虫托管 |
| 动态页面执行 | Pinchtab | MIT | HTTP API 形态，易被 Worker 调用 |
| 队列/去重 | Valkey | BSD-3-Clause | 许可友好，适合任务状态与幂等控制 |
| Raw 对象存储 | SeaweedFS | Apache-2.0 | 许可宽松，适合原始证据长期留存 |
| 事务数据库 | PostgreSQL | PostgreSQL License | 任务状态与主数据存储 |
| 分析引擎 | ClickHouse | Apache-2.0 | 高并发聚合分析 |
| 检索 | OpenSearch | Apache-2.0 | 全文检索与筛选 |
| 质量治理 | dbt-core + GE | Apache-2.0 | 建模和质量门禁 |

> 说明：`MinIO` 当前官方仓库页明确标注“主仓库不再维护”，平台版不作为新建默认选型。

---

## 5. 三层采集逻辑落地（平台版）

统一执行策略（高 -> 低）：

1. 官方 API 层：优先合规 API，控制配额与成本。  
2. 网页抓取层：HTTP/Browser Worker 常规抓取。  
3. 种子兜底层：仅在 API 不可用或 ROI 不达标时启用。  

三层都进入统一任务模型：

`crawl_job -> fetch_task -> normalize_task -> quality_task -> publish_task`

---

## 6. 平台版与开源版的边界

平台版保留商业差异化能力（建议）：

1. 多租户隔离与配额计费。  
2. 高级策略中心（自动限流、自动降级、动态阈值调参）。  
3. SLA 级观测（链路追踪、审计报表、成本看板）。  
4. 企业交付能力（私有化部署模板、升级通道、运维工具链）。  

---

## 7. 90天落地路线（MVP）

### 0-30 天

1. Airflow 接管编排，形成统一 DAG。  
2. 打通 Crawlab/Pinchtab 的任务触发与结果回写。  
3. 建立 Raw/Silver/Gold 最小闭环。  

### 31-60 天

1. 三层采集策略上线（API/网页/种子兜底）。  
2. 上线质量门禁与发布门禁。  
3. 形成可观测闭环（任务、质量、成本）。  

### 61-90 天

1. 完成脚本退役验收（连续 30 天生产 0 次脚本直调）。  
2. 上线收费能力内测版（租户、配额、基础审计）。  
3. 建立版本化发布流程（平台版与开源版并行演进）。  

### 90-180 天（扩展期）

1. 从“内测版收费能力”扩展到正式商用发布。  
2. 完成企业级能力补齐（高级策略中心、SLA 观测、交付工具链）。  
3. 完成重点客户场景的容量与稳定性压测。  

---

## 8. 验收指标（平台版）

1. 编排统一率：生产任务 100% 由 Airflow 编排。  
2. Worker 覆盖率：`fetch/normalize/quality/publish` 达到 100%。  
3. 质量达标率：发布数据满足 coverage/confidence/freshness 阈值。  
4. 运营可见性：任一任务在 5 分钟内可定位失败点与证据链。
