# 全平台数据采集与智能分析平台架构方案

> 面向舆情监控、商品评价分析、情报研究的综合平台
> 
> 版本：v1.0
> 日期：2026-03-23
> 状态：方案设计阶段
> 
> **项目背景**：本项目为 papaweb 的 Greenfield 重启版本，采用全新架构实现全平台数据采集与智能分析能力。

## 术语表

| 术语 | 英文 | 说明 |
|------|------|------|
| **主题** | Topic | 用户定义的研究主题，如"某品牌舆情监控" |
| **运行** | TopicRun | 主题的一次具体执行，包含采集时间、策略版本等信息 |
| **证据** | Evidence | 采集到的单条内容及其分析结果，平台最小真源单元 |
| **证据集** | EvidenceSet | 经过筛选的证据集合，用于生成报告 |
| **交付物** | Artifact | 最终生成的报告、图表、数据导出等成果 |
| **方面级情感** | Aspect-Based Sentiment | 针对特定方面（如质量、价格）的情感判断 |
| **Worker** | Worker | 执行具体任务的服务组件，如采集、分析等 |

## 1. 项目背景与目标

### 1.1 背景

当前互联网信息分散在多个平台（社交媒体、电商平台、新闻站点等），用户需要一种能力：
- **统一采集**：从多个平台自动采集相关信息
- **智能整理**：利用 AI 对海量信息进行摘要、分类、情感分析
- **按需生成**：根据用户需求生成报告、图表、数据导出等多种交付物

### 1.2 核心目标

构建一个"全平台数据采集 + AI 智能整理 + 按需内容生成"的研究平台：

```
用户输入主题/需求 → 平台自动采集多源信息 → AI 整理分析 → 生成定制化交付物
```

**参考模式**：类似 Google NotebookLM 的"Source-Process-Generate"工作流，但专注于中文互联网多平台数据采集和舆情/情报分析场景。

### 1.3 典型应用场景

| 场景 | 采集源 | 分析维度 | 交付物 |
|------|--------|----------|--------|
| 品牌舆情监控 | 微博、小红书、抖音、新闻 | 情感趋势、热点话题、传播路径 | 舆情日报、危机预警 |
| 商品评价分析 | 淘宝、京东、亚马逊 | 方面级情感（质量/价格/服务） | 评价分析报告、改进建议 |
| 竞品情报研究 | 多平台综合 | 产品特性、用户反馈、市场定位 | 竞品对比报告 |
| 热点事件追踪 | 社交媒体 + 新闻 | 时间线、关键节点、舆论走向 | 事件复盘报告 |

## 2. 架构总览

### 2.1 四大平面架构

```
┌─────────────────────────────────────────────────────────────────┐
│                        Access Plane                              │
│  用户接入层：IM / Web / API / 外部系统集成                        │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│                      Control Plane                               │
│  控制层：意图理解、任务编译、策略决策、工作流编排                  │
│  （AI 深度参与策略制定，但不直接执行采集）                        │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│                     Execution Plane                              │
│  执行层：全平台采集、标准化、智能分析、质量管控                    │
│  （确定性执行，可审计、可回放）                                   │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│                   Data & Delivery Plane                          │
│  数据与交付层：主题管理、证据沉淀、报告生成、多形态交付            │
└─────────────────────────────────────────────────────────────────┘
```

### 2.2 核心设计原则

| 原则 | 说明 | 价值 |
|------|------|------|
| **AI 策略 + Worker 执行** | AI 负责制定采集和分析策略，Worker 负责确定性执行 | 既有 AI 智能，又有系统稳定性 |
| **真源驱动** | Topic/Evidence/Artifact 三级真源模型 | 数据可追溯、可审计、可复现 |
| **多平台适配** | 统一适配层对接不同平台（社交媒体、电商、新闻） | 一次开发，全平台复用 |
| **按需生成** | 同一批 Evidence 可生成多种交付物 | 满足不同场景需求 |

## 3. 核心数据模型

### 3.1 三级真源模型

```
Topic（研究主题）
  └── TopicRun（一次采集运行）
        └── Evidence（证据项） ← 最小真源单元
              ├── 原始内容（文本/媒体/元数据）
              ├── AI 分析结果（情感/主题/实体）
              └── 采集上下文（时间、平台、策略版本）
                    └── EvidenceSet（证据集合）
                          └── Artifact（交付物）
```

### 3.2 Evidence 模型（核心）

Evidence 是平台的最小真源单元，包含：

```typescript
interface Evidence {
  // 基础标识
  evidenceId: string;
  topicId: string;
  topicRunId: string;
  
  // 来源信息
  platform: 'weibo' | 'xiaohongshu' | 'douyin' | 'taobao' | 'jd' | 'news';
  itemType: 'post' | 'video' | 'comment' | 'product_review';
  url: string;
  capturedAt: Date;
  
  // 原始内容
  rawContent: {
    title?: string;
    body: string;
    author: string;
    publishTime: Date;
    mediaUrls?: string[];  // 图片/视频
    interactions: {
      likes: number;
      comments: number;
      shares: number;
    };
    // 评论列表（如果是帖子/视频）
    comments?: Comment[];
  };
  
  // AI 分析结果
  analysis: {
    // 情感分析（核心）
    sentiment: {
      postLevel: SentimentResult;           // 帖子级情感
      commentLevel?: SentimentResult[];     // 评论级情感
      aspectLevel?: AspectSentiment[];      // 方面级情感（商品评价）
      stats: SentimentStats;                // 情感统计
    };
    
    // 主题提取
    topics: string[];
    
    // 实体识别
    entities: {
      name: string;
      type: 'person' | 'brand' | 'product' | 'location';
    }[];
    
    // 分析元数据
    modelVersion: string;
    analyzedAt: Date;
  };
  
  // 质量评分
  qualityScore: number;
  
  // 审计字段
  taskSpecId: string;      // 生成该 Evidence 的任务规格
  policyVersion: string;   // 策略版本
}
```

### 3.3 情感分析模型设计

针对商品评价、帖子评论等场景，设计多层级情感分析：

```typescript
interface SentimentResult {
  score: number;        // -1.0 ~ 1.0，负面到正面
  label: 'positive' | 'neutral' | 'negative';
  confidence: number;   // 0.0 ~ 1.0
  model: string;        // 使用的模型标识
}

interface AspectSentiment {
  aspect: string;       // 方面名称：质量、价格、服务、物流等
  sentiment: SentimentResult;
  evidence: string;     // 支撑该判断的原文片段
}

interface SentimentStats {
  totalComments: number;
  positiveCount: number;
  neutralCount: number;
  negativeCount: number;
  positiveRatio: number;
  aspectDistribution: {  // 各方面情感分布
    aspect: string;
    positiveRatio: number;
  }[];
}
```

**情感分析模型选型**：

| 模型 | 适用场景 | 特点 |
|------|----------|------|
| BERT-LoRA 微调 | 中文社交媒体 | 针对微博、小红书等平台微调 |
| Qwen3 小参数 | 通用中文内容 | 轻量、快速 |
| 多语言模型 | 跨境平台 | 支持英文评价分析 |
| 方面级情感模型 | 商品评价 | 识别质量/价格/服务等多维度情感 |

## 4. 详细架构设计

### 4.1 Access Plane（接入层）

**职责**：接收用户输入，提供多形态交互

```
用户输入方式：
├── Web 界面（主要）
│   ├── 新建研究任务
│   ├── 运行中心（查看进度）
│   ├── 证据浏览器
│   └── 报告中心
├── API 接口
│   └── 供外部系统调用
└── IM 机器人（扩展）
    └── 通过聊天方式下任务
```

### 4.2 Control Plane（控制层）

**核心职责**：AI 深度参与策略制定，生成确定性执行计划

```
用户输入: "分析某品牌手机最近一周的用户评价"
    ↓
Intent Router（AI）
    ├── 识别主题：某品牌手机
    ├── 识别平台：淘宝、京东、小红书
    ├── 识别时间：最近一周
    └── 识别分析维度：评价内容、情感、方面级分析
    ↓
Task Compiler（AI）
    ├── 生成采集策略
    │   ├── 平台：['taobao', 'jd', 'xiaohongshu']
    │   ├── 关键词：['品牌名', '手机型号']
    │   ├── 采集字段：['评价内容', '评分', '图片', '追评']
    │   └── 反爬策略：{频率控制, 代理轮换}
    ├── 生成分析策略
    │   ├── 情感分析：启用方面级（质量/价格/服务/物流）
    │   ├── 主题提取：识别用户关注焦点
    │   └── 实体识别：提取产品特性
    └── 生成交付策略
        ├── 报告类型：评价分析报告
        ├── 图表：情感分布饼图、方面级对比柱状图
        └── 数据导出：原始评价 CSV
    ↓
Policy Gate
    ├── 评估合规性：是否符合平台规则
    ├── 评估成本：预计 token 消耗、存储成本
    └── 评估风险：反爬风险、数据敏感度
    ↓
WorkflowSpec（确定性、可审计、可回放）
```

**关键设计**：AI 只负责"制定策略"，不负责"执行采集"。所有策略显性化为 WorkflowSpec，可追溯、可复现。

### 4.3 Execution Plane（执行层）

**核心职责**：确定性执行采集和分析任务

#### 4.3.1 服务架构

```
services/
├── worker-discovery/          # 发现待采集 URL
│   └── 输入：平台、关键词
│   └── 输出：URL 列表（带优先级）
│
├── worker-fetch-http/         # HTTP 采集
│   └── 常规网页、API 接口
│
├── worker-fetch-browser/      # 浏览器采集
│   └── 动态内容、反爬对抗
│
├── worker-fetch-media/        # 媒体采集
│   └── 视频、音频、图片下载
│
├── worker-normalize/          # 标准化处理
│   └── 统一字段格式、清洗数据
│
├── worker-analysis/           # 智能分析（含情感分析）
│   ├── 情感分析模块
│   │   ├── 整体情感判断
│   │   ├── 评论级情感分析
│   │   └── 方面级情感提取
│   ├── 主题提取
│   └── 实体识别
│
└── worker-quality/            # 质量评分
    └── 评估 Evidence 可信度
```

#### 4.3.2 多平台适配器设计

```
packages/adapters/
├── social/                    # 社交媒体适配器
│   ├── weibo.mjs             # 微博
│   ├── xiaohongshu.mjs       # 小红书
│   └── douyin.mjs            # 抖音
│
├── ecommerce/                 # 电商平台适配器
│   ├── taobao.mjs            # 淘宝
│   ├── jd.mjs                # 京东
│   └── amazon.mjs            # 亚马逊
│
└── news/                      # 新闻站点适配器
    └── ...
```

每个适配器实现统一接口：
- `discover(keywords)`：发现待采集 URL
- `fetch(url)`：采集单条内容
- `extract(html)`：提取结构化字段
- `getAntiCrawlPolicy()`：返回平台反爬策略

#### 4.3.3 情感分析模块（worker-analysis）

```
services/worker-analysis/
└── src/
    ├── analyzers/
    │   ├── sentiment/
    │   │   ├── index.mjs              # 情感分析入口
    │   │   ├── models/
    │   │   │   ├── bert-lora.mjs      # BERT 中文微调
    │   │   │   ├── qwen3.mjs          # Qwen3 小参数
    │   │   │   ├── multilingual.mjs   # 多语言模型
    │   │   │   └── aspect-based.mjs   # 方面级情感
    │   │   ├── aggregator.mjs         # 情感聚合统计
    │   │   └── config.mjs             # 模型路由配置
    │   ├── topic-extraction.mjs
    │   └── entity-recognition.mjs
    └── index.mjs
```

**情感分析流程**：

```
输入：NormalizedItem（标准化后的内容）
    ↓
1. 模型选择（根据平台 + 内容类型）
    ├── 微博/小红书 → BERT-LoRA / Qwen3
    ├── 淘宝/京东评价 → 方面级情感模型
    └── 英文内容 → 多语言模型
    ↓
2. 帖子级情感分析
    └── 标题 + 正文 → 整体情感
    ↓
3. 评论级情感分析（批量）
    ├── 逐条评论分析
    ├── 方面级情感（商品评价场景）
    │   └── 识别质量/价格/服务/物流等维度
    └── 提取关键情感片段
    ↓
4. 情感聚合统计
    ├── 计算正负中性比例
    ├── 各方面情感分布
    └── 识别情感趋势
    ↓
5. 输出：SentimentAnalysis 对象
    ↓
存入 Evidence.analysis.sentiment
```

## 5. 典型工作流程

### 5.1 商品评价分析场景

```
用户输入："分析 iPhone 16 在天猫和京东的用户评价"
    ↓
Control Plane
    ├── 识别主题：iPhone 16
    ├── 识别平台：天猫(tmall)、京东(jd)
    ├── 识别内容类型：商品评价
    └── 识别分析维度：方面级情感分析
    ↓
Task Compiler 生成 WorkflowSpec
    ├── 采集策略
    │   ├── 平台：tmall, jd
    │   ├── 关键词：['iPhone 16', 'iPhone16']
    │   ├── 采集字段：评价内容、星级、图片、追评
    │   └── 采集深度：最近 1000 条评价
    ├── 分析策略
    │   ├── 情感分析：启用方面级
    │   ├── 方面定义：['质量', '价格', '服务', '物流', '续航', '拍照']
    │   └── 主题提取：识别用户关注焦点
    └── 交付策略
        ├── 报告：评价分析报告
        ├── 图表：各方面情感对比图、情感趋势图
        └── 导出：原始评价 CSV
    ↓
Execution Plane
    ├── worker-discovery → 发现商品评价页面 URL
    ├── worker-fetch-browser → 采集评价数据（反爬对抗）
    ├── worker-normalize → 标准化评价字段
    ├── worker-analysis
    │   ├── 方面级情感分析
    │   │   ├── "电池很耐用" → 续航: positive
    │   │   ├── "价格有点贵" → 价格: negative
    │   │   └── "物流很快" → 物流: positive
    │   ├── 主题提取
    │   └── 实体识别
    └── worker-quality → 质量评分
    ↓
Data & Delivery Plane
    ├── Evidence 存储（含情感分析结果）
    ├── EvidenceSet 构建
    └── Artifact 生成
        ├── 评价分析报告
        ├── 各方面情感对比柱状图
        ├── 情感趋势折线图
        └── 原始评价数据导出
```

### 5.2 品牌舆情监控场景

```
用户输入："监控某品牌最近一周的社交媒体舆情"
    ↓
Control Plane 识别为"长期监控任务"
    ↓
Watch Scheduler 创建定时任务
    ├── 每 2 小时采集一次
    ├── 平台：微博、小红书、抖音
    └── 关键词：品牌名、产品名
    ↓
每次执行：
    ├── 采集新内容
    ├── 情感分析（整体情感）
    ├── 与历史数据对比
    └── 检测异常波动（负面情感激增）
    ↓
生成舆情日报：
    ├── 情感趋势变化
    ├── 热点话题 TOP10
    ├── 负面内容预警
    └── 传播路径分析
```

## 8. 风险评估与应对

| 风险 | 影响 | 应对措施 |
|------|------|----------|
| 反爬对抗 | 高 | 多策略轮换、代理池、浏览器指纹 |
| 平台规则变化 | 中 | 适配器模块化、快速迭代 |
| 数据合规 | 高 | 仅采集公开数据、遵守 robots.txt |
| AI 成本 | 中 | 本地小模型 + API 大模型混合 |
| 情感分析准确性 | 中 | 多模型融合、人工标注反馈 |

### 8.1 情感分析评估指标

| 指标 | 说明 | 目标值 |
|------|------|--------|
| **准确率** | 情感分类正确的比例 | ≥ 85% |
| **F1-Score** | 精确率和召回率的调和平均 | ≥ 0.82 |
| **方面级准确率** | 方面识别和情感判断同时正确 | ≥ 75% |
| **一致性** | 相同内容多次分析结果一致 | ≥ 95% |

**评估方法**：
1. 人工标注 1000 条测试数据（覆盖各平台、各类型）
2. 定期抽样人工复核（每月 5%）
3. 用户反馈收集和模型迭代

## 9. 与参考项目的区别

| 参考项目 | 核心特点 | 我们的区别 |
|----------|----------|-----------|
| **BettaFish** | AI Agent 直接驱动采集 | AI 制定策略，Worker 确定性执行，更可审计 |
| **NotebookLM** | 用户上传文档，AI 整理 | 主动采集多平台数据，专注舆情/情报场景 |
| **传统爬虫** | 单一平台、固定规则 | 多平台统一适配、AI 动态策略 |

## 10. 总结

本方案构建一个**全平台数据采集与智能分析平台**，核心创新点：

1. **AI 策略 + Worker 执行**：既利用 AI 智能制定策略，又保证系统确定性
2. **三级真源模型**：Topic/Evidence/Artifact 确保数据可追溯、可复现
3. **多层级情感分析**：帖子级 + 评论级 + 方面级，满足商品评价分析需求
4. **多平台统一适配**：一套架构覆盖社交媒体、电商、新闻等多类平台

**下一步行动**：进入 Phase 1，实现核心数据模型。
