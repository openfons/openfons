一、文档 1：《全平台数据采集与智能分析平台架构方案 - v2.md》
文档定位
这是 papaweb 项目 Greenfield（全新重启）版本的完整目标态架构方案，详细定义了项目最终要实现的产品能力、技术架构、核心模型与实施规划，是整个项目的「终局蓝图」。
核心内容解读
1. 项目核心定位与目标
解决的核心痛点：互联网信息分散在社交媒体、电商、新闻等多个平台，用户需要一站式的「统一采集 - 智能分析 - 按需生成报告」能力
核心目标：构建一套用户输入主题/需求 → AI分解任务 → 并行采集分析 → 生成定制化交付物的闭环平台
典型场景：品牌舆情监控、商品评价分析、竞品情报研究、热点事件追踪
核心创新：融合 Multi-Agent 的灵活性与 Worker 流水线的确定性，既保证 AI 的智能适配能力，又保证执行过程可控、可审计、可复现。
2. 核心架构哲学：Agent 决策 + Worker 执行
这是整个方案最核心的设计，彻底解决了纯 Agent 架构不可控、纯流水线架构僵化的问题：
表格
层级	核心职责	核心价值
上层 Multi-Agent	需求理解、任务分解、策略决策、生成确定性 Workflow	处理不确定性，提供灵活性与智能性
下层 Worker+Workflow	按既定计划确定性执行，每一步可审计、可回放、失败可重试	保证执行可控、稳定、低成本
铁律约束：Agent 只负责决策与生成执行计划，绝对不直接参与具体执行，从架构上避免了 AI 直接驱动浏览器带来的不可控风险。
3. 核心数据模型：三级真源模型
这是整个平台的「数据地基」，所有功能都围绕这个模型构建，确保所有数据可追溯、可审计、可复现：
plaintext
Topic（研究主题，如"iPhone16用户评价分析"）
└── TopicRun（主题的一次具体执行，含时间、策略版本）
└── Evidence（平台最小真源单元，含原始内容+AI分析结果+采集上下文）
└── EvidenceSet（经过筛选的证据集合）
└── Artifact（最终交付物，如报告、图表、数据导出）

核心设计：Evidence 是唯一真源，所有分析结果、交付物都必须有对应的原始证据支撑，杜绝 AI 凭空生成内容，满足舆情、情报等场景的严谨性要求。
配套设计：针对商品评价、舆情场景，设计了帖子级 - 评论级 - 方面级的三层情感分析模型，可精准识别用户对质量、价格、服务、物流等不同维度的评价倾向。
4. 系统四大平面架构
方案将整个系统拆分为职责清晰、边界明确的四层架构，每层各司其职，避免逻辑混乱：
Access Plane（接入层）：负责用户与外部系统接入，支持 IM、Web、API 等多种入口
Control Plane（控制层）：Multi-Agent 协作层，核心是理解用户意图、分解任务、生成可执行的 Workflow，是整个系统的「大脑」
Execution Plane（执行层）：Skill+Worker 执行层，核心是按 Workflow 完成具体的采集、分析、生成任务，内置了完整的反爬对抗、验证码处理、搜索服务集成能力
Data & Delivery Plane（数据与交付层）：负责证据沉淀、数据存储、报告生成与多形态交付，是整个系统的「资产沉淀中心」
5. 关键核心能力设计
Skill 化能力封装：将所有能力拆分为可复用的 Skill 模块（如微博采集 Skill、情感分析 Skill、报告生成 Skill），模块化设计，易扩展、易维护，借鉴了字节跳动 DeerFlow 的设计理念。
完整的反爬对抗体系：设计了四层防御策略，从网络层代理池、浏览器层指纹伪装、行为层真人操作模拟，到策略层动态频率调整与熔断，配套三层验证码降级方案（本地识别→第三方 API→人工兜底），保障采集的稳定性。
生产级工作流引擎：选型 Temporal 作为工作流引擎，原生支持天 / 周 / 月级别的长周期任务（如 7×24 小时舆情监控），具备 Exactly-Once 执行、故障自动恢复、状态持久化等生产级能力。
6. 实施路线与风险控制
分 6 个阶段落地：从基础框架搭建、核心数据模型实现、Skill 框架开发，到控制层、执行层、交付层实现，最终完成平台化建设。
明确了核心风险与应对：重点针对反爬对抗、平台规则变化、数据合规、AI 成本等高影响风险，制定了对应的应对措施。
明确了合规与性能要求：仅采集公开数据、遵守 robots 协议、用户隐私脱敏，同时制定了明确的性能指标（如 1000 条评价采集延迟 < 5 分钟）。
二、文档 2：《统一风控对抗平台架构方案.md》
文档定位
这是 papaweb 项目的总架构书，是连接「现状」与「文档 1 目标态」的顶层规划，核心回答了「我们当前在哪、要往哪去、核心边界是什么、按什么顺序落地」的问题，是整个项目的「路线总纲」。
核心内容解读
1. 核心定位与北极星目标
明确了项目的本质：不是做一个更强的爬虫脚本集合，而是构建一个可支撑长期研究、监控、报告与情报消费的平台。
北极星目标：打造自然语言输入 → 结构化任务 → 确定性执行 → 证据沉淀 → 多形态交付 → 长期消费的完整闭环。
核心升级方向：从「抓取结果平台」，彻底升级为「Topic/Evidence/Artifact 为核心的证据平台」。
2. 现状基线与核心问题
文档没有空谈愿景，而是先清晰对齐了当前仓库的真实现状：
当前真实主链路：query-api → worker-discovery → worker-fetch-http → worker-normalize → worker-quality → publish_worker
当前核心短板：
没有真正的 Topic/Evidence/Artifact 真源模型，无法支撑长期研究、历史追溯、多租户隔离
现有gold_tool_entities只是「最新结果视图」，不是可冻结、可追溯的交付真源
仓库目录混乱，大量上游镜像、参考代码、占位目录和正式运行时代码混在一起，团队容易误把资产当运行时
源站配置双轨并行，没有统一的 Source Registry 真源，不同模块解释口径不一致
3. 核心架构铁律（必须严格遵守）
这是整个架构书的核心，为项目划定了不可突破的边界，避免方向跑偏：
系统必须分为四大平面：严格按照 Access/Control/Execution/Data&Delivery 四层划分，所有模块必须归属到明确的平面，禁止把控制逻辑、抓取逻辑、数据逻辑混写。
大模型只负责编译，不负责执行：大模型仅负责理解需求、生成结构化的 Task/Workflow/ReportSpec，绝对不能直接驱动浏览器、下发页面操作，这条规则写死，不可突破。
未来系统必须以 Topic/Evidence/Artifact 为核心：这是所有功能的基础，没有这个核心模型，PDF 导出、长期监控、门户、权限控制都是空中楼阁。
Source Registry 必须只有一个真源：收编现有source-sites.v1.json，禁止双轨配置并行，避免同一个站点两套配置、不同 Worker 解释口径不同。
明确gold_tool_entities的定位：仅作为迁移期的兼容视图，不是最终证据真源，禁止再承接交付物存储、长期监控等语义。
4. 目标态总架构与核心对象模型
目标态架构与文档 1 的四大平面完全对齐，明确了每个平面的核心组件，同时强调：风控对抗是 Execution Plane 的核心子域，长期监控建立在 Topic/TopicRun 之上，报告门户建立在 Evidence/Artifact 之上。
定义了整个系统的核心对象：TaskSpec（任务是什么）、WorkflowSpec（任务怎么跑）、ReportSpec（任务怎么交付）、Topic/TopicRun（研究主题与执行批次）、Evidence/EvidenceSet/Artifact（证据与交付物），明确了每个对象的核心字段与职责。
5. 正确的落地顺序（核心价值）
文档彻底纠正了「先做门户、先做 PDF 导出」的错误方向，明确了从内到外的落地顺序：
Phase 0：冻结现状与拆分资产区，把地基修平
Phase 1：重建核心数据模型（Topic/Evidence/Artifact）
Phase 2：重建 Source Registry，统一源站真源
Phase 3：补齐 Control Plane（意图理解、任务编译）
Phase 4：补齐多引擎执行与长期监控能力
Phase 5：补齐交付层（模板、报告、多格式渲染）
Phase 6：补齐门户与运营层
三、文档 3：《统一风控对抗平台 - Phase0 实现级设计.md》

表格
分类	包含目录	定位说明
正式运行时	services / 下的 query-api、worker-discovery 等主链路服务，packages / 共享库，infra/、scripts/	当前真实运行的主链路代码，是平台的核心

3. 项目核心架构回顾
本项目采用四大平面分层架构，核心设计原则为「Agent 决策 + Worker 确定性执行」，核心数据模型为「Topic/Evidence/Artifact 三级真源模型」，所有能力采用 Skill 化模块化设计，具体架构如下：
接入层（Access Plane）：负责用户与外部系统接入，支持 IM、Web、API 等多种入口；
控制层（Control Plane）：Multi-Agent 协作层，负责意图理解、任务分解、策略决策、生成标准化 WorkflowSpec 执行计划；
执行层（Execution Plane）：Skill+Worker 执行层，负责按 WorkflowSpec 完成采集、分析、生成等具体任务，内置反爬对抗、验证码处理等核心能力；
数据与交付层（Data & Delivery Plane）：负责证据沉淀、数据存储、报告生成与多形态交付。

1. Skill 开发规范

必须添加完整的异常捕获，内部异常必须封装为统一的格式返回，禁止异常向上抛出导致整个 Workflow 崩溃；

WorkflowSpec 是任务执行的唯一标准，必须遵循以下规范：
必须使用固定的 JSON 格式，字段定义必须统一，禁止添加自定义的非标准字段；
