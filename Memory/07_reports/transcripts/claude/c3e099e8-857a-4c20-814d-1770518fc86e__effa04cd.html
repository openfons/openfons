### CRITICAL

C1. db.prepare(UPSERT_SQL).run(...params) — spread of 18 args hits V8 argument limit on Node 22/24

better-sqlite3's Statement.run() accepts either a rest spread or a single array. Spreading 18 values is fine today, but the real problem is that run(...params) where params is unknown[] is not type-safe — TypeScript infers the spread as unknown[] and the call compiles only because run is typed as run(...bindParameters: BindingDictionary): RunResult. The actual runtime type of each element is string | number, but TypeScript cannot verify that because params is built via a .map() returning string | number | ''. If ManifestEntry ever gains a field whose value survives the typeof value === 'number' branch as something else (e.g. a boolean from a future schema change), it silently coerces to a string — no compile error, no runtime error, wrong data. The fix is db.prepare(UPSERT_SQL).run(params) (single array argument), which is also what reviewer 1 flagged for a different reason. Noting here because the type-safety angle was missed.

C2. FTS4 content table docid = t.rowid join is broken after any DELETE or VACUUM

SEARCH_SQL joins on transcripts_fts.docid = t.rowid. With a content table (content='transcripts'), FTS4 stores the rowid of the source row at index time. If any row in transcripts is deleted and re-inserted (which the upsert does via ON CONFLICT DO UPDATE — SQLite's UPDATE path preserves rowid, but a future DELETE + INSERT would not), or if VACUUM is run (which renumbers rowids in non-WAL mode), the docid→rowid mapping silently diverges. The REBUILD_FTS_SQL re-syncs it, but only if called. Since searchTranscripts only rebuilds when segdir count is zero (already flagged as fragile by reviewer 1), a post-vacuum search returns rows joined to the wrong transcript. This is a data-correctness bug, not just a reliability issue.
