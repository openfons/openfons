## Release Workflow: .github/workflows/release.yml

### CRITICAL

1. Race condition: version commit can push on top of a stale ref, then the next push triggers publish of the wrong state.

The workflow triggers on every push to main. When changesets are present, it versions, commits, and pushes back to main. That push to main triggers a second run of the same workflow. The concurrency key release-${{ github.ref }} is release-refs/heads/main for both runs, so the second run will cancel the first if it's still running, or run after it. This is the intended two-step dance. However:

- If someone pushes a normal commit to main while the "version" run is in progress (between git add -A and git push), the git push will fail because the remote has moved ahead. There is no git pull --rebase or retry logic. The version commit is lost silently (the step fails, but the job doesn't necessarily surface this clearly since subsequent steps are gated on has == 'true').
- Worse: if two PRs merge in quick succession, both trigger version runs. The concurrency group will cancel one, but the git push in the survivor can still conflict with the other's push. There's no locking beyond the concurrency group.

Recommendation: Add cancel-in-progress: true to the concurrency block (it's missing, so the default is false). Also consider adding git pull --rebase before git push, or use a GitHub App token with branch protection bypass to make the push atomic.

2. Changeset detection is fragile and can produce false positives.

   if ls .changeset/*.md 2>/dev/null | grep -qv README.md; then

ls output is piped to grep -qv README.md. This filters out lines containing README.md, but:
- If a changeset file happens to have README.md in its (randomly generated) name, it will be incorrectly filtered out.
- More importantly, ls outputs bare filenames. If there's only README.md in .changeset/, ls outputs README.md, grep -qv excludes it, exit code 1, has=false -- correct. But ls *.md on some shells/locales can behave unexpectedly with glob expansion.
- A safer approach: find .changeset -maxdepth 1 -name '*.md' ! -name 'README.md' | grep -q .

This is borderline CRITICAL because a false negative means changesets never get versioned, and a false positive means the version step runs with nothing to do (which changeset version handles gracefully, but then you get an empty "chore: release" commit pushed to main).

### HIGH

3. git add -A after versioning stages everything in the repo, including any unexpected artifacts.

After pnpm install --no-frozen-lockfile, the lockfile changes. git add -A will stage that plus any other file that pnpm install or changeset version might create (e.g., temp files, .turbo cache if not gitignored). If .gitignore is incomplete, unintended files get committed to main.

Recommendation: Be explicit: git add .changeset/ pnpm-lock.yaml packages/*/package.json package.json or at minimum CHANGELOG.md and the files changesets actually touches.

4. No --provenance flag on publish despite requesting id-token: write.

The permissions block grants id-token: write with the comment # npm provenance, but the release script in package.json is pnpm publish -r --access public --no-git-checks -- no --provenance flag. The permission is granted but unused. Either add --provenance to the publish command or remove the id-token: write permission (principle of least privilege).

### MEDIUM

5. pnpm install --no-frozen-lockfile in CI is a supply-chain risk.

This runs after changeset version mutates package.json files, so the lockfile is stale. Understood. But --no-frozen-lockfile allows resolution of new versions of transitive dependencies that weren't in the original lockfile. A compromised or yanked transitive dep could be pulled in at this point.

Recommendation: Consider running pnpm install --frozen-lockfile first (already done), then after versioning, run pnpm install (without --frozen-lockfile is unavoidable here, but document why). Alternatively, only update workspace protocol versions: pnpm install --filter ... to limit scope.

6. The "chore: release" commit message is generic.

It doesn't include version numbers or package names. When looking at git history, every release looks identical. Consider using the output of changeset version or at least including the version: chore: release v$(node -p "require('./packages/cli/package.json').version").

7. Missing cancel-in-progress on concurrency.

As mentioned in issue 1, the concurrency block:

   concurrency: release-${{ github.ref }}

defaults to cancel-in-progress: false, meaning if a version-commit push triggers a new run while the old one is still going, both run simultaneously. This wastes CI minutes and can cause the race described above. Add cancel-in-progress: true.

### LOW

8. No build step before publish check. The release script does pnpm build && pnpm publish, so build happens at publish time. But if the version step changed workspace dependency versions, there's no verification that the build still passes before the version commit is pushed to main. A broken build gets committed, and the next run (publish) will fail. Consider adding pnpm build as a validation step after versioning.

9. Removed pull-requests: write. This is fine since the workflow no longer creates PRs via changesets/action. Just noting it's intentional and correct.

## CLI File: packages/cli/src/main.ts

### HIGH

10. Alias in for daemon install shadows a JavaScript/shell keyword and may confuse users.

in is a reserved word in JavaScript and a keyword in many shells (for x in ...). While commander handles it fine as a subcommand alias, it's a poor UX choice. If a user types openmnemo daemon in in a script context or tries tab-completion, it could cause confusion. Consider inst instead.

11. Alias l for locale conflicts with common shell usage.

Single-letter l is commonly aliased to ls in many shell configurations. While this doesn't cause a technical conflict (it's a subcommand argument, not a standalone command), it's unintuitive. Users might expect openmnemo l to list something. Consider loc.

### MEDIUM

12. No alias collision validation.

The aliases are hardcoded and currently don't collide, but there's no programmatic check. If a future contributor adds a command starting with d, s, r, etc., they might not realize the alias is taken. Commander will throw at runtime, but a test asserting alias uniqueness would catch this earlier.

13. import command aliased to imp -- consider im.

imp is three characters, which is longer than most other aliases (single or two chars). For consistency, im would match the pattern better. Minor, but the inconsistency stands out: i, up, imp, d, l, s, r, dm, rp. The three-letter imp breaks the pattern.

### LOW

14. Program description mentions aliases but doesn't list them all.

The description says (e.g. s=search, d=discover) but doesn't mention the others. Users running openmnemo --help see only two examples. Either list all aliases in the description or remove the examples and let --help output show them naturally (commander prints aliases next to command names).

15. No test coverage for aliases.

There don't appear to be tests verifying that aliases resolve to the correct commands. If an alias is accidentally changed or removed, nothing catches it. A simple parameterized test that invokes each alias with --help and checks the output would suffice.

## Summary

The release workflow has the most consequential issues: the race condition around git push without rebase/retry (CRITICAL), the fragile changeset detection (CRITICAL), and the overly broad git add -A (HIGH). The unused provenance permission is a quick fix.

The CLI aliases are clean and functional. The main concerns are the in and l alias choices (HIGH for potential confusion) and the lack of alias collision tests (MEDIUM). Everything else is polish.
