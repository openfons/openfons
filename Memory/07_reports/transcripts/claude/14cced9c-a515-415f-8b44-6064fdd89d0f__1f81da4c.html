function writeVbsLauncher(scriptPath: string): string {
  const vbsPath = vbsLauncherPath()
  mkdirSync(dirname(vbsPath), { recursive: true })
  // Inside VBScript strings, double-quote is escaped as ""
  const escaped = scriptPath.replace(/"/g, '""')
  const content = [
    'Set WshShell = CreateObject("WScript.Shell")',
    `WshShell.Run "node ""${escaped}"" daemon run-once", 0, False`,
  ].join('\r\n') + '\r\n'
  writeFileSync(vbsPath, content, 'utf-8')
  return vbsPath
}

function installSchtasks(scriptPath: string, seconds: number): number {
  if (isSchtasksRegistered()) {
    process.stderr.write("Heartbeat is already registered in Task Scheduler. Use 'uninstall' first.\n")
    return 1
  }

  const minutes = Math.max(1, Math.floor(seconds / 60))
  const vbsPath = writeVbsLauncher(scriptPath)
  const trCommand = `wscript.exe /B "${vbsPath}"`

  try {
    execCommand('schtasks', [
      '/create', '/tn', TASK_NAME, '/sc', 'minute', '/mo', String(minutes),
      '/tr', trCommand, '/f',
    ])
  } catch {
    process.stderr.write('Failed to create scheduled task.\n')
    return 1
  }

  process.stdout.write(`Heartbeat registered in Task Scheduler (every ${minutes}m).\n`)
  return 0
}

## Code Review: Windows Task Scheduler / VBScript 路径 (cmd-daemon.ts)

### 总体评估

execCommand 底层使用 execFileSync（非 exec/execSync），这意味着 schtasks 的参数列表在操作系统层面是安全的——不会触发 shell 解析。但 VBScript 内容本身是一个独立的字符串拼接层，安全性需要单独审查。

---

## 问题列表

### CRITICAL

无

---

### HIGH

#### H1: vbsPath 本身含双引号时 trCommand 会崩溃

const trCommand = `wscript.exe /B "${vbsPath}"`

vbsPath 来自 resolve(homedir(), '.memorytree', 'heartbeat-launcher.vbs')。

homedir() 在 Windows 上通常不含引号，但并未做任何转义。若 homedir 路径中含双引号（极端情况，但理论上可能），trCommand 字符串就会被 schtasks 解析错误。

更重要的是：此字符串作为 /tr 参数的值传入 execCommand，而 execFileSync 会将整个字符串作为单一参数传给 schtasks。schtasks 自己再解析 /tr 参数值时按 CMD 语法处理，内嵌引号如果不规范会导致任务注册成功但运行时路径错误。

实际风险程度：在路径含空格的典型 Windows 机器（如 C:\Users\John Doe\）上，vbsPath 为 C:\Users\John Doe\.memorytree\heartbeat-launcher.vbs，外层已有双引号包裹，空格情况是安全的。但 vbsPath 自身含 " 时没有任何保护——这是一个结构性漏洞，即便当前场景不易触发。

---

#### H2: scriptPath 在 VBScript 内仅对 " 做转义，对 ^ % ! 等 VBScript 特殊字符未处理

const escaped = scriptPath.replace(/"/g, '""')
const content = [
  'Set WshShell = CreateObject("WScript.Shell")',
  `WshShell.Run "node ""${escaped}"" daemon run-once", 0, False`,
]

WshShell.Run 接受的命令字符串由 VBScript 解释器传给 CreateProcess，不经过 CMD shell，所以 ^、&、| 等 CMD 特殊字符不是问题。

但 % 是 VBScript 字符串中的普通字符，同样无害。

VBScript 字符串内真正的风险：若 scriptPath 包含 " 本身（即 " 出现在路径中，Windows 路径规范中不允许，但如果上层传入了脏数据），双引号转义为 "" 是 VBScript 字符串字面量的正确做法，这一点处理是正确的。

综合来看，对 scriptPath 的转义逻辑在 Windows 正常路径范围内是正确的，但：
- 没有对 scriptPath 做前置校验（如拒绝含 " 的路径）
- 转义后的内容直接嵌入 .vbs 文件，不是参数数组形式，是字符串模板注入结构

严格来说这是 MEDIUM，但由于 scriptPath 来自 heartbeatScriptPath()（由 process.argv[1] 或 import.meta.url 派生，非用户直接输入），实际攻击面很窄，所以整体可接受——但设计上存在注入结构。

---

### MEDIUM

#### M1: trCommand 中 vbsPath 含空格时行为依赖 schtasks 的引号解析

const trCommand = `wscript.exe /B "${vbsPath}"`

此字符串作为 /tr 的单一参数值传给 execFileSync，schtasks 会将其存入注册表作为任务命令。

Windows schtasks /tr 参数要求：若路径含空格，外层引号必须存在。当前写法对 vbsPath 含空格是正确的（已有引号包裹）。

但 vbsPath 路径分隔符来自 Node.js resolve()，在 Windows 上会返回反斜杠路径，例如：
C:\Users\John Doe\.memorytree\heartbeat-launcher.vbs

这对 schtasks 和 wscript.exe 均正常工作——此处无问题，但需确认 resolve() 在 Windows 上确实返回反斜杠而非正斜杠（Node.js path.resolve 在 win32 上行为正确，已确认无误）。

---

#### M2: writeVbsLauncher 写入失败时不抛出异常，但调用方 installSchtasks 会继续执行

function writeVbsLauncher(scriptPath: string): string {
  const vbsPath = vbsLauncherPath()
  mkdirSync(dirname(vbsPath), { recursive: true })
  writeFileSync(vbsPath, content, 'utf-8')
  return vbsPath
}

writeFileSync 失败会抛出，但 installSchtasks 并没有 try-catch 包裹 writeVbsLauncher 调用——它只 catch 了 execCommand('schtasks', ...) 的错误。

// installSchtasks 中：
const vbsPath = writeVbsLauncher(scriptPath)  // 若此处抛出，异常会向上冒泡
const trCommand = `wscript.exe /B "${vbsPath}"`
try {
  execCommand('schtasks', [...])
} catch {
  process.stderr.write('Failed to create scheduled task.\n')
  return 1
}

若 writeVbsLauncher 抛出（例如磁盘满、权限不足），错误信息是原生 Node.js ENOSPC/EACCES，不是友好的用户提示。

---

#### M3: uninstallSchtasks 总是输出成功消息，即使任务从未注册过

function uninstallSchtasks(): number {
  try {
    execCommand('schtasks', ['/delete', '/tn', TASK_NAME, '/f'], { allowFailure: true })
  } catch {
    // best effort
  }
  // ...
  process.stdout.write('Heartbeat removed from Task Scheduler.\n')  // 无条件输出
  return 0
}

与 installSchtasks 的幂等性保护（检查 isSchtasksRegistered()）不对称。uninstall 在任务不存在时同样输出"已移除"，可能误导用户。

对比：uninstallLaunchd 用 existsSync(plistPath) 做了前置检查，行为更诚实。Linux 的 uninstallCron 则是过滤式的，天然幂等。

---

#### M4: VBScript 使用 wscript.exe /B 但文件编码写为 UTF-8

writeFileSync(vbsPath, content, 'utf-8')

Windows 的 wscript.exe 默认按系统 ANSI 代码页（如 GBK/CP936）读取 .vbs 文件，不是 UTF-8。当前 VBScript 内容是纯 ASCII，所以无实际问题。

但若 scriptPath 包含中文字符（如 C:\Users\张三\...），UTF-8 编码的 VBScript 文件在 GBK 系统上会乱码，导致路径解析失败。

中文 Windows 用户名是极其常见的场景（本 repo 的目标用户包括中文用户），这是一个隐性的高风险路径。

---

### LOW

#### L1: TASK_NAME = 'MemoryTree Heartbeat' 含空格，作为 /tn 参数传入

execFileSync 将其作为独立参数传入，schtasks 会正确处理带空格的任务名，无需引号。这是正确的。但任务名含空格在 schtasks GUI 中显示分组可能不直观。可考虑改为 MemoryTree\Heartbeat 使用子文件夹形式（纯风格建议）。

---

#### L2: uninstallSchtasks 的注释 // best effort 吞掉了所有异常

try {
  execCommand('schtasks', ['/delete', '/tn', TASK_NAME, '/f'], { allowFailure: true })
} catch {
  // best effort
}

已传入 allowFailure: true，根据 execCommand 实现，失败时返回空字符串而不抛出。外层 try-catch 实际上是多余的双重保护，造成轻微混淆。

## 路径安全性矩阵

| 路径场景 | scriptPath (VBS内容) | vbsPath (/tr参数) | 结论 |
|---|---|---|---|
| 普通路径，无空格 | 安全 | 安全 | OK |
| 含空格（如 C:\Users\John Doe） | 安全（"" 包裹） | 安全（外层 " 包裹） | OK |
| 含括号（如 Program Files (x86)） | 安全 | 安全 | OK |
| 含中文（如 C:\用户\张三） | VBS 文件 UTF-8 编码问题（M4） | schtasks 参数层安全 | 存在风险 |
| 含双引号（非法 Windows 路径，但可能被上层传入） | 已转义为 "" | vbsPath 路径不含引号，安全 | scriptPath 已处理，vbsPath 无需处理 |
| 含 %（如环境变量展开场景） | VBScript 不展开 %，安全 | schtasks 会展开 %VAR% | 理论风险，实际路径不含 % |

## 与 uninstall 的对称性分析

| 操作 | install | uninstall |
|---|---|---|
| 幂等保护 | 有（isSchtasksRegistered() 检查） | 无（无条件执行+输出成功） |
| VBS 文件处理 | 创建 | 删除（有 existsSync 检查）|
| 错误反馈 | 友好提示 | 静默（best effort） |
| 与 launchd 对称性 | 一致 | 不一致：launchd 用 existsSync 判断，schtasks 直接执行 |

installLaunchd 用文件存在性判断幂等，installSchtasks 用 isSchtasksRegistered() 查询系统——方法不同但都有保护。uninstallSchtasks 缺少等价的"是否注册"检查是最主要的对称性缺失。
