- TLS 指纹伪装（用 [utls](https://github.com/refraction-networking/utls) 或在 Pinchtab 的 chromedp 层面走真实浏览器 TLS）
- HTTP/2 指纹一致性（SETTINGS frame 顺序、WINDOW_UPDATE 大小）
- 代理池统一管理，从 MediaCrawler 移植 proxy_ip_pool.py 逻辑到 Go 侧，作为独立 sidecar

### L2 浏览器指纹 — Pinchtab 胜出，但缺一致性

Pinchtab 的 canvas 噪声、WebGL、字体、硬件伪装已经很好。但现在的问题是：每次轮换指纹后，各维度之间不一致。比如 UA 说 Mac + Safari，但 navigator.platform 返回 Win32，WebGL vendor 返回 Intel — 这种矛盾本身就是检测信号。

需要新增：
- 身份档案（Identity Profile）：一个 profile 包含一致的 OS/Browser/Screen/GPU/Language/Timezone 组合
- 预定义 50+ 真实设备档案，从真机采集
- 指纹轮换时整体切换，不是单维度随机

### L3 行为指纹 — Pinchtab 有基础，但不够深

Pinchtab 的 Bézier 鼠标、打字模拟已经不错。但平台现在检测的是更高层的行为模式：

- 你打开页面后直接定位到目标元素 → 机器人
- 你从不滚动页面 → 机器人
- 你每次停留时间完全一样 → 机器人

需要新增：
- 页面浏览模拟：先滚动浏览，随机停留，再定位目标
- 停留时间分布：正态分布，不是固定值
- 滚动行为：惯性滚动、回滚、速度变化
- 鼠标闲置轨迹：不操作时也有微小移动

### L4 平台签名 — MediaCrawler 只覆盖了 XHS

这是最关键的一层。7 个平台各有不同的签名机制：

| 平台 | 签名机制 | MediaCrawler 现状 | 对抗策略 |
|------|---------|-------------------|---------|
| 小红书 | x-s, x-t, x-s-common | 有 (playwright_sign + xhs_sign) | 保留，通过 Pinchtab 浏览器执行 |
| 抖音 | a_bogus, msToken, ttwid, _signature | 无 | 必须浏览器内执行 JS，不可逆向 |
| B站 | wbi 签名 (img_key + sub_key) | 无 | 可纯算法实现，定期更新 key |
| 快手 | did + captcha | 无 | 设备 ID 绑定 + 验证码服务 |
| 微博 | cookie 为主 | 基础 cookie 管理 | 增强 cookie 生命周期管理 |
| 贴吧 | tbs token | 基础 | 简单，保持 session 即可 |
| 知乎 | x-zse-96 (基于 x-zst-81) | 无 | 浏览器内执行，类似抖音 |

核心思路：放弃纯算法逆向签名，统一走 Pinchtab 浏览器执行。原因：
1. 抖音的 a_bogus 每隔几周更新混淆，纯逆向维护成本极高
2. Pinchtab 本身就是真实浏览器环境，直接在页面上下文里调用平台自己的签名函数
3. XHS 的 window.mnsv2 方案（MediaCrawler 已有）正好可以推广到其他平台

### L5 验证码应答 — MediaCrawler 只有滑块

MediaCrawler 的 OpenCV 滑块识别 + easing 轨迹生成是好的基础，但平台验证码类型远不止滑块：

| 类型 | 平台 | 现状 | 方案 |
|------|------|------|------|
| 滑块拼图 | XHS, 抖音 | MediaCrawler 有 | 保留，接入 Pinchtab 的 human.MouseMove |
| 点选文字 | 抖音, B站 | 无 | 接第三方打码 API 或本地 OCR |
| 旋转图片 | 快手 | 无 | 图像相似度匹配 |
| 行为验证 | 全平台 | 无 | L3 行为层覆盖 |
| 短信验证 | 全平台 | 无 | 接码平台 API（手动兜底） |

    50→## 3. L1 — 网络身份层
    51→
    52→### 3.1 现状问题
    53→
    54→- MediaCrawler 有代理池但无 TLS 指纹管理
    55→- Pinchtab 完全没有代理支持
    56→- 抖音、小红书已部署 JA3/JA4 指纹检测 — Go `net/http` 或 Python `httpx` 的 TLS 握手特征与真实 Chrome 完全不同，换再多 IP 也会被识别
    57→
    58→### 3.2 设计方案
    59→
    60→#### 3.2.1 统一代理路由器 (Proxy Router)
    61→
    62→从 MediaCrawler `proxy_ip_pool.py` 移植核心逻辑到 Go，作为 Pinchtab sidecar：
    63→
    64→```
    65→packages/proxy-router/
    66→├── pool.go           # IP 池管理（加载、验证、过期刷新）
    67→├── providers/
    68→│   ├── provider.go   # Provider 接口
    69→│   ├── kuaidaili.go  # 快代理
    70→│   ├── wandou.go     # 豌豆 HTTP
    71→│   └── static.go     # 静态 IP 列表（自建代理）
    72→├── selector.go       # 选择策略（随机/轮询/地域亲和）
    73→└── middleware.go      # HTTP 中间件，自动注入代理
    74→```
    75→
    76→关键特性：
    77→- 身份绑定：同一 Identity Profile 始终使用同一出口 IP（直到过期）
    78→- 自动刷新：过期前 30s 预加载新 IP，零中断切换
    79→- 健康检查：定期验证代理可用性，自动剔除失效 IP
    80→- 地域亲和：中国平台优先使用国内 IP

    82→#### 3.2.2 TLS 指纹伪装
    83→
    84→两种策略并行：
    85→
    86→| 场景 | 方案 | 说明 |
    87→|------|------|------|
    88→| 浏览器模式 | Pinchtab chromedp | 真实 Chrome TLS 栈，天然一致 |
    89→| API 直连模式 | utls (Go) | 模拟 Chrome/Firefox 的 ClientHello |

    93→- HTTP/2 SETTINGS frame 顺序匹配

   169→## 5. L3 — 行为指纹层
   170→
   171→### 5.1 现状问题
   172→
   173→Pinchtab 的 Bézier 鼠标轨迹和打字模拟是好的基础，但平台检测的是更高层的行为模式：
   174→- 打开页面后直接定位目标元素 → 机器人
   175→- 从不滚动页面 → 机器人
   176→- 每次停留时间完全一样 → 机器人
   177→- 没有鼠标闲置微动 → 机器人
   178→
   179→### 5.2 设计方案 — 行为模拟增强
   180→
   181→在 Pinchtab 的 `internal/human/` 基础上扩展：
   182→
   183→```
   184→internal/human/

   191→
   192→#### 5.2.1 滚动行为 (scroll.go)
   193→
   194→```go
   195→// 模拟真实用户滚动：惯性、回滚、速度变化
   196→type ScrollBehavior struct {
   197→    Direction    string   // "down", "up", "mixed"
   198→    TotalPixels  int      // 总滚动距离
   199→    Segments     []Segment // 分段：快速浏览 → 减速 → 停留 → 继续
   200→}
   201→
   202→type Segment struct {
   203→    Pixels   int           // 本段滚动距离
   204→    Duration time.Duration // 本段耗时
   205→    Pause    time.Duration // 段后停留
   206→    Ease     string        // "inertia", "linear", "decelerate"
   207→}
   208→```
   209→
   210→关键特性：
   211→- 惯性滚动：快速滑动后逐渐减速，模拟触控板/鼠标滚轮物理特性
   212→- 回滚：5-10% 概率向上回滚一小段（用户回看内容）
   213→- 速度变化：浏览区域快速滚动，内容区域减速
   214→- 停留点：在图片、标题等元素附近自动停留 1-5 秒
   215→
   216→#### 5.2.2 页面浏览模式 (browse.go)
   217→
   218→定义三种浏览模式，根据任务类型选择：
   219→
   220→| 模式 | 行为 | 适用场景 |
   221→|------|------|---------|
   222→| casual | 先滚动浏览全页 → 随机点击 1-2 个非目标链接 → 返回 → 定位目标 | 搜索结果页 |
   223→| focused | 快速扫描 → 直接定位目标区域 → 短暂停留后操作 | 详情页数据提取 |
   224→| explorer | 多次滚动 → 随机停留 → 偶尔悬停在元素上 → 缓慢浏览 | 发现页/推荐流 |
   225→
   226→#### 5.2.3 闲置行为 (idle.go)
   227→
   228→用户不操作时也有微小动作：
   229→- 鼠标微漂移：每 2-8 秒移动 1-5 像素
   230→- 页面焦点切换：偶尔触发 blur/focus 事件（模拟切换标签页）
   231→- 滚动位置微调：±10-30 像素的小幅滚动
   232→
   233→#### 5.2.4 时间分布 (timing.go)
   234→
   235→所有时间参数使用正态分布或对数正态分布，不使用固定值：
   236→
   237→```go
   238→// 页面停留时间：对数正态分布，中位数 8 秒，长尾到 60 秒
   239→func PageDwellTime() time.Duration {
   240→    mu, sigma := 2.08, 0.7  // ln(8) ≈ 2.08
   241→    sample := math.Exp(rand.NormFloat64()*sigma + mu)
   242→    return time.Duration(clamp(sample, 2, 60)) * time.Second
   243→}
   244→
   245→// 两次操作间隔：正态分布，均值 1.5 秒，标准差 0.5 秒
   246→func ActionInterval() time.Duration {
   247→    sample := rand.NormFloat64()*0.5 + 1.5
   248→    return time.Duration(clamp(sample, 0.3, 5.0) * float64(time.Second))
   249→}
   250→```

   252→## 6. L4 — 平台签名层
   253→
   254→### 6.1 核心策略
   255→
   256→放弃纯算法逆向，统一走 Pinchtab 浏览器内执行。原因：
   257→1. 抖音 a_bogus 每隔几周更新混淆，纯逆向维护成本极高
   258→2. Pinchtab 是真实 Chrome 环境，直接调用平台自己的签名函数
   259→3. XHS 的 `window.mnsv2` 方案可推广到所有平台
   260→
   261→### 6.2 各平台签名方案
   262→
   263→#### 6.2.1 小红书 (XHS)
   264→
   265→| 签名字段 | 生成方式 | 来源 |
   266→|----------|---------|------|
   267→| x-s | Playwright 注入 `window.mnsv2` | MediaCrawler `playwright_sign.py` |
   268→| x-t | 时间戳 | 同上 |
   269→| x-s-common | CRC32 + 自定义 Base64 | MediaCrawler `xhs_sign.py` |
   270→| x-b3-traceid | 随机 hex | 同上 |
   271→
   272→迁移方案：将 `playwright_sign.py` 的 JS 注入逻辑移植为 Pinchtab evaluate 调用：
   273→
   274→```
   275→POST /tabs/{id}/evaluate
   276→{
   277→  "expression": "window._webmsxyw(url, data)",
   278→  "awaitPromise": true
   279→}
   280→```
   281→
   282→`x-s-common` 的纯算法部分（`xhs_sign.py` 中的 `mrc` + `b64_encode`）可移植为 Go 实现，无需浏览器。
   283→
   284→#### 6.2.2 抖音 (Douyin)
   285→
   286→| 签名字段 | 生成方式 | 说明 |
   287→|----------|---------|------|
   288→| a_bogus | 浏览器内执行 | 重度混淆 JS，必须在真实浏览器环境运行 |
   289→| msToken | Cookie 获取 | 首次访问时由服务端下发 |
   290→| ttwid | Cookie 获取 | 设备标识 Cookie |
   291→| _signature | 浏览器内执行 | 部分接口需要 |
   292→
   293→对抗策略：
   294→1. 用 Pinchtab 打开抖音页面，等待 JS 加载完成
   295→2. 通过 evaluate 注入 hook，拦截签名函数
   296→3. 后续请求直接调用被 hook 的函数生成签名
   297→4. msToken/ttwid 通过 Cookie 管理自动维护
   298→
   299→```javascript
   300→// 注入到抖音页面的 hook 脚本
   301→(function() {
   302→  const origFetch = window.fetch;
   303→  window.__signQueue = [];
   304→  window.fetch = function(url, opts) {
   305→    // 拦截已签名的请求，提取签名参数
   306→    if (url.includes('a_bogus')) {
   307→      const params = new URLSearchParams(url.split('?')[1]);
   308→      window.__lastSign = {
   309→        a_bogus: params.get('a_bogus'),
   310→        msToken: params.get('msToken')
   311→      };
   312→    }
   313→    return origFetch.apply(this, arguments);
   314→  };
   315→})();
   316→```

   323→| img_key / sub_key | API 获取 | 从 nav 接口获取，拼接后取特定位置字符 |
   324→| buvid3/buvid4 | Cookie | 设备标识 |

   329→- buvid 通过 Pinchtab Cookie 管理维护

   332→// wbi 签名核心逻辑（可纯算法实现）
   333→func WbiSign(params map[string]string, imgKey, subKey string) string {
   334→    mixinKey := getMixinKey(imgKey + subKey)
   335→    // 参数排序 + 拼接 + MD5
   336→    sorted := sortParams(params)
   337→    query := encodeQuery(sorted) + mixinKey
   338→    return md5sum(query)
   339→}

   496→### 8.2 自适应限速引擎
   497→
   498→不使用固定 interval，而是根据平台反馈动态调整：
   499→
   500→```
   501→信号检测 → 策略决策 → 频率调整
   502→```
   503→
   504→| 信号 | 含义 | 响应动作 |
   505→|------|------|---------|
   506→| HTTP 200 连续成功 | 安全 | 缓慢提速（每 10 次成功 +5%，不超过人类速率上限） |
   507→| HTTP 429 / 频率限制 | 被限流 | 指数退避（2s → 4s → 8s → 16s） |
   508→| 出现验证码 | 被怀疑 | 降速 50% + 触发 Challenge Solver |
   509→| HTTP 403 / 封禁 | 身份暴露 | 立即切换身份 + 代理，原身份进入冷却 |
   510→| 返回空数据/假数据 | 静默风控 | 最危险，需要数据校验层检测 |
   511→| 连接重置 / TLS 错误 | 网络层封锁 | 切换代理 IP，检查 TLS 指纹 |
   512→
   513→```go
   514→// 自适应限速核心逻辑
   515→type RateLimiter struct {
   516→    baseInterval time.Duration  // 基础间隔
   517→    currentRate  float64        // 当前速率倍数 (0.1 ~ 1.0)
   518→    successCount int            // 连续成功次数
   519→    platform     string         // 平台标识
   520→}
   521→
   522→func (r *RateLimiter) OnResponse(status int, hasCaptcha bool) {
   523→    switch {
   524→    case status == 200 && !hasCaptcha:
   525→        r.successCount++
   526→        if r.successCount%10 == 0 && r.currentRate < 1.0 {
   527→            r.currentRate = min(r.currentRate*1.05, 1.0)
   528→        }
   529→    case status == 429:
   530→        r.currentRate *= 0.5
   531→        r.successCount = 0
   532→    case hasCaptcha:
   533→        r.currentRate *= 0.5
   534→        r.successCount = 0
   535→    case status == 403:
   536→        r.currentRate = 0.1  // 降到最低
   537→        r.successCount = 0
   538→    }
   539→}
   540→
   541→func (r *RateLimiter) NextInterval() time.Duration {
   542→    jitter := time.Duration(rand.Float64() * 0.3 * float64(r.baseInterval))
   543→    return time.Duration(float64(r.baseInterval)/r.currentRate) + jitter
   544→}
   545→```
