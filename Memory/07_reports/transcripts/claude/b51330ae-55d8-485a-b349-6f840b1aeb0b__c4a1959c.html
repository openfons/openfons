# Automating Changesets: Key Concepts

## Recommended Automation Flow

The quickest path involves two steps:

1. Install the changeset bot into your repository
2. Add the GitHub Action to your repository

## Ensuring Pull Requests Have Changesets

There are two main approaches:

**Non-blocking approach:** Pull requests can merge without changesets, and missing changesets don't fail CI. The changeset bot prompts contributors and offers maintainers a convenient link to add changesets themselves.

**Blocking approach:** To enforce changesets, add this CI step:
```
changeset status --since=main
```

This exits with code 1 if no new changesets exist since the main branch. For changes that shouldn't trigger releases (like test or build-only updates), use `changeset --empty` to create a special changeset that doesn't release anything.

## Running Version and Publish Commands

**Using the GitHub Action:**
The action automatically creates and maintains a version PR with an up-to-date `changeset version` run. It recreates the PR when merged and optionally handles releases when changes reach the base branch.

**Manual workflow alternative:**
If you prefer not using the action, the recommended process involves a release coordinator who:
- Halts merging to the base branch
- Pulls the branch, runs `changeset version`, and creates a versioning PR
- Merges versioning changes back
- Pulls again and runs `changeset publish`
- Pushes release tags with `git push --follow-tags`
- Resumes merging

The documentation notes this manual approach requires multiple pulls and can be finicky, so teams should adapt it to their circumstances.

# Snapshot Releases Overview

Snapshot releases enable testing of changes without permanent version updates. The workflow uses modified versioning and publishing commands to generate temporary release versions.

## Core Workflow

The process involves two main steps:

1. **Versioning**: Run `yarn changeset version --snapshot` to set all package versions to `0.0.0-TIMESTAMP` format. You can customize this with a tag: `yarn changeset version --snapshot [tag-name]` produces versions like `0.0.0-tag-name-TIMESTAMP`.

2. **Publishing**: Execute `changeset publish --tag [tag-name]` to release packages. The `--tag` flag prevents the snapshot from becoming the `latest` npm release, which is critical—without it, standard installations would pull the snapshot version.

## Key Alternative: `--no-git-tag` Flag

For teams publishing snapshots locally or from CI environments, the `--no-git-tag` flag offers flexibility. When running `changeset publish --no-git-tag`, the tool "skip[s] creating git tags for published snapshot packages." This allows stable releases to maintain proper git tags while snapshot releases avoid cluttering the repository.

## Important Branching Consideration

Unlike standard releases, snapshot branch changes should **not** be merged to main. The documentation emphasizes: "the snapshot is intended for installation only, not to represent the correct published state of the repo." Teams should preserve the version and tag information separately without integrating snapshot changes back into the primary codebase.

# Changesets Release Action Overview

**How It Works:**
The action monitors your repository's base branch for changesets. When new changesets are detected, it automatically creates or updates a pull request with bumped package versions and updated changelogs. Once merged, you can either publish manually or have the action handle publishing.

**Key Configuration Options:**

- **publish** – Command to build and publish packages (optional)
- **version** – Command for versioning; defaults to `changeset version` if not provided
- **commit** – Commit message (defaults to "Version Packages")
- **title** – PR title (defaults to "Version Packages")
- **setupGitUser** – Configures git user as "github-actions[bot]" (defaults to true)
- **createGithubReleases** – Whether to create GitHub releases after publishing (defaults to true)
- **commitMode** – Choose between "git-cli" or "github-api" for pushing changes (defaults to git-cli)
- **cwd** – Specify project directory if not at repository root

**Publishing Behavior:**
The action cannot skip PR creation entirely. However, it offers two publishing approaches:

1. **Automatic Publishing** – Provide the `publish` input with your build/publish command. The action creates the PR, and upon merge, automatically publishes.

2. **Custom Publishing** – Omit the `publish` input and use the `hasChangesets` output to trigger your own publishing logic in a separate workflow step when changesets are absent.

**Outputs:**
- `published` – Boolean indicating if publishing occurred
- `publishedPackages` – JSON array of published packages with names and versions

# Changesets Version & Publish Workarounds for Protected Branches

## Core Challenge

The fundamental issue is that "you need to bump versions and remove the changeset files from git - both of those usually require an additional commit to be created." This creates friction in protected branch workflows where direct commits aren't allowed.

## Suggested Workarounds

**Auto-merge approach:** One practical solution involves using GitHub Actions to automatically merge the version PR. A workflow listens for the Release action completion, then uses `actions/github-script` to merge the changeset-release PR via the GitHub API with a squash merge strategy.

**Combined workflow:** A more comprehensive approach chains multiple steps: create the version PR, automatically merge it, then immediately run the publish step. This achieves the desired "version and publish" sequence in a single workflow execution, eliminating the manual approval step between operations.

**Configuration flag:** Community members suggested adding an `automerge: true` configuration option to changesets, allowing developers to skip manual PR approval while still maintaining an audit trail of version changes.

## Architectural Considerations

Maintainers noted that supporting true combined execution would require architectural changes—specifically, tracking package versions via git tags rather than updating `package.json` files, which would eliminate the need for version-bump commits entirely.

## 1. Recommended Workflow for Changesets in a Small Team / Solo Developer Monorepo

The standard changesets workflow is:

1. Make changes on a feature branch.
2. Run changeset (or changeset add) to create a changeset markdown file describing the bump type and summary.
3. Commit the changeset file alongside your code changes.
4. Merge to main.
5. Either the CI creates a "Version Packages" PR, or you run changeset version locally.
6. After versioning, changeset publish pushes to npm.

For solo developers, the workflow can be simplified:
- Set "commit": true in .changeset/config.json so that changeset version auto-commits the version bumps and changelog updates, reducing manual git steps.
- Use changeset status before merging to preview what will be released.
- Use changeset --empty for commits that should not trigger a release (test-only, docs-only changes).
- The ignore config option can exclude internal/non-publishable packages from versioning.

---

## 2. Local changeset version + Commit vs. Letting the GitHub Action Create a Version PR

There are two distinct approaches:

### Approach A: Let changesets/action create a Version PR (the "official" way)

The action detects changeset files on main, runs changeset version, and opens/updates a PR titled "Version Packages." Merging that PR triggers the publish step.

Pros:
- Fully automated -- no local steps beyond adding changeset files.
- Provides a reviewable PR with the exact version bumps and changelog diffs before publishing.
- Works well for teams where multiple people need to review releases.
- The action keeps the PR up to date as more changesets land on main.

Cons:
- Requires GitHub Actions to have permission to create PRs (Settings > Actions > General > "Allow GitHub Actions to create and approve pull requests").
- Adds an extra merge step between "code lands on main" and "packages are published."
- Can feel like overhead for a solo developer -- you're reviewing your own version bump PR.
- Race conditions are possible if multiple merges happen in quick succession (mitigated with the concurrency key in the workflow).

### Approach B: Run changeset version locally, commit, then let CI publish

You run changeset version on your machine, commit the resulting version bumps and changelog updates, push to main, and CI detects there are no remaining changeset files and runs changeset publish.

Pros:
- No PR creation needed from Actions -- works in orgs that restrict Actions from creating PRs.
- Simpler mental model for solo developers: you control exactly when versions are bumped.
- Fewer moving parts in CI.
- You can review the version changes locally before pushing.

Cons:
- Manual step that can be forgotten or done inconsistently.
- The official docs call this approach "finicky" because it requires pulling, versioning, committing, pushing, then publishing in sequence.
- No built-in audit trail of the version bump as a separate reviewable PR.
- You must remember to run git push --follow-tags to push the git tags created by changeset version.

---

## 3. Publishing Directly Without a Version PR (When Actions Cannot Create PRs)

The changesets/action does not have a built-in flag to skip PR creation. However, there are several proven workarounds:

### Option1: Custom CI workflow (recommended for your situation)

Write your own GitHub Actions workflow that runs changeset version and changeset publish directly on main, without using changesets/action at all:

name: Release
on:
  push:
    branches: [main]
concurrency: release-${{ github.ref }}
jobs:
  release:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 22registry-url: https://registry.npmjs.org
      - run: npm ci
      - run: npx changeset version
      - run: |
          git config user.name "github-actions[bot]"
          git config user.email "github-actions[bot]@users.noreply.github.com"
          git add -A
          git diff-index --quiet HEAD || git commit -m "chore: version packages"
          git push- run: npx changeset publish
        env:
          NODE_AUTH_TOKEN: ${{ secrets.NPM_TOKEN }}
      - run: git push --follow-tags

The concurrency key serializes runs to avoid race conditions. This approach versions and publishes in a single workflow run with no PR involved.

### Option 2: Local version + CI publish only

Run changeset version locally, commit and push. In CI, detect that there are no changeset files remaining and run changeset publish. The changesets/action supports this via its hasChangesets output -- when hasChangesets is false, it means versions have already been bumped and you can publish.

### Option 3: Auto-merge the Version PR

If you can enable PR creation but want to skip manual approval, add a step after the changesets action that auto-merges the Version PR using gh pr merge --squash. This is the approach used by TanStack and others.

### Option 4: Run changeset version locally (simplest)

For a solo developer, the simplest path: run changeset version locally when you're ready to release, commit the results, push, and let CI handle only the changeset publish step. This avoids the Version PR entirely and keeps full control in your hands.

## Summary / Recommendation for Your Setup

Given that you are a solo developer on the openmnemo monorepo and may have restrictions on Actions creating PRs, the pragmatic choice is:

- Add changesets during development as usual (changeset add).
- When ready to release, run changeset version locally, review the bumps, commit, and push.
- CI workflow triggers on push to main, detects no remaining changesets, and runs changeset publish.
- Alternatively, use the custom workflow above (Option 1) to have CI do both version + publish in one shot with no PR.

Either way, the concurrency key on the workflow is important to prevent race conditions.
