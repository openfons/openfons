The memorytree-workflow project is a Claude Code skill that provides persistent Git-tracked memory to code repositories. Through recent discussions, several architectural decisions were made that need to be reflected in the documentation:

2. Global singleton: 1 machine = 1 scheduled task. It scans 3 client directories (~/.claude/projects/, ~/.codex/sessions/, ~/.gemini/) and matches transcripts to all registered projects.

3. auto_push = true by default: Memory is a user asset; local loss is unrecoverable. First install prompts user to confirm, then silent execution.

4. Global config at ~/.memorytree/config.toml: heartbeat interval (300s), watch dirs, registered projects, auto_push (true), log_level (info).

5. Alerts system at ~/.memorytree/alerts.json: When push fails N times or other errors accumulate, written here. On Skill activation, displayed to user.

6. Git remote pre-check: Before push, check if remote exists. No remote → skip push, log warning, alert user on next Skill activation.

Use this file when configuring or troubleshooting the MemoryTree background heartbeat.

## Goal
- One sentence: Keep transcript imports, cleaning, and git commits running without model tokens or user intervention.

## Core Rules
1. heartbeat.py runs once per invocation, then exits (stateless, idempotent)
2. OS-native scheduler calls it on interval (cron / launchd / Task Scheduler)
3. Global singleton: 1 machine = 1 scheduled task, scanning all registered projects
4. Client directories scanned: ~/.claude/projects/, ~/.codex/sessions/, ~/.gemini/
5. Each run: scan → incremental import → clean → git commit + push → exit
6. tmux loop mode (watch) exists only for development/debugging, not production

## Daemon CLI
- memorytree-daemon install — detect OS, register scheduled task, prompt user to confirm
- memorytree-daemon uninstall — remove scheduled task
- memorytree-daemon run-once — single manual execution (debugging)
- memorytree-daemon watch — tmux loop mode (dev/debug only)
- memorytree-daemon status — show scheduler registration state + last run result

## Execution Flow
1. Acquire lock file (~/.memorytree/heartbeat.lock) — skip if previous run still active
2. Load config from ~/.memorytree/config.toml
3. For each registered project: discover new transcripts, import, clean, stage Memory/**, commit, push (if auto_push and remote exists)
4. On push failure: log warning, increment failure counter, write to alerts.json after N consecutive failures
5. Release lock, exit

## Sensitive Info Scanning
- During transcript cleaning, scan for common patterns: API keys, passwords, tokens, connection strings
- If found: log warning with file path and line, do NOT auto-delete (avoid false positives)
- User can review warnings in heartbeat log

## Git Remote Pre-Check

- No remote → skip push, log warning, write alert for next Skill activation
