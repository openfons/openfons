- No semantic markers — code/discussion/decision/question types unmarked; embedding might not distinguish "question" queries from "narrative" passages
- High Chinese content — works fine if using multilingual embeddings (e.g., multilingual-e5-large), but monolingual English models would struggle

3. Chunking Strategy for RAG:
- Current format supports: Session-level retrieval (find whole session by topic/date/client)
- Needs enhancement for: Sub-message retrieval (find specific paragraph within a message)
- Recommended: Add hierarchical markers (e.g., ## Topic: ... headers within assistant messages) to enable paragraph-level indexing

### 5. Current Limitations from RAG Perspective

| Aspect | Current | RAG Ideal |
|--------|---------|-----------|
| Metadata per message | Timestamp + role only | Timestamp, role, topic, type, entities |
| Content searchability | Session-level (FTS5 on title/cwd/branch) | Message/paragraph-level content search |
| Semantic markers | None | Topic headers, decision flags, code blocks |
| Chunking boundaries | Message borders | Semantic paragraphs within messages |
| Client/project context | In manifest only | Redundant per-message for faster retrieval |
| Tool events | Excluded (summary only) | Optionally indexed as separate entities |

### 6. Summary

The current clean markdown format is well-designed for archival and conversation browsing but sub-optimal for AI/RAG without enhancement:

- Noise level: Excellent (92-98% signal retention)
- Metadata density: Moderate (session-level good, message-level sparse)
- Embedding compatibility: Works for Chinese/multilingual models; semantic boundaries unclear
- Chunking readiness: Message-level granularity is the boundary; sub-message topics require manual parsing
- Typical token chunk: 100-250 tokens per message; 500-1000 tokens per ideal RAG chunk (3-8 messages)

To optimize for RAG, consider:
1. Adding ## Topic: ... section headers within long assistant messages (minimal encoding overhead)
2. Extending SQLite FTS5 to index clean markdown file content (not just metadata)
3. Adding optional semantic type markers ([DECISION], [QUESTION], [CODE]) inline
4. Generating sub-message anchors (paragraph indices) for finer-grained retrieval
