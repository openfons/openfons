### CRITICAL

1. FTS4 tokenize=unicode61 is silently unsupported in sql.js (db.ts line 111)

FTS4 only ships with the simple, porter, and icu tokenizers. The unicode61 tokenizer is an FTS5 addition. The sql.js WASM build bundles SQLite without the unicode61 extension, so the CREATE VIRTUAL TABLE call will silently succeed (SQLite falls back to simple) or throw a runtime error depending on the build — neither outcome is documented or tested.

Action: Either remove tokenize=unicode61 to use the simple default, or migrate back to FTS5 if the sql.js build supports it. The switch from FTS5 to FTS4 was presumably made because sql.js's default WASM build omits the FTS5 module; that should be confirmed and documented in a comment. There is no test covering the case where tokenize=unicode61 is rejected.

2. searchTranscripts runs CREATE_FTS and REBUILD_FTS on every search call — potentially catastrophic on a large index (db.ts lines 145-146)

REBUILD_FTS causes a full re-index of the entire transcripts table on each call to searchTranscripts. For a 10,000-row database this blocks the event loop for seconds and bloats WAL. Moving the FTS table creation to upsertSearchIndex (triggered only when data changes) and removing the unconditional REBUILD_FTS from searchTranscripts would be the correct fix. Alternatively, use an FTS with content="" (external content) and rebuild only in the write path.

FTS4's MATCH operator uses a mini-query language. Characters like ", , -, (, ) have special meaning. A query such as " (a lone double-quote) will raise a SQLite parse error that propagates as a thrown exception to the CLI, printing a raw SQLite error message to stderr. More problematic: (asterisk-space) or ) can silently corrupt the query semantics. While the binding uses a prepared statement parameter (preventing SQL injection proper), FTS MATCH interprets the parameter value as a query expression — it is not a plain string literal. The error is caught in cmdSearch and printed to stderr, so it is not a security hole, but the user experience is poor and the behavior is undocumented.

Action: Wrap the query in double-quotes to force a phrase search ("${escapedQuery}"), or strip/escape FTS metacharacters before passing them to MATCH. At minimum, document the behavior and add a test for special-character input.
