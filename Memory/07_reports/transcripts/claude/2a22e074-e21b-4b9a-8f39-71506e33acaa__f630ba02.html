## CRITICAL

## HIGH

H1: recall alias r conflicts with report alias rp — no, but r could be expected for report

This isn't a technical conflict (commander will resolve r to recall and rp to report without ambiguity), but from a UX perspective, a user who types openmnemo r expecting report will get recall instead. This is a judgment call — recall is arguably more frequently used in a quick workflow, so it's defensible. Just worth noting.

H2: init alias i shadows potential import alias expectation

import got imp (3 chars) while init got i (1 char). A user might intuitively try openmnemo i somefile expecting import. Again, not a commander conflict, but a UX friction point. init is typically run once per repo, while import is used repeatedly — the more frequent command arguably deserves the shorter alias.

## MEDIUM

M1: locale alias l — low-value command gets a prime single-letter alias

locale is a diagnostic/utility command that most users will rarely invoke directly. Giving it the l alias consumes a prime single-letter that could go to a more common command (e.g., list if one is ever added). Not a bug, but worth reconsidering if the command surface grows.

M2: daemon status alias st — no conflict, but s would clash with top-level search

The subcommand aliases (ins, un, ro, w, st under daemon; b, sv under report) are scoped to their parent command group, so they don't conflict with top-level aliases. This is correct. No issue here — just confirming it's clean.

M3: --no-ai flag inversion logic is correct but non-obvious

Line 268: noAi: opts.ai === false — commander's --no-ai sets opts.ai to false. This works correctly, but the double-negative (noAi is true when opts.ai is false) is a minor readability concern. Consider renaming the internal field to skipAi or ai (inverted) for clarity.

M4: search --limit default fallback inconsistency

Line 161: !Number.isNaN(n) && n > 0 ? n : 20 — if the user passes --limit 0, they get 20 (the fallback), but the help text says "0 = no limit". This is a bug: passing --limit 0 should yield unlimited results, not 20. The condition should be:

limit: !Number.isNaN(n) ? (n === 0 ? 0 : Math.max(n, 1)) : 20,

Or simply: limit: !Number.isNaN(n) && n >= 0 ? n : 20

## LOW

L1: No --version alias -V

Commander adds -V by default for .version(), so this is already handled. No issue.

L2: discover --limit uses parseInt with || 0 fallback (line 122)

This is fine — parseInt('0', 10) returns 0, and 0 || 0 is 0. No bug, just noting the pattern differs from the search limit handling.

L3: Missing alias for import — im would be more intuitive than imp

imp is 3 characters. im would be shorter and equally unambiguous. Minor.

L4: Help text consistency

- discover --client says "all, codex, claude, gemini, doubao" (uses all as default)
- import --client says "auto, codex, claude, gemini, doubao" (uses auto as default)

These are semantically different (all = scan everything, auto = detect from file), so the difference is intentional and correct. Just worth verifying the downstream code handles both keywords.

## Recommended Upgrade Command

The package is @openmnemo/cli published to npm with a bin entry mapping openmnemo to ./dist/main.js. The recommended install/upgrade commands:

# Install globally
npm install -g @openmnemo/cli

# Upgrade to latest
npm update -g @openmnemo/cli

# Or pin a specific version
npm install -g @openmnemo/cli@latest

There's no npx shim or postinstall script — it's a straightforward global npm binary.
