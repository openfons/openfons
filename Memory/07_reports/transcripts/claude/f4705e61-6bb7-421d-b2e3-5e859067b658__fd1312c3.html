   109→## Features
   110→
   111→### Memory Management
   112→
   113→| Memory Type | Behavior |
   114→|-------------|----------|
   115→| **Goals** | Changed only after explicit user confirmation of a scope or requirement change. Versioned. |
   116→| **Todos** | Kept in sync with the active goal. Updated when progress changes or next task becomes clearer. |
   117→| **Chat Logs** | Append-only. Never rewritten or deleted. |
   118→| **Knowledge** | Stores architecture decisions, product ideas, operating constraints. Goals may also hold scope-relevant items. |
   119→| **Archive** | Stale context moved here or to knowledge files to keep active files concise. |
   120→
   121→### Multi-Client Transcript Import
   122→
   123→Import AI chat history from three clients:
   124→
   125→| Client | Source |
   126→|--------|--------|
   127→| **Codex** | `~/.codex/sessions/**/rollout-*.jsonl` |
   128→| **Claude Code** | `transcript_path` if exposed; otherwise `~/.claude/projects/<project>/*.jsonl` |
   129→| **Gemini CLI** | Saved chats first; checkpoints under `~/.gemini/tmp/<hash>/checkpoints` if enabled |
   130→
   131→Key principles:
   132→
   133→- Raw transcripts preserved unchanged as source of truth
   134→- Cleaned Markdown generated by **deterministic code** (not model tokens)
   135→- Current repo only stores transcripts belonging to this project; others go to global archive only
   136→- First import asks once whether raw transcripts may be committed to the repo
   137→
   138→### Cross-Project Search
   139→
   140→When searching all memories or chat history:
   141→
   142→1. Query `search.sqlite` for fast full-text lookup
   143→2. Use `sessions.jsonl` for session-level metadata filtering (client, project, date range)
   144→3. Load matching cleaned Markdown for context; confirm exact wording against raw transcripts
   145→4. Combine with the current repo's `Memory/` files
   146→5. Fall back to scanning `clean/` directory if the index does not exist
   147→
   148→### Session Continuity
   149→
   150→When you open a new session and want to continue where you left off, ask MemoryTree to show your most recent conversation:
   151→
   152→> "Look at my last chat" / "看看我最近的聊天" / "Continue where I left off"
   153→
   154→The skill will:
   155→
   156→1. **Instant sync** — immediately scan all three client directories (not waiting for the scheduled heartbeat)
   157→2. **Locate** — find the latest session for the current project across Claude Code, Codex, and Gemini CLI
   158→3. **Summarize** — generate a continuation summary: what was discussed, what was tried, and what remains unresolved
   159→4. **Handoff** — you can say "continue" and pick up exactly where the previous session stopped
   160→
   161→This works **cross-client**: a Codex session can be recovered from Claude Code, and vice versa.
   162→
   163→For details see [`references/transcript-archive.md`](references/transcript-archive.md) and [`references/heartbeat-scheduling.md`](references/heartbeat-scheduling.md).

   165→### Git Safety
   166→
   167→MemoryTree follows strict isolation rules to never interfere with your repo's workflow:
   168→
   169→| Rule | Description |
   170→|------|-------------|
   171→| File isolation | Only stages/commits `Memory/**` (excluding `raw/` until approved) and `AGENTS.md` (when managed by MemoryTree) |
   172→| Branch isolation | Uses dedicated branch `memorytree/<scope>-<date>-<slug>` |
   173→| Commit format | `memorytree(<scope>): <subject>` or repo-compatible equivalent |
   174→| PR isolation | Separate PR for MemoryTree-only changes |
   175→| Auto-merge | Only when repo still enforces its review and CI checks |
   176→| Auto-push | On by default; skips if no remote; fast-forward only |
   177→| Protected branches | Never overrides branch protection or CI rules |
   178→
   179→The skill stops and asks before proceeding in 8 scenarios: mixed product code, AGENTS.md conflicts, unclear target branch, repo forbids bot pushes, CI conflicts, unknown raw transcript permission, cross-project transcripts, or protected branch refusal.
   180→
   181→For the full policy see [`references/git-policy.md`](references/git-policy.md).

   209→## Background Heartbeat
   210→
   211→The heartbeat automates transcript discovery, import, cleaning, commit, and push without consuming model tokens or requiring human intervention.
   212→
   213→**Execution flow**:
   214→
   215→```
   216→Acquire lock → Load config.toml → Iterate projects → Discover → Import
   217→→ Clean → Commit → (auto_push) Push → Release lock → Exit
   218→```
   219→

   221→
   222→```bash

   226→### CLI commands
   227→
   228→| Command                     | Description |
   229→|-----------------------------|-------------|
   230→| `memorytree-daemon install`   | Register the heartbeat with the OS scheduler (cron / launchd / Task Scheduler). |

   241→| `heartbeat_interval` | `"5m"`   | Interval between heartbeat executions. |
   242→| `log_level`          | `"info"` | Log verbosity (`debug`, `info`, `warn`, `error`). |

   244→### Design constraints
   245→
   246→- Single execution, stateless, idempotent — safe to re-run at any interval
   247→- One computer = one scheduled heartbeat task
   248→- Will not silently register — always confirms before `install`
