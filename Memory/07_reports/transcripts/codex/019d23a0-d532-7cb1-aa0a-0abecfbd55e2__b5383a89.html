## 1. 文档目的

本文件用于统一团队对以下问题的理解：

1. 我们到底在做什么系统
2. 哪些是当前已冻结的架构原则
3. 哪些能力是第一阶段必须落地的
4. 当前规划中明确采用、按需启用、参考借鉴了哪些开源技术

本文件优先基于当前仓库的根规划文档整理，不把临时脑暴或尚未进入主规划的探索项目写成既定事实。

## 1.1 当前工程阶段

当前仓库仍处于 Greenfield Bootstrap 阶段，意味着：

1. 当前最确定的是架构边界、真源边界、迁移边界和第一阶段范围。
2. 当前最不应该误读的是“目标态能力已经全部实现”。
3. 所有技术清单都必须区分：当前主线假设、按需启用备选、参考借鉴。

## 1.2 编排决策状态

当前规划资料里同时存在 `Airflow` 和 `Temporal` 两条历史来源，因此这里明确收口如下：

1. 对 OpenFons 当前主线假设，按现有 SoT 采用 `Airflow` 作为当前唯一编排真源。
2. `Temporal` 保留为历史方案中的强参考和后续演进备选，不作为当前 bootstrap 默认编排基线。
3. 如果未来要从 `Airflow` 切换到 `Temporal`，必须通过单独的 ADR / RFC 和迁移评审，不允许两套编排长期并行。

## 2. 项目定位

OpenFons 要做的不是一个更强的抓取脚本仓库，而是一个 source-native 平台：

`自然语言 / 界面 / API -> TaskSpec / WorkflowSpec -> Skill / Worker 执行 -> Evidence 沉淀 -> Artifact 交付`

目标场景包括：

1. 品牌舆情监控
2. 商品评价分析
3. 竞品情报研究
4. 热点事件追踪
5. 长期 watch 与周期性交付

## 3. 北极星与非目标

### 3.1 北极星

我们要建立的是：

1. 可把用户意图编译为结构化任务的控制层
2. 可稳定执行多源采集与分析的执行层
3. 可沉淀 Topic / Evidence / Artifact 的证据层
4. 可输出多种交付物的交付层

### 3.2 明确非目标

以下方向不应作为主架构：

1. 不做“大模型直接驱动浏览器”的默认执行架构
2. 不做“一个万能 Agent 统管所有任务”的黑盒系统
3. 不做“只靠脚本和临时工具拼起来”的生产链路
4. 不把 `gold_tool_entities` 之类的兼容视图当最终真源

## 4. 架构总原则

### 4.1 四大平面

1. `Access Plane`
   IM / Web / API / 外部系统入口
2. `Control Plane`
   Intent Router、Task Compiler、Policy Gate、Scheduler
3. `Execution Plane`
   Source Registry、Skill、Worker、风控对抗与适配层
4. `Data & Delivery Plane`
   Topic、TopicRun、Evidence、EvidenceSet、Artifact、Renderer、Portal

### 4.2 统一编译链

无论入口来自哪里：

1. Skill
2. Web 界面
3. API

最终都必须进入同一条链路：

`Intent -> TaskSpec -> WorkflowSpec -> Worker Execution -> Evidence -> Artifact`

### 4.3 角色边界

1. Agent 负责理解、拆解、编译与策略决策
2. Worker 负责确定性执行
3. Skill 是可复用能力单元
4. LLM 浏览器自动化只能作为受控兜底 Skill，不是默认执行层

## 5. 真源模型

### 5.1 任务编译真源

1. `TaskSpec`：这次任务是什么
2. `WorkflowSpec`：这次任务怎么跑
3. `ReportSpec`：这次任务怎么交付

### 5.2 研究与交付真源

1. `Topic`
2. `TopicRun`
3. `Evidence`
4. `EvidenceSet`
5. `Artifact`

### 5.3 真源规则

1. `Source Registry` 是来源配置真源
2. `TaskSpec / WorkflowSpec / ReportSpec` 是任务编译真源
3. `Topic / TopicRun / Evidence / Artifact` 是研究与交付真源
4. 兼容视图只能作为迁移层，不得冒充最终真源

## 6. 第一阶段工程范围

### 6.1 必须优先落地

1. `packages/contracts`
2. `packages/domain-models`
3. 最小 shared 基础模块
4. 最小执行闭环
5. 最小控制面 `control-web`
6. 最小运维面 `ops-web`

### 6.2 第一阶段执行闭环

建议保持克制，只跑通：

`seed -> discovery -> fetch-http -> normalize -> quality -> publish-evidence`

### 6.3 暂缓事项

第一阶段不应同时做大的浏览器链路、多媒体链路、复杂门户、大而全后台。

## 8. 当前主规划采用与按需扩展策略

### 8.1 当前主规划

当前主规划强调：

1. 先把契约和数据模型立住
2. 先跑通确定性执行闭环
3. 先保留控制面和运维面的最小可用界面
4. 先让 Skill、界面和 API 共用同一条编译层

### 8.2 按需启用

以下能力属于按条件启用，不应在第一阶段预埋复杂度：

1. Kafka
2. 更强检索与分析组件
3. 浏览器自动化主链路
4. 更复杂的门户与商业化边界能力

## 10. 开源技术使用约束

### 10.1 一律先区分“主规划”与“参考”

任何组件进入主线前，都必须先回答：

1. 它是第一阶段默认启用，还是按需启用
2. 它是正式组件，还是参考项目
3. 它是否会改变当前的真源边界和执行边界

### 10.2 一律坚持单一真源

1. 编排真源不能双轨并行
2. 来源配置真源不能双轨并行
3. 研究与交付真源不能退化成兼容视图

### 10.3 一律先契约后实现

1. 先定义 `TaskSpec / WorkflowSpec / ReportSpec`
2. 再定义 `Topic / TopicRun / Evidence / Artifact`
3. 再推进 Worker、Skill、UI 和 API

### 10.4 一律禁止“AI 直接统管执行”

LLM 只能做意图理解、任务分解、策略规划和中间态编译，不得直接成为默认浏览器执行主链。

## 11. 近期落地建议

### 11.1 技术优先级

1. 先把 `contracts` 和 `domain-models` 立住
2. 先做最小控制面与运维面骨架
3. 先跑通最小 Worker 闭环
4. 先补测试和可观测护栏

### 11.2 不建议的做法

1. 不要一开始就同时接入所有来源
2. 不要一开始就引入所有可选组件
3. 不要为了演示效果，把参考项目直接塞进主仓变成“伪实现”
4. 不要在没有中间态契约的情况下先堆界面

## 1. 一句话说明

OpenFons 要做的不是一个更强的爬虫工具，而是一套面向企业研究、舆情监控、商品评价分析与情报生产的 source-native 平台：

`用户意图 -> 结构化任务 -> 确定性执行 -> 证据沉淀 -> 多形态交付`

目标态将同时支持自然语言、界面和 API 三种入口，但底层统一进入同一条任务编译与执行链路。

## 1.1 当前状态说明

为了避免误读，先明确两点：

1. 当前仓库是新产品线的 Greenfield 启动仓，不是已经商业化交付完成的成熟平台。
2. 下面文档中的很多能力描述，属于明确的目标态和产品路线，而不是“今天已经全部上线”。

## 2. 我们看到的问题

今天企业在做品牌监测、竞品分析、商品评价研究、热点事件跟踪时，普遍会遇到四类问题：

1. 信息源分散在微博、小红书、抖音、电商平台、新闻站点等多个渠道，人工收集成本高。
2. 传统爬虫工具只能“抓到数据”，不能把数据稳定转成可复用的研究资产。
3. 纯大模型工作流虽然灵活，但执行不可控、不可审计、成本波动大。
4. 很多报告没有证据链，难以复核，无法沉淀为长期可运营的情报资产。

这意味着市场真正缺的，不是一个新的采集脚本仓库，而是一套能把采集、分析、审计、交付连起来的平台。

## 3. OpenFons 的解决方案

OpenFons 的产品定义是：

> 用户通过自然语言、界面或 API 提交研究任务，系统将需求编译为结构化计划，调用 Skill 和 Worker 完成多源信息接入、采集执行、分析研判、证据沉淀与交付。

它的核心设计不是“让大模型直接做所有事情”，而是：

1. Agent 负责理解需求和编译计划。
2. Worker 负责确定性执行。
3. Evidence 负责作为最小证据真源。
4. Artifact 负责作为正式冻结的交付物。

这样既保留 AI 的灵活性，又保留企业级系统必须具备的可控性、可审计性和可复现性。

## 4. 目标态产品形态

OpenFons 不是单一形态产品，而是一个统一底座上的多入口平台。

### 4.1 用户入口

1. 自然语言入口：适合快速发起研究任务。
2. 控制面界面：适合像传统 SaaS 一样点击操作、查看运行、重试、取消。
3. API 入口：适合集成到企业现有系统。

### 4.2 平台核心

1. Control Plane：理解意图，生成 `TaskSpec / WorkflowSpec / ReportSpec`。
2. Execution Plane：调用 Skill 与 Worker 完成采集、分析和质量控制。
3. Data & Delivery Plane：沉淀 `Topic / TopicRun / Evidence / Artifact`，输出报告、图表、导出文件、预警消息。

### 4.3 交付结果

1. 舆情日报与预警
2. 商品评价分析报告
3. 竞品对比研究报告
4. CSV / JSON / Markdown / PDF 导出
5. Dashboard 与长期 watch 结果

## 4.4 当前已完成与正在建设

当前更接近下面这个状态：

1. 已完成：项目命名、主仓定位、核心架构原则、迁移边界、核心对象模型与阶段路线规划。
2. 正在建设：`TaskSpec / WorkflowSpec / ReportSpec`、`Topic / TopicRun / Evidence / Artifact` 等核心契约与模型。
3. 尚未完整落地：成熟控制面、成熟运维面、完整 Skill 体系、长期 watch、正式交付中心。

## 5. 为什么是现在

我们认为现在是这个项目的窗口期，原因有三点：

1. AI 让“意图输入”大幅简化，但企业依然需要稳定、审计和成本可控的执行系统。
2. 公开信息源越来越多，企业对品牌、市场、竞争和舆情的持续感知需求显著增长。
3. 市场上的很多产品要么偏“采集工具”，要么偏“聊天研究助手”，中间缺少一层“证据优先、可长期运营的平台化系统”。

OpenFons 试图补上的，正是这层中间市场。

## 7. 核心差异化

### 7.1 不是“采集工具”，而是“研究资产平台”

平台核心对象是 `Topic / TopicRun / Evidence / Artifact`，而不是一堆临时抓取结果。

### 7.2 不是“纯 Agent”，而是“Agent 决策 + Worker 执行”

大模型不直接驱动浏览器，不直接充当主执行层，避免高成本和低可控。

### 7.3 不是“只有聊天”，而是“聊天、界面、API 三入口统一”

无论入口来自哪里，最终都进入同一条编译链，不会出现一套给聊天、一套给后台的双轨系统。

### 7.4 不是“报告先行”，而是“证据先行”

所有交付物都需要回到 Evidence 证据集，适合严肃研究、审计与复核场景。

### 7.5 不是“一次性任务”，而是“长期 watch 与持续消费”

平台路线从第一天就围绕长期主题、周期运行、历史沉淀和门户化交付来规划。

## 8. 商业模式方向

当前更适合采用“开源底座 + 商业化能力”的组合路径。

### 8.1 开源层

1. 开源主仓和基础能力
2. 统一契约、基础 Worker、基础 Skill
3. 社区可运行版本和标准部署骨架

### 8.2 商业层

1. SaaS 托管版本
2. 私有化部署与企业支持
3. 高级控制面与运维能力
4. 高价值连接器、策略中心、审计与治理能力
5. 行业模板和报告交付能力

## 9. 护城河判断

OpenFons 的护城河不只来自某一个模型或某一个抓取器，而来自四层叠加：

1. 多源适配与执行策略积累
2. 稳定的任务编译中间态与平台契约
3. 证据真源与交付真源体系
4. 长期 watch、质量门禁、运维治理带来的平台经验

换句话说，真正难复制的不是“能抓”，而是“能长期稳定地抓、分析、交付并沉淀”。

- Source Registry 维护每个平台的 adapter。

- Artifact renderer 再把它们渲染成 Next.js 网页、文章、视频脚本。
