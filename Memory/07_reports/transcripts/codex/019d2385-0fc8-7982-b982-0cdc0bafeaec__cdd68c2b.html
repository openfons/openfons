## Git And Safety

1. Never assume MemoryTree can override the repo's branch model.
2. Default to isolated MemoryTree-only Git operations: stage only MemoryTree-owned files, use a MemoryTree-scoped commit title, push to a dedicated branch, and open a dedicated PR.
3. Enable auto-merge only for MemoryTree-only PRs, and only when the repository still enforces its required review and CI checks.
4. Keep repo-local raw transcript mirrors and the global transcript archive synchronized on every import. The global index must remain safe for concurrent multi-project updates.
5. Keep the repository mirror limited to transcripts that belong to the current repository. Unrelated or ambiguous transcripts belong in the global archive only.
6. Keep raw transcript files under `Memory/06_transcripts/raw/**` in the repo, but exclude them from automatic staging until the user explicitly approved raw transcript uploads for that repository.
7. If the diff mixes product code, shared policy files, cross-project transcript archives, or unclear ownership, stop and ask the user before committing or pushing.
8. If the repo has protected branches or PR-only rules, follow them even if older MemoryTree rules say otherwise.
9. When `auto_push` is enabled, the heartbeat process pushes automatically after committing. If no Git remote is configured, the push is skipped. If a push fails, the heartbeat retries once. In either case an alert is written to `~/.memorytree/alerts.json`. See `references/git-policy.md`.
10. During transcript cleaning, the heartbeat scans for sensitive information (API keys, passwords, tokens). Matches are logged as warnings only 鈥?no automatic deletion or redaction. See `references/heartbeat-scheduling.md`.

11. Do not modify the host repository's CI or E2E workflow definitions by default. If the user explicitly approves a MemoryTree-only optimization, prefer path-filtered or lightweight workflows for `Memory/**` and managed `AGENTS.md` PRs without weakening required branch protection.

### 5.1 先做 skill-ready foundation，不做无限重构

本阶段不追求一次性把所有视频域代码重写完，而是优先完成对 skill 直接必要的工程改造。

### 5.2 不把自然语言理解直接绑定到旧脚本

自然语言编辑不应直接翻译成一次性 `clip_text.txt` 或旧脚本命令，而应先落到结构化编辑对象，再执行渲染。

### 5.3 模型升级不是第一优先级

更强模型可以提高效果，但无法替代工程契约、结构化中间产物、服务边界和稳定执行链路。

### 5.4 旧链路保留过渡价值，但不应成为最终能力层

旧 `stage0~9` 可作为过渡执行引擎的一部分，但不应成为 OpenClaw skill 直接调用的最终接口。

## 6. 目标架构

建议收敛为下面这条主链路：

1. `source asset`
2. `transcript/timeline`
3. `clip plan`
4. `subtitle draft`
5. `cover spec`
6. `render manifest`
7. `final render`

其中关键点是把可编辑能力前移到渲染之前。自然语言的职责不是直接操作 ffmpeg，而是修改结构化 artifact。

### 6.1 核心 artifact

建议优先定义 5 个结构化中间产物：

- `transcript.json`
  保存分句、时间戳、说话人、原始识别文本、标准化文本、token 级或字级时间信息。

- `clip_plan.json`
  保存选中的片段、片段顺序、裁切范围、偏移量、静音处理策略、保留原因等。

- `subtitle_draft.json`
  保存字幕文本、分段结果、时间范围、样式配置、是否自动换行、是否高亮等。

- `cover_spec.json`
  保存封面标题分行、文案来源、字体、颜色、布局、背景图来源、展示时长等。

- `render_manifest.json`
  保存最终渲染所需的输入引用、版本号、依赖 artifact、输出参数、渲染模式等。

### 6.2 建议的数据流

建议数据流如下：

1. 输入视频
2. 生成 `transcript.json`
3. 基于 transcript 生成 `clip_plan.json`
4. 基于 clip plan 和 transcript 生成 `subtitle_draft.json`
5. 基于论坛信息或文案生成 `cover_spec.json`
6. 将上述 artifact 汇总为 `render_manifest.json`
7. 渲染最终视频

后续多轮自然语言编辑，优先修改的是：

- `clip_plan.json`
- `subtitle_draft.json`
- `cover_spec.json`

而不是重新强行走完整旧流水线。

## 7. 分阶段路线

## Phase 0：Skill-ready foundation

### 目标

先把旧链路里最影响稳定性和服务化的问题止血，让视频处理能力具备被上层接口稳定调用的基础。

### 核心任务

- 修复 FunClip 字幕切分、时间戳和状态契约问题
- 替换不安全的 `eval()` 反序列化
- 收敛 `videoclipper` 中 `sentences/zimu` 双轨状态
- 统一 `sd_switch` 等参数语义
- 去掉危险清理逻辑和全局副作用
- 让 `stage5`、`stage6` 从交互式脚本变为非交互接口

### 输出

- 稳定的 transcript 生成结果
- 稳定的字幕分句与时间戳契约
- 非交互的剪辑和文案生成入口
- 不依赖人工输入的可编排流水线最小闭环

### 验收信号

- FunClip 相关单测恢复稳定
- `stage5`、`stage6` 可完全参数化调用
- worker 可以稳定触发视频处理而不依赖命令行交互

## Phase 1：结构化 artifact 落地

### 目标

建立自然语言可编辑的结构化中间层，为后续服务接口和 skill 打底。

### 核心任务

- 定义 `transcript.json` schema
- 定义 `clip_plan.json` schema
- 定义 `subtitle_draft.json` schema
- 定义 `cover_spec.json` schema
- 定义 `render_manifest.json` schema
- 将旧链路输出逐步映射到这些 artifact

### 输出

- 一套版本化 artifact 契约
- 一条从输入视频到 artifact 的生成链路
- 一组可供上层编辑的标准对象

### 验收信号

- 任一视频任务执行后，至少能完整产出 5 个核心 artifact
- 片段编辑、字幕编辑、封面编辑都可在不重跑全链路的情况下完成对象层更新

## Phase 2：服务化接口

### 目标

把 artifact 生产、读取、修改和渲染能力收敛为服务接口，而不是脚本入口。

### 核心任务

- 增加视频上传入口或对象存储接入入口
- 新增“分析视频并产出 artifact”接口
- 新增“更新 clip plan”接口
- 新增“更新 subtitle draft”接口
- 新增“更新 cover spec”接口
- 新增“触发渲染”接口
- 新增“查询 artifact 与渲染状态”接口

### 推荐接口族

- `upload_video`
- `analyze_video`
- `get_artifacts`
- `update_clip_plan`
- `update_subtitle_draft`
- `update_cover_spec`
- `render_video`
- `get_render_result`

### 验收信号

- 不经过命令行交互也能完成上传、分析、编辑、渲染
- 上层系统只需要操作接口和 artifact，不需要了解旧流水线内部细节

## Phase 3：OpenClaw skill

### 目标

基于稳定接口与 artifact，构建真正可多轮编辑的 OpenClaw 剪辑 skill。

### skill 应具备的能力

- 接收上传视频
- 调用分析接口获取 transcript 和初稿
- 将自然语言解释为对 `clip_plan/subtitle_draft/cover_spec` 的修改
- 支持多轮追改
- 在用户确认后触发渲染

### 明确限制

skill 不直接驱动旧 `stage0~9`。
skill 的职责是编排接口和修改 artifact，而不是耦合底层脚本细节。

### 验收信号

- 能完成“上传视频并初步生成剪辑、字幕、封面”
- 能完成“把第二段字幕缩短一点”
- 能完成“封面改成三行，第二行强调热词”
- 能完成“把第三段镜头提前 0.5 秒”
- 最终可渲染输出并回读结果

## 8. 第一批开发任务清单

以下任务建议作为当前最优先一批工作项：

### A. FunClip 止血

- 修复 `zimu.py` 的强制分割、时间戳刷新与中英混合切分问题

- 去除 `stage5.py` 中的模型选择、编码选择、预处理选择交互

### C. artifact 设计

- 先写 5 个 artifact 的 schema 初稿
- 明确 artifact 之间的引用关系与版本字段
- 明确哪些字段允许自然语言编辑，哪些字段只能由系统生成

### D. 服务接口最小闭环

- 明确上传能力方案
- 实现分析接口
- 实现 artifact 查询接口
- 实现 3 类编辑接口
- 实现最终渲染接口

## 9. 当前不建议做的事情

- 不建议先做大规模模型替换
- 不建议先做完整视频域大重构
- 不建议先把新架构壳全部补完再开始
- 不建议先做 OpenClaw skill 的 UI 或提示词细节
- 不建议让 skill 直接调用旧 `stage0~9`

## 10. 当前推荐执行顺序

建议严格按以下顺序推进：

1. 完成 Phase 0 止血
2. 完成 artifact 设计和落地
3. 完成服务化最小闭环
4. 再实现 OpenClaw skill

如果资源有限，优先顺序进一步细化为：

1. FunClip 契约修复
2. `stage5/stage6` 非交互化
3. artifact schema
4. 服务接口
5. OpenClaw skill

## 11. 下一步立即执行项

下一轮开发建议直接从下面 3 件事启动：

1. 先列出 Phase 0 的精确改造清单，按文件拆到可编码级别。
2. 同时起草 5 个 artifact 的 schema 初稿。
3. 基于 schema 反推服务接口和 skill 工具边界。

这 3 件事完成后，再进入实际编码改造。

## 12. 计划结论

当前视频域最合理的路线不是“先做 skill，再倒逼底层修补”，而是“先把底座补到可供 skill 调用，再做 skill”。

具体来说：

- 先补稳定性
- 再补结构化
- 再补服务接口
- 最后对接 OpenClaw skill

这条路线既能控制风险，也最符合当前代码现实与文档共识。
