## Quick Start

### 1. Install

```bash
npm install -g @openmnemo/cli
```

Verify:

```bash
openmnemo --version
```

### 2. Initialize your project

Run this inside any Git repository you want to track:

```bash
openmnemo init --root . --project-name "my-project"
```

This creates a `Memory/` directory structure in your repo for storing transcripts, notes, and reports.

### 3. Import your AI conversations

Auto-discover and import all local transcripts (Claude Code, Codex, Gemini CLI):

```bash
openmnemo discover --root .
```

To import a specific file manually:

```bash
openmnemo import --source /path/to/transcript.jsonl --root .
```

```bash
openmnemo report build --root . --output ./Memory/07_reports --no-ai
```

# Register with the OS scheduler (cron / launchd / Task Scheduler)

## Commands

### `openmnemo init`

Initialize a MemoryTree workspace in a repository.

```bash
openmnemo init --root <path> [options]

Options:
  --root <path>           Repository root (default: .)
  --project-name <name>   Project name shown in reports
  --goal-summary <text>   Initial goal description
  --locale <locale>       Language: auto, en, or zh-cn (default: auto)
  --force                 Overwrite existing generated files
```

### `openmnemo discover`

Scan your home directory for AI transcripts and import new ones.

```bash
openmnemo discover --root <path> [options]

Options:
  --root <path>           Repository root (default: .)
  --client <client>       Filter by client: all, codex, claude, gemini (default: all)
  --scope <scope>         current-project or all-projects (default: all-projects)
  --limit <n>             Import at most N transcripts
  --format <format>       Output: text or json (default: text)
```

### `openmnemo import`

Import a single transcript file.

```bash
openmnemo import --source <path> --root <path> [options]

Options:
  --source <path>         Path to the transcript file (required)
  --root <path>           Repository root (default: .)
  --client <client>       Override auto-detection: codex, claude, gemini, doubao
```

### `openmnemo report build`

Generate a static HTML dashboard from imported transcripts.

```bash
openmnemo report build --root <path> [options]

Options:
  --root <path>           Repository root (default: .)
  --output <path>         Output directory (default: ./Memory/07_reports)
  --no-ai                 Skip AI summaries (no API key needed)
  --model <model>         AI model for summaries (default: claude-haiku-4-5-20251001)
  --locale <locale>       Report language: en or zh-CN (default: en)
```

### `openmnemo report serve`

Serve the built report locally over HTTP.

```bash
openmnemo report serve --dir ./Memory/07_reports --port 10086
```

Then open `http://localhost:10086` in your browser.

### `openmnemo recall`

Show the most recent session summary (useful in AI agent hooks).

```bash
openmnemo recall --root . --project-name "my-project"
```

### `openmnemo search`

Full-text search over all imported transcripts (searches title, cwd, and branch).

```bash
openmnemo search --query "authentication bug"
```

Options:
```
--query <text>          Search query (required). Multiple words = AND match.
--global-root <path>    Override global transcript root
--limit <n>             Maximum results (default: 20)
--format <format>       Output: text or json (default: text)
```

> **Note:** The query is split into terms by whitespace; all terms must appear in the document (AND logic). Example: `--query "auth bug"` matches sessions where both `auth` and `bug` appear in the title, cwd, or branch.

## 记忆体系技术架构

### 三路混合检索

```
用户查询
    ↓
并行三路检索：
  ① SQLite FTS4    → 关键词精确匹配（title/cwd/branch/content/commit_layer）
  ② sqlite-vec     → 语义向量相似度（embedding 1536维）
  ③ SQLite 图关系表 → 实体关系追踪（人物/项目/技术/概念/session/commit）
    ↓
RRF（Reciprocal Rank Fusion）合并排序
    ↓
返回 top-N session，附带摘要 + 关联 commit
```

Phase 0 三路检索全部基于 `better-sqlite3` 单一引擎，共用一个 .sqlite 文件，零额外进程。

### 图 Schema

节点类型：`Person` / `Project` / `Technology` / `Concept` / `Session` / `Commit`

关系类型：
- `(Person)-[:DISCUSSED]->(Concept)`
- `(Person)-[:WORKED_ON]->(Project)`
- `(Session)-[:CONTAINS]->(Concept)`
- `(Session)-[:RELATED_TO]->(Commit)`
- `(Concept)-[:DEPENDS_ON]->(Technology)`
- `(Concept)-[:EVOLVED_FROM]->(Concept)`
- `(Project)-[:USES]->(Technology)`

### LLM 提取层

heartbeat 同步时，用 Haiku 从对话全文提取：
- **关键事实** → embedding → sqlite-vec
- **实体 + 关系** → 节点/边 → SQLite 图关系表（graph_nodes + graph_edges）
- **对话全文** → FTS 索引 → SQLite FTS

### SQLite 引擎迁移

Phase 0 关键变更：`sql.js`（WASM）→ `better-sqlite3`（原生绑定）

迁移原因：
- `sqlite-vec` 是 C 扩展，需要原生 SQLite，无法在 sql.js WASM 中加载
- `better-sqlite3` 同步 API，性能比 sql.js 快 5-10x
- 有 prebuild 二进制，CI 无需 node-gyp 编译
- FTS4 + sqlite-vec + 图关系表共用同一个 SQLite 实例，事务一致性好

### 图存储方案（Phase 0）

Phase 0 不引入专用图数据库（kuzu npm 包已废弃），改用 SQLite 关系表模拟：

```sql
CREATE TABLE graph_nodes (
  id TEXT PRIMARY KEY,       -- 'person:张三'
  type TEXT NOT NULL,        -- Person | Project | Technology | Concept | Session | Commit
  properties TEXT            -- JSON
);

CREATE TABLE graph_edges (
  source_id TEXT NOT NULL,
  target_id TEXT NOT NULL,
  type TEXT NOT NULL,         -- DISCUSSED | WORKED_ON | CONTAINS | ...
  properties TEXT,
  PRIMARY KEY (source_id, target_id, type)
);
```

Phase 0 数据规模（个人用户，<1万节点）下，SQL JOIN 查询与专用图库性能差距为零。
图索引是**派生数据**（源自对话记录），随时可删除重建，迁移零风险。

### 可插拔存储后端（适配器模式）

| 阶段 | SQLite 引擎 | 向量后端 | 图后端 | 部署方式 |
|------|------------|---------|--------|---------|
| Phase 0（个人） | `better-sqlite3` | `sqlite-vec` 扩展 | SQLite 关系表 | 单文件，零额外进程 |
| Phase 1（团队） | `better-sqlite3` | `Qdrant` | `Neo4j Community` | Docker 独立服务 |
| Phase 1+（托管） | — | `Pinecone` / `Weaviate` 等 | `Memgraph` / `Neo4j Aura` | 云托管 |

所有后端实现统一的 `SearchAdapter` / `VectorAdapter` / `GraphAdapter` 接口，配置文件一行切换，上层代码零感知。`factory.ts` 根据配置创建适配器实例（工厂模式）。升级后端时需运行迁移脚本重新 index（数据源不变，仅重建索引）。

## AI Chat 功能架构

### 战略定位

报告页 AI Chat 是**独立 Web 客户端的 0 号原型**：
- 先在现有报告页嵌入 chat widget，让用户养成"用自然语言查记忆"的习惯
- server 端 `/api/chat` 就是未来 Web 客户端的后端 API 雏形
- 前端 chat widget 代码直接演进为独立客户端的核心组件

### 两种部署模式

```
本地版（openmnemo report serve）
  用户查询
      ↓
  POST /api/chat  →  server 端
      ↓                  ↓ 1. SQLite FTS 检索相关 session（RAG context）
      ↓                  ↓ 2. 拼装 system prompt + context + 用户问题
      ↓                  ↓ 3. 调用 Claude API（streaming）
  SSE stream  ←──────────┘
  前端逐字渲染

GitHub Pages 版（纯静态）
  用户查询
      ↓
  检测无本地 server → 降级模式
      ↓ 1. 客户端 JSON 索引搜索（build 时预生成）
      ↓ 2. 展示相关 session 列表（无 LLM 调用）
      ↓    或：用户本地起 openmnemo proxy，前端连本地代理
  前端渲染结果
  （注：Anthropic API 不支持浏览器直连，GitHub Pages 版 Phase 0.6 先做搜索展示，LLM 调用 Phase 1 解决）
```

### 技术选型

**自研，不引入第三方 chat UI 框架**，原因：
- 现有报告系统是纯 inline HTML + vanilla JS，零构建工具依赖
- 引入 React/Vue 组件库会破坏零依赖设计，增加维护成本
- 自研可完全控制交互形态、API 设计，直接演进为独立客户端

**实现规模估算**：
- `packages/report/src/render/chat.ts` — chat widget HTML/CSS/JS（~150行）
- `packages/cli/src/cmd-report.ts` — 扩展 `/api/chat` 端点（~120行）
- `packages/report/src/chat/` — server 端 RAG + streaming 逻辑（~150行）

function normalizeMixedHits(
  hits: DataLayerSearchHit[],
  limit?: number,
): DataLayerSearchHit[] {

    hits
      .filter((hit) => hit.ref.kind !== 'session' && typeof hit.session?.session_id === 'string')
      .map((hit) => hit.session!.session_id),
  )

  return hits
    .filter((hit) => !(hit.ref.kind === 'session'
      && typeof hit.session?.session_id === 'string'
      && coveredSessions.has(hit.session.session_id)))
    .sort(compareMixedHits)
    .slice(0, toEffectiveRetrievalLimit(limit))
}
