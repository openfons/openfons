1. 浏览器自动化、视频下载、基础搜索、媒体下载、OCR 和部分平台 SDK 在当前环境下是可落地的。

5. `Reddit`、`X`、`TikTok`、`Facebook` 都存在更明显的账号、会话、风控或匿名访问限制。

7. `Camoufox` 应保留为高阻力站点兜底，不适合作为默认浏览器链路。
8. 第一阶段最值得先打透的仍是 `YouTube + Reddit + Hacker News + Amazon`，再逐步推进 `X + TikTok`。
9. `MediaCrawler` 对国内新媒体平台采集很关键，但它更适合做"国内平台适配器参考样板"和实验室基线，不适合未经改造直接进入 OpenFons 商业主线。
10. `pinchtab` 的 Docker 服务化路线已经可以进入候选底座，但必须把 token、allowedDomains 和实例安全策略当成一等配置项。
11. `camoufox` 现在可以确认"本机真实可用"，适合进入后续 `MediaCrawler + Camoufox` 的受控联调阶段。
12. `scrapling` 作为新验证的反检测采集框架，与 `camoufox` 形成技术路线互补：`camoufox` 走浏览器自动化路线，`scrapling` 走 HTTP 客户端+自适应解析路线。建议后续在高阻力站点场景下对比两者的成功率、成本和易用性。

## 1. 重新收口后的目标

本轮补充文档明确一件事：

OpenFons 的底层本质仍然是多来源采集与信息搜集系统，但平台价值不止于“抓到数据”，而是要把采集结果清洗为证据，再经过人工 + AI 的深度分析，最终生产为可分发、可复用、可变现的内容资产。

## 2. 核心原则

### 2.1 采集仍是底层，不是终点

1. 平台底层仍以 crawler / adapter / browser automation / media download 为基础能力。
2. 采集层的职责是稳定地获得公开信息、媒体素材、评论、时间线和上下文。
3. 采集结果必须沉淀为可复核的证据，而不是直接跳到“自动生成结论”。

### 2.2 分析必须坚持人工 + AI 协同

1. AI 负责聚类、提炼、比对、生成候选结论和辅助写作。
2. 人工负责判断研究问题、修正误读、挑选证据、确定叙事和最终发布。
3. 任何面向 SEO 或 YouTube 变现的内容，都不应是无人工判断的大批量模板化输出。

### 2.3 第一优先交付物是网页型“投行式报告”

1. 最重要的内容产物不是普通博客，而是结构化、可索引、可复核、适合长期沉淀的报告页。
2. 报告页应具备类似投行 PPT 的叙事节奏，但交付形式优先选择 Next.js 网页，而不是静态 PDF。
3. 视频、文章、博客、newsletter、知识卡片等，属于从报告页进一步派生的内容层。

### 2.4 北美 niche 市场优先

1. 主题发现、关键词体系、平台优先级和变现逻辑都优先围绕北美市场设计。
2. 优先选择英文互联网中可持续产出利基主题、用户讨论和消费信号的平台。
3. 不以“覆盖所有平台”为初始目标，而以“找到高价值主题并稳定产出内容资产”为目标。

## 3. 端到端生产链路

### 3.1 主链路

建议将主链路明确为：

`niche discovery -> source collection -> normalization -> evidence set -> human + AI analysis -> report spec -> Next.js report -> video/article derivatives -> distribution -> monetization`

### 3.2 分层职责

1. `Discovery`
   发现值得研究和发布的北美 niche 主题、产品、品牌、趋势、事件或讨论群体。
2. `Collection`
   从多个来源抓取帖子、评论、视频、商品评价、媒体附件和时间线。
3. `Normalization`
   将不同来源的原始结果映射到统一字段和统一证据模型。
4. `Evidence`
   建立主题级证据集，为后续分析、审计和内容引用提供真源。
5. `Analysis`
   由人工 + AI 对证据做聚类、对比、归因、评分、叙事和结论构建。
6. `Report Rendering`
   将分析结论渲染为 Next.js 报告页，作为主要公开交付。
7. `Derivative Content`
   从报告页反向生成文章、博客、视频脚本、讲解视频和社媒分发素材。
8. `Monetization`
   通过 SEO 流量、YouTube 播放量、广告、赞助、联盟、线索与高阶付费内容获取收益。

## 5. 多爬虫协调架构

### 5.1 不做单一万能爬虫

建议明确放弃“一个 crawler 统管所有平台”的思路，改为“多 crawler / 多 adapter / 多执行策略协同”。

### 5.2 建议的 worker 分工

1. `api-worker`
   适用于 Reddit、Hacker News、部分 Amazon、部分 YouTube 场景。
2. `http-worker`
   适用于普通 HTML 页面、静态详情页、搜索结果页和列表页。
3. `browser-worker`
   适用于动态页面、登录态页面、瀑布流、滚动加载和 JS 渲染页面。
4. `media-worker`
   适用于视频、音频、字幕、缩略图和媒体附件下载。
5. `comment-worker`
   适用于评论树、回复层级、评论展开和情绪信号抓取。
6. `normalize-worker`
   适用于统一字段映射、清洗、去重、语言识别、实体提取。
7. `analysis-worker`
   适用于聚类、主题归纳、观点分布、证据打标签。
8. `artifact-worker`
   适用于报告页、文章、脚本、视频分镜和导出产物生成。
9. `cn-social-worker`
   适用于小红书、抖音、快手、B站、微博、贴吧、知乎等国内平台，建议借鉴 `MediaCrawler` 的登录态保存、平台拆分与 WebUI/CLI 结构，但实现保持自研。

### 5.3 调度策略

每个平台适配器都应遵守统一调度顺序：

1. `API-first`
   有稳定官方接口或成熟封装时优先走 API。
2. `HTTP-second`
   当页面稳定且无需浏览器渲染时优先走轻量抓取。
3. `Browser-fallback`
   当页面强依赖 JS、登录态或滚动加载时再进入 Playwright。
4. `Anti-detect-fallback`
   只有在常规 Playwright 无法稳定获取结果时，才进入 Camoufox 等高成本兜底组件。

### 5.4 Camoufox 的定位

`Camoufox` 不应成为默认浏览器底座，而应定位为：

1. 高阻力页面的受控兜底能力
2. 只在特定 source、特定任务、特定代理策略下启用
3. 由平台层显式记录启用原因、成本、成功率与风险

推荐角色是：

`Playwright 默认执行 -> Pinchtab/Browser Bridge -> Camoufox 特种兜底`

而不是：

`所有浏览器任务都走 Camoufox`

| 队列与状态 | `Valkey` | 去重、节流、冷却、运行态计数 |
| 真源存储 | `PostgreSQL` | 任务、证据、产物和元数据真源 |
| 原始对象存储 | `SeaweedFS` | raw HTML、JSON、截图、媒体对象 |
| 站内搜索 / 检索 | `OpenSearch` | 后续报告与证据检索 |
| 报告站 | `Next.js` | 公开网页报告、SEO 落地页、内容分发载体 |

### 6.2 建议按需启用

| 层 | 按需组件 | 启用条件 |
| --- | --- | --- |
| 浏览器反检测 | `Camoufox` | 常规 browser worker 成功率不足时 |
| 浏览器桥接服务 | `Pinchtab` | 需要把浏览器能力以独立 Docker 服务方式暴露给 worker 或 agent 时 |
| 爬虫运维面 | `Crawlab` | worker 数量增多、运行态管理变复杂时 |
| 网页提取 | `Firecrawl` | 需要统一的 scrape/search/extract 接口时 |
| 长任务编排 | `Temporal` | 进入更复杂的人机协同和可回放流程时 |
| 图像/媒体补充 | `gallery-dl` | 多图片、多素材、多站点附件抓取时 |

### 6.3 不建议的做法

1. 不要在第一阶段同时接完所有平台。
2. 不要把 `Camoufox` 作为所有浏览器任务的统一入口。
3. 不要把参考仓库直接嵌进主仓充当“伪实现”。
4. 不要在证据模型还没稳定前就大规模铺内容生成。
5. 不要忽略 `pinchtab` 默认 token 鉴权和 allowedDomains 这类安全边界，避免把 Docker 化浏览器直接暴露成无保护的公共入口。

## 7. Next.js 报告站设计

### 7.1 为什么是 Next.js

Next.js 适合这条路线的原因在于：

1. 适合输出可索引的 HTML 页面，而不是只产出 PDF。
2. 适合静态生成、增量更新和主题化内容站。
3. 官方支持 `metadata`、`sitemap`、`robots` 等 SEO 基础能力。
4. 适合同时承载报告页、专题页、目录页、视频落地页和文章页。

### 7.2 报告站建议信息架构

建议在现有双后台之外，新增一个面向外部内容分发的 `report-web`：

```text
apps/
  control-web/
  ops-web/
  report-web/
```

`report-web` 建议承载以下路由：

1. `/reports/[slug]`
   单篇核心报告页
2. `/topics/[slug]`
   长期追踪主题页
3. `/markets/[slug]`
   某一垂直市场或 niche 聚合页
4. `/companies/[slug]`
   品牌、产品或竞品档案页
5. `/videos/[slug]`
   对应讲解视频落地页
6. `/sources/[platform]/[id]`
   可公开的证据引用页或来源说明页

### 7.3 “投行式报告页”建议结构

一篇主报告页不应只是博客长文，而应更接近结构化 deck：

1. 开场结论：一句话 thesis 和核心判断
2. Executive Summary：3 到 5 条最重要发现
3. Market Snapshot：市场规模、趋势、信号摘要
4. Competitive Landscape：竞品对比表、品牌定位图、价格带和功能带
5. Voice of Customer：来自评论、论坛、视频评论的观点聚类
6. Timeline / Momentum：时间线、热度变化、传播节点
7. Risks / Unknowns：结论边界、证据缺口和未证实假设
8. Evidence Appendix：关键来源、截图、链接、引用摘要
9. Update Log：版本、更新时间和主题持续跟踪记录

### 7.4 渲染链建议

报告页生成不应直接从 raw data 到最终页面，而应至少经过：

`EvidenceSet -> Analysis Brief -> Human Review -> ReportSpec -> Next.js Renderer`

这样才能保证：

1. 结论不是裸模型幻觉
2. 页面结构可复用
3. 视频、博客、newsletter 都能从同一份 `ReportSpec` 派生

## 8. 视频生产链

### 8.1 视频不是独立体系，而是报告页的派生产物

建议把视频生产明确为：

`ReportSpec -> VideoOutline -> Script -> Storyboard -> Voiceover / Editing -> YouTube Publish`

### 8.2 视频生产建议流程

1. 先完成网页报告页
2. 从报告页提取 5 到 10 分钟讲解结构
3. 生成脚本初稿、镜头卡和图表切换点
4. 由人工审阅论点、删去空话和模板化表述
5. 再进入旁白、配图、剪辑、字幕和发布

### 8.3 视频变现约束

1. 视频内容必须有清晰叙事和独立价值，不能只是网页朗读。
2. 必须加入人工选择、人工判断和人工编排，避免模板化批量生成。
3. 视频与网页之间应相互导流，但内容不应只是机械镜像。

## 9. SEO 与内容变现边界

### 9.1 SEO 原则

1. 公开站点要服务真实用户检索意图，而不是只做程序化收录。
2. 每篇报告页都应有独立主题、独立证据、独立结论和独立图表。
3. 不应批量生成只换关键词、不换观点的“伪研究页”。

### 9.2 YouTube 原则

1. 平台路线应避免“模板化、大批量、低差异”的 AI 视频。
2. 视频必须体现人工选题、证据筛选、观点组织和讲解价值。
3. 收益能力来自持续的高质量选题和讲解，而不是纯流水线堆量。

### 9.3 当前推荐的收益组合

1. Google SEO 流量
2. YouTube 播放和频道收益
3. 行业 newsletter / 博客导流
4. 高价值主题报告合集或会员化内容
5. 后续再考虑赞助、联盟、咨询线索和企业版能力

## 10. 工程路线建议

### 10.1 Phase 0：跑通最小闭环

1. 优先接入 `YouTube / Reddit / Hacker News / Amazon`
2. 跑通 `discovery -> collect -> normalize -> evidence -> report`
3. 先以人工驱动的视频制作流程验证内容产出质量
4. 先做少量高质量主题页，而不是大量低质量页面

### 10.2 Phase 1：形成稳定内容生产线

1. 加入 `X` 与 `TikTok` 的趋势信号采集
2. 建立主题库、报告模板库、图表模板库
3. 把 `ReportSpec` 扩展到文章、博客、newsletter
4. 开始做主题级长期 watch 和周更内容节奏

### 10.3 Phase 2：扩大来源和分发

1. 根据收益表现决定是否补 `Facebook` 和更多论坛源
2. 强化视频生产、频道矩阵和系列化栏目
3. 构建更成熟的 SEO 内链、专题页、聚合页和更新策略
4. 再评估是否需要更复杂的编排和运维体系

## 11. 当前最值得坚持的判断

1. OpenFons 不是“内容自动生成器”，而是“采集驱动的研究内容工厂”。
2. 最核心的资产不是单次文章，而是 `Topic / Evidence / ReportSpec` 这套可复用中间层。
3. 最强的壁垒不是单一爬虫，而是“北美 niche 主题选择 + 多源证据整合 + 人工 + AI 分析 + 报告与视频双重交付”。
4. 如果这条路线成立，OpenFons 不仅能做研究平台，也能做内容资产平台和收益型媒体基础设施。

## 10. 开源技术使用约束

### 10.1 一律先区分“主规划”与“参考”

任何组件进入主线前，都必须先回答：

1. 它是第一阶段默认启用，还是按需启用
2. 它是正式组件，还是参考项目
3. 它是否会改变当前的真源边界和执行边界

### 10.2 一律坚持单一真源

1. 编排真源不能双轨并行
2. 来源配置真源不能双轨并行
3. 研究与交付真源不能退化成兼容视图

### 10.3 一律先契约后实现

1. 先定义 `TaskSpec / WorkflowSpec / ReportSpec`
2. 再定义 `Topic / TopicRun / Evidence / Artifact`
3. 再推进 Worker、Skill、UI 和 API

### 10.4 一律禁止“AI 直接统管执行”

LLM 只能做意图理解、任务分解、策略规划和中间态编译，不得直接成为默认浏览器执行主链。

## 1. 一句话说明

OpenFons 要做的不是一个更强的爬虫工具，而是一套面向企业研究、舆情监控、商品评价分析与情报生产的 source-native 平台：

`用户意图 -> 结构化任务 -> 确定性执行 -> 证据沉淀 -> 多形态交付`

目标态将同时支持自然语言、界面和 API 三种入口，但底层统一进入同一条任务编译与执行链路。

## 1.1 当前状态说明

为了避免误读，先明确两点：

1. 当前仓库是新产品线的 Greenfield 启动仓，不是已经商业化交付完成的成熟平台。
2. 下面文档中的很多能力描述，属于明确的目标态和产品路线，而不是“今天已经全部上线”。

## 2. 我们看到的问题

今天企业在做品牌监测、竞品分析、商品评价研究、热点事件跟踪时，普遍会遇到四类问题：

1. 信息源分散在微博、小红书、抖音、电商平台、新闻站点等多个渠道，人工收集成本高。
2. 传统爬虫工具只能“抓到数据”，不能把数据稳定转成可复用的研究资产。
3. 纯大模型工作流虽然灵活，但执行不可控、不可审计、成本波动大。
4. 很多报告没有证据链，难以复核，无法沉淀为长期可运营的情报资产。

这意味着市场真正缺的，不是一个新的采集脚本仓库，而是一套能把采集、分析、审计、交付连起来的平台。

## 3. OpenFons 的解决方案

OpenFons 的产品定义是：

> 用户通过自然语言、界面或 API 提交研究任务，系统将需求编译为结构化计划，调用 Skill 和 Worker 完成多源信息接入、采集执行、分析研判、证据沉淀与交付。

它的核心设计不是“让大模型直接做所有事情”，而是：

1. Agent 负责理解需求和编译计划。
2. Worker 负责确定性执行。
3. Evidence 负责作为最小证据真源。
4. Artifact 负责作为正式冻结的交付物。

这样既保留 AI 的灵活性，又保留企业级系统必须具备的可控性、可审计性和可复现性。

## 4. 目标态产品形态

OpenFons 不是单一形态产品，而是一个统一底座上的多入口平台。

### 4.1 用户入口

1. 自然语言入口：适合快速发起研究任务。
2. 控制面界面：适合像传统 SaaS 一样点击操作、查看运行、重试、取消。
3. API 入口：适合集成到企业现有系统。

### 4.2 平台核心

1. Control Plane：理解意图，生成 `TaskSpec / WorkflowSpec / ReportSpec`。
2. Execution Plane：调用 Skill 与 Worker 完成采集、分析和质量控制。
3. Data & Delivery Plane：沉淀 `Topic / TopicRun / Evidence / Artifact`，输出报告、图表、导出文件、预警消息。

### 4.3 交付结果

1. 舆情日报与预警
2. 商品评价分析报告
3. 竞品对比研究报告
4. CSV / JSON / Markdown / PDF 导出
5. Dashboard 与长期 watch 结果

## 4.4 当前已完成与正在建设

当前更接近下面这个状态：

1. 已完成：项目命名、主仓定位、核心架构原则、迁移边界、核心对象模型与阶段路线规划。
2. 正在建设：`TaskSpec / WorkflowSpec / ReportSpec`、`Topic / TopicRun / Evidence / Artifact` 等核心契约与模型。
3. 尚未完整落地：成熟控制面、成熟运维面、完整 Skill 体系、长期 watch、正式交付中心。

## 5. 为什么是现在

我们认为现在是这个项目的窗口期，原因有三点：

1. AI 让“意图输入”大幅简化，但企业依然需要稳定、审计和成本可控的执行系统。
2. 公开信息源越来越多，企业对品牌、市场、竞争和舆情的持续感知需求显著增长。
3. 市场上的很多产品要么偏“采集工具”，要么偏“聊天研究助手”，中间缺少一层“证据优先、可长期运营的平台化系统”。

OpenFons 试图补上的，正是这层中间市场。

## 6. 目标客户与典型场景

### 6.1 目标客户

1. 品牌与市场团队
2. 电商与产品团队
3. 咨询与研究机构
4. 企业战略、风险、公共事务团队
5. 有长期舆情、竞品、情报跟踪需求的中大型组织

### 6.2 核心场景

1. 品牌舆情监控：持续跟踪品牌声量、负面波动、热点话题。
2. 商品评价分析：提取质量、价格、服务、物流等方面级反馈。
3. 竞品情报研究：汇总多平台信息，形成可交付的竞品分析材料。
4. 热点事件跟踪：形成时间线、传播路径、情绪走向与预警。

## 7. 核心差异化

### 7.1 不是“采集工具”，而是“研究资产平台”

平台核心对象是 `Topic / TopicRun / Evidence / Artifact`，而不是一堆临时抓取结果。

### 7.2 不是“纯 Agent”，而是“Agent 决策 + Worker 执行”

大模型不直接驱动浏览器，不直接充当主执行层，避免高成本和低可控。

### 7.3 不是“只有聊天”，而是“聊天、界面、API 三入口统一”

无论入口来自哪里，最终都进入同一条编译链，不会出现一套给聊天、一套给后台的双轨系统。

### 7.4 不是“报告先行”，而是“证据先行”

所有交付物都需要回到 Evidence 证据集，适合严肃研究、审计与复核场景。

### 7.5 不是“一次性任务”，而是“长期 watch 与持续消费”

平台路线从第一天就围绕长期主题、周期运行、历史沉淀和门户化交付来规划。

## 8. 商业模式方向

当前更适合采用“开源底座 + 商业化能力”的组合路径。

### 8.1 开源层

1. 开源主仓和基础能力
2. 统一契约、基础 Worker、基础 Skill
3. 社区可运行版本和标准部署骨架

### 8.2 商业层

1. SaaS 托管版本
2. 私有化部署与企业支持
3. 高级控制面与运维能力
4. 高价值连接器、策略中心、审计与治理能力
5. 行业模板和报告交付能力

## 9. 护城河判断

OpenFons 的护城河不只来自某一个模型或某一个抓取器，而来自四层叠加：

1. 多源适配与执行策略积累
2. 稳定的任务编译中间态与平台契约
3. 证据真源与交付真源体系
4. 长期 watch、质量门禁、运维治理带来的平台经验

换句话说，真正难复制的不是“能抓”，而是“能长期稳定地抓、分析、交付并沉淀”。

## 10. 当前阶段与近期里程碑

### 10.1 当前阶段

当前仓库处于 Greenfield Bootstrap 阶段，重点不是堆功能，而是先把未来平台的架构边界和真源边界立住。

当前更准确的说法是：

1. 已有明确产品方向与技术路线。
2. 已有核心规划文档和仓库定位。
3. 正在进入“契约、模型、最小闭环”的工程化阶段。
4. 尚未进入“完整商业产品已上线”的阶段。

### 10.2 第一阶段目标

1. 建立核心契约：`TaskSpec / WorkflowSpec / ReportSpec`
2. 建立核心数据模型：`Topic / TopicRun / Evidence / EvidenceSet / Artifact`

## 1. 结论先行

### 关于 `camoufox`

这轮仍然把 `camoufox` 归入“宿主机优先”而不是 Docker 优先，原因很实际：

1. 本轮真实成功的是宿主机启动、真实打开页面、真实保存截图和 HTML。
2. 后续你们要接账号、cookie、浏览器指纹、可能还要与 `MediaCrawler` 或 CDP 联调，宿主机集成的调试成本更低。
3. `camoufox` 理论上不是不能容器化，但至少在当前批次里，我们没有把 Docker 版跑成主验证路径，因此不能把它写成“已验证的 Docker 部署能力”。
