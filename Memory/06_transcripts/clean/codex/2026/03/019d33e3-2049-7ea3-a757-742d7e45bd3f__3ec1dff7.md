   - 更适合做页面、追踪页、比较器，还是暂不进入公开流程
   - 有没有 `工具站 / 数据产品 / 订阅` 潜力
   它是前置判断的唯一外部真源。

2. `TopicRun / Raw Evidence / Artifact`
   这是原始采集层。
   它保留：
   - 搜索结果
   - HTML
   - 截图
   - 评论
   - 官方价格页 capture
   - 原始文本摘录
   - 运行日志和失败记录
   这一层不负责“结论”，只负责“保真”。

3. `EvidenceSet`
   这是结构化证据层。
   它把原始采集整理成可引用、可复核、可支持 claim 的证据对象。
   页面、视频、文章都不该直接吃 raw evidence，而应该先吃这一层。

4. `ReportSpec`
   这是第一阶段的主交付编译层。
   它决定：
   - 这次页面的标题、slug、受众、地区、语言
   - 页面 section
   - 哪些结论可以写
   - 哪些风险必须保留
   - 更新日志怎么挂
   v1 默认外部盈利主线先走这一层。

5. `Derived Content Specs`
   这是派生内容层。
   它不从用户原话直接生成，而是从 `ReportSpec` 和 `EvidenceSet` 派生：
   - `VideoOutline`
   - `Script`
   - `ArticleBrief`
   - `NewsletterBrief`
   - 以后类似 NotebookLM 风格的多形态内容
   这样能保证多渠道内容共用同一真源，不各说各话。

6. `Product Opportunity Hints`
   这是产品机会层。
   这里不是 v1 立刻单独做一个新正式契约，而是先作为 `OpportunitySpec` 内部字段保留：
   - 能不能长成工具站
   - 能不能长成价格比较器
   - 能不能长成追踪页
   - 能不能长成会员 / 订阅产品
   也就是说，v1 先判断机会，不急着把所有工具化路线单独产品化。

**真源规则**
我建议再写死 4 条判断规则：

1. 用户原话不是最终真源，`OpportunitySpec` 才是
2. 原始采集不是最终结论，`EvidenceSet` 才是
3. 页面不是唯一产物，`ReportSpec` 只是当前第一交付面
4. 视频、文章、博客、newsletter 不能各自重新脑补，必须派生自同一套 `EvidenceSet / ReportSpec`
