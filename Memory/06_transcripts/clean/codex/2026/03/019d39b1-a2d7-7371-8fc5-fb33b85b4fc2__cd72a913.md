**这些英文词，通俗对应什么**
- `OpportunitySpec`
  立项单。就是“这个题要不要做、做哪个方向、给谁看”的说明书。
- `TaskSpec`
  施工任务单。就是“接下来具体要干什么”。
- `WorkflowSpec`
  施工流程单。就是“这些任务按什么顺序跑”。
- `TopicRun`
  一次正式调研任务。比如“AI procurement 这次调研”。
- `SourceCapture`
  单条原始抓取记录。比如某个官网价格页、某篇官方文档、某个社区讨论的抓取结果。
- `CollectionLog`
  采集日志。记录“什么时候抓了、抓成功没、失败没、用了什么方式”。
- `Evidence`
  单条可用证据。不是原始网页本身，而是“从原始资料里整理出的、可引用的一条证据”。
- `EvidenceSet`
  证据包。把一组证据整理好，供报告引用。
- `ReportSpec`
  报告蓝图。决定页面标题、结构、风险提示、更新时间这些。
- `Artifact`
  最终成品。比如网页、文章、视频脚本之类。

   位置在 [report-page.tsx](D:/demo1/openfons/apps/report-web/src/pages/report-page.tsx)

**所以你刚才那 7 步，为什么是对的**
因为你现在最缺的，不是“再写更多说明文档”，也不是“先上数据库”。

你最缺的是：

**先拿出第一条真的、能跑通的证据链。**

也就是先证明这件事真能工作：

`用户问题 -> 立项 -> 调研采集 -> 证据包 -> 报告页面`

一旦这条链通了，后面再加数据库、再做第二案例，都是放大和复用。

如果现在反过来先做持久化，会有两个风险：
- 还没想清楚的数据结构先被写死
- 以后发现 schema 不对，要大改会很痛苦

所以“先跑通一条真链，再固化存储”是合理的。

**还有一个很实际的问题**
你说的第 1 步也确实该先做，因为当前待办里引用了一份“说已经存在”的计划文件，但这个文件实际上不在仓库里。

也就是说：
- todo 里写“有这份计划”
- 实际仓库里没有

这会导致后面执行时大家以为已经有施工图，其实没有。

**我再用一句最通俗的话总结**
现在不是“方向错了”，而是：

**脑子里已经想清楚要盖什么房子了，图纸也大致统一了，但工地上目前只搭了一个样板门脸，还没把地基、钢筋、验收链条真正跑起来。**

所以最合理的下一步就是：

1. 先把缺失的施工计划文件补出来
2. 再把“证据链的数据结构”定下来
3. 再只拿 `AI procurement` 这个案例跑通第一条真链
4. 再让网页真正显示证据来源
5. 最后再上数据库和第二案例
