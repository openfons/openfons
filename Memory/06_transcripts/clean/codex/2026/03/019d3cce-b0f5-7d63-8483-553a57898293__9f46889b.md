**公允判断**

- 这样不只是 skill 能调，后面 `control-api`、批任务、工作流、网页后台、人工审核入口，都能走同一套能力网关。
- 这和你们现在的 SoT 是一致的，因为你们已经在强调 `TaskSpec / WorkflowSpec -> TopicRun / SourceCapture / CollectionLog -> EvidenceSet` 这条受控链，而不是“skill 自己偷偷干活”。  
  相关本地文档已经有这个方向：[开放源平台技术团队说明.md](D:/demo1/openfons/docs/sot/开放源平台技术团队说明.md) [开放源平台当前正式架构说明.md](D:/demo1/openfons/docs/sot/开放源平台当前正式架构说明.md)

**我建议的正确分层**
- `SuperAgent / Planner`
  只负责接收目标、拆任务、决定调用谁，不直接承载所有具体能力。
- `Gateway`
  统一暴露能力入口，例如 `search.run`、`crawl.capture`、`extract.parse`、`sandbox.exec`、`memory.recall`。
- `Capability Providers`
  真正对接 Bing、Google、Baidu、DuckDuckGo、Brave、Tavily、浏览器采集、HTTP 抓取、文件解析等。
- `Skills`
  不是能力本体，而是“组合能力的 SOP/策略包”。比如“competitive-serp-scan”“pricing-capture”“region-policy-check”。
- `Contracts`
  统一输入输出，尤其是 `SearchRequest / SearchResult / SearchRun / SourceCapture / CollectionLog`。
- `Sandbox`
  只给高风险或高耦合动作用，比如浏览器自动化、执行外部代码、复杂采集器。
- `Memory`
  不应该是一团聊天记录，而应该拆成项目记忆、运行记忆、SOP 记忆、证据记忆。

**你的这个想法里，哪些我认为是对的**

- 不要让 `skill` 和 `gateway` 都各自写一套业务逻辑。
