我觉得最值得盯的几类是：

- [Firecrawl](https://github.com/firecrawl/firecrawl)：学它的 `scrape/search/extract` 统一接口和 schema-first 输出，不要把平台价值压缩成“抓网页”。
- [Crawlab](https://github.com/crawlab-team/crawlab)：学它的运行态、日志、重试、分布式 worker 管理，不要把它当成你们的差异化。
- [MediaCrawler](https://github.com/NanmiCoder/MediaCrawler)：学中文平台连接器和反爬实战，不要把“采集仓库”直接等同于“平台产品”。
- [DeerFlow](https://github.com/bytedance/deer-flow) 和 [deep-research](https://github.com/dzhng/deep-research)：学规划层和研究工作流，不要让 Agent 直接绕过 `Worker -> Evidence`。
- [OpenCTI](https://github.com/OpenCTI-Platform/opencti)：学 connector framework、evidence/observable 建模、审计与长期沉淀，不要照搬它偏重的领域模型。
- [Obsei](https://github.com/obsei/obsei)：学监听、告警、低代码编排感，但别把它当现代核心底座。
- [Bright Data competitive-intelligence](https://github.com/brightdata/competitive-intelligence)：值得看它怎么把“采集 + 分析 + 报告”包装成产品，但它更像 demo/reference，不像成熟开源主干。
- [PinchTab](https://github.com/pinchtab/pinchtab)：可以参考 browser 操作抽象，但要谨慎评估它在多租户、分布式和生产治理上的适配度。

**怎么取长补短**

- 取 `Firecrawl` 的接口简洁，别学它“提取优先、证据偏薄”。
- 取 `OpenCTI` 的证据链、连接器和审计模型，别学它一开始就把模型做得过重。
- 取 `Crawlab` 和 `MediaCrawler` 的 source 适配与运行经验，别把 source adapter 当产品本体。
- 取 `DeerFlow / deep-research` 的 planner 层能力，别让 agent 成为默认执行主链，这一点你们现在的文档边界是对的。
- 取 `Obsei` 的持续监听和告警感，别把“报表/聊天”误当护城河。
- 你们真正的护城河，应该继续押在 4 件事上：统一任务编译中间态、Evidence 真源、长期 watch、以及高价值来源连接器。
