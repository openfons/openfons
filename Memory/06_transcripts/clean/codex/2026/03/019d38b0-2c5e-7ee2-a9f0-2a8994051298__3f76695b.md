设计第 2 段：对象设计与审核门禁

先把这条链拆成 6 个对象，职责必须单一，不能混层。

1. `SeedInput`
   只保存用户原始提问，不做解释。
   建议字段：
   - `seed_id`
   - `raw_question`
   - `captured_at`
   - `language`
   - `confirmed_by_user`
   - `notes`
   规则：
   - 这条对象一旦冻结，不允许“润色后替换原题”
   - 后续所有对象都只能引用它，不能重写它

2. `TopicRun`
   把原题编译成一次可执行研究任务，但仍不直接下结论。
   建议字段：
   - `topic_run_id`
   - `seed_id`
   - `run_goal`
   - `scope`
   - `out_of_scope`
   - `research_questions`
   - `candidate_angles`
   - `candidate_cohorts`
   - `normalization_frame`
   - `status`
   - `blockers`
   这层回答的是：
   - 这次到底在研究什么
   - 不研究什么
   - 准备用什么比较口径
   - 当前有哪些未决问题

3. `SourceCapture`
   一次原始采集动作，只负责保真，不负责解释。
   建议字段：
   - `capture_id`
   - `topic_run_id`
   - `source_type`
   - `source_label`
   - `url`
   - `captured_at`
   - `capture_method`
   - `raw_excerpt`
   - `structured_payload`
   - `screenshot_ref`
   - `html_ref`
   - `status`
   - `capture_notes`
   规则：
   - 一次访问一个来源，就是一条 capture
   - 原文片段、截图、结构化结果至少保留一种
   - 没采到就标失败，不允许事后补写成“已采”

4. `CollectionLog`
   记录这次 run 到底采了什么、没采到什么、为什么没采到。
   建议字段：
   - `log_id`
   - `topic_run_id`
   - `captures[]`
   - `attempted_sources[]`
   - `failed_sources[]`
   - `timeline`
   - `operator_notes`
   - `open_gaps`
   这层是审计账本。
   `SourceCapture` 是单条证据原件，`CollectionLog` 是全局采集台账。

5. `Evidence`
   从原始采集里抽出的“可引用单元”。
   建议字段：
   - `evidence_id`
   - `topic_run_id`
   - `claim_candidate`
   - `evidence_type`
   - `derived_from_capture_ids`
   - `source_class`
   - `freshness`
   - `risk_level`
   - `use_as`
   - `notes`
   `source_class` 固定先用：
   - `official`
   - `community`
   - `commercial`
   - `inference`
   `use_as` 固定先用：
   - `claim-ready`
   - `reference-only`
   - `blocked`

6. `EvidenceSet`
   面向某一个选定 page thesis 的经过审核证据集合。
   建议字段：
   - `evidence_set_id`
   - `topic_run_id`
   - `page_thesis`
   - `cohort_freeze`
   - `normalization_frame`
   - `included_evidence_ids`
   - `excluded_evidence_ids`
   - `key_claims`
   - `known_unknowns`
   - `review_summary`
   规则：
   - `EvidenceSet` 必须比原始 `Evidence` 更窄
   - 它只服务一个明确 thesis，不能又想做总览、又想做比较、又想做国家页

审核门禁我建议固定成 3 道，不通过就不能进 `EvidenceSet`：

1. `真实性门`
   能不能回到原始 capture
   过不了就直接 `blocked`
2. `归因门`
   这条内容到底是官方、社区、商业，还是模型推断
   分不清就不能写成硬结论
3. `风险门`
   即使是真的，能不能直接支撑页面结论
   高风险内容只能进 `reference-only` 或 `risk note`

对 `AI procurement` 这个案例，我建议额外加一条强规则：

`任何价格、计费、地区可用性结论，若没有明确时间戳和来源页，不得进入最终 claim-ready evidence`

这一段你看是否对。如果对，我继续第 3 段：把这条原始提问真正编译成第一版 `TopicRun` 设计。
