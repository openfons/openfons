## 北极星目标
将 OpenFons 当前主线切换为“平台级配置中心”，先完成统一的 `PluginType / PluginInstance / SecretRef / ProjectBinding / ResolvedRuntimeConfig` 底座，使 `control-api` 成为受控配置的唯一管理入口，并为 `search-gateway`、认证浏览器链路与后续 crawler adapter 提供同一套配置、校验、解析与 masking 规则。

## 范围
- 以 `docs/superpowers/specs/2026-04-07-platform-plugin-config-center-design.md` 作为当前设计基线。
- 当前主线先写并 review `docs/superpowers/plans/2026-04-07-platform-plugin-config-center.md`，不直接开始实现代码。
- 平台内建插件类别先锁定为 `search-provider`、`browser-runtime`、`crawler-adapter`、`account-source`、`cookie-source`、`proxy-source`。
- 统一区分 repo 可见配置与本地私有 secret，并通过 `SecretRef` 解析。
- 实现顺序固定为：第 1 批 `contracts + config-center core`，第 2 批 `control-api` 配置中心 API，第 3 批接入 `search-gateway` 与 `browser-runtime`，第 4 批接入 `yt-dlp / twscrape / TikTokApi / PRAW / MediaCrawler` 这类 crawler adapter。

## 非目标
- 本轮不继续扩写 `AI procurement / direct API vs OpenRouter` 公开 SEO 页面能力。
- 本轮不直接实现配置中心前端 UI 页面。
- 本轮不把 raw secret 写入 repo 可见配置。
- 本轮不引入数据库、远端 vault 或动态插件市场。
- 本轮不在计划 review 之前开始第 1 批代码实现。

## 验收信号
- 仓库内存在 `docs/superpowers/plans/2026-04-07-platform-plugin-config-center.md`，并明确 4 批实现顺序、文件边界与 TDD 切入点。
- `contracts`、配置中心 core、`control-api` API、`search-gateway`、`browser-runtime`、crawler adapter 的职责分层清楚，不再各自发明 secret / binding / masking 逻辑。
- 计划中明确 repo 配置目录、secret 目录、校验结果结构、runtime resolver 输出结构与 masked 管理视图边界。
- 第 1 批开始前，测试入口、关键类型与最小 seed config 已在计划中明确。
