## 北极星目标
OpenFons v1 的唯一北极星：把真实用户问题稳定编译成可验证的 `OpportunitySpec`、`EvidenceSet` 和 `ReportSpec`，先交付可盈利的网页报告，并为视频、文章与工具/订阅机会保留统一真源。

## v1 真源产物视图
1. `OpportunitySpec`
   原始用户问题的最终判定内容。它回答用户真正要什么、值不值得做、适合哪种页面或内容方向，以及是否存在工具站、数据产品或订阅机会。
2. `TopicRun / Source Captures / Collection Logs`
   原始采集层。这里保留搜索结果、HTML、截图、评论、价格页 capture、运行日志和失败记录，负责保真，不负责结论，不等于最终交付 Artifact。
3. `EvidenceSet`
   结构化证据层。它把原始采集整理成可引用、可复核、可支持 claim 的证据对象，供页面、视频、文章等交付共用。
4. `ReportSpec`
   第一阶段主交付编译层。它定义标题、slug、受众、地区、语言、section、必须保留的风险边界和更新日志，是网页报告的直接上游真源。
5. `Derived Content Specs`
   派生内容层。它从 `ReportSpec` 和 `EvidenceSet` 继续派生 `VideoOutline`、`Script`、`ArticleBrief`、`NewsletterBrief` 以及未来类似 `NotebookLM` 的多形态内容。
6. `ProductOpportunityHints`
   产品机会逻辑层。它先作为 `OpportunitySpec` 的内部字段保留，用于判断比较器、追踪页、工具站、会员或订阅产品的后续潜力，而不是新的平级外部契约。

## 范围
- 固定 v1 北极星和 6 层真源产物视图，避免系统目标在“采集验证 / 页面盈利 / 内容工厂 / 工具站”之间反复摇摆。
- 先以 `report-web / Next.js` 网页报告作为第一阶段默认盈利交付，但不把它误写成系统唯一产物。
- 保持从同一套真源派生视频、文章、博客、newsletter 和未来多形态内容。
- 在 `OpportunitySpec` 中保留工具站、数据产品和订阅机会 hint，供后续产品化评估使用。
- 将现有采集能力验证成果视为 `TopicRun / Source Captures / Collection Logs` 的基础设施，而不是独立终点。
- 将现有采集能力验证成果视为 `TopicRun / Source Captures / Collection Logs` 的基础设施，而不是独立终点。
- 明确 6 层视图是对真源与产物的表达，不替代正式执行链中的 `TaskSpec / WorkflowSpec`。

## 非目标
- 本轮不同时把视频、博客、newsletter、工具站和订阅产品全部产品化上线。
- 本轮不让页面、视频、文章或工具各自维护独立事实源。
- 本轮不跳过 `Opportunity Judge` 和 `User Confirmation` 直接进入批量采集或内容生成。
- 本轮不在没有单独 ADR / RFC 的情况下新增与 `ReportSpec` 平级的正式工具交付契约。

## 验收信号
- 当前项目活跃目标与待办都以同一条北极星描述系统目标，不再反复切换目标表达。
- 至少一个真实用户问题可以完整映射为 6 层产物树。
- `ReportSpec` 可以作为网页报告、视频、文章、newsletter 的共享上游真源。
- `OpportunitySpec` 明确保留产品机会 hint，用于后续判断比较器、追踪页、会员或订阅方向。
- 现有采集产物能够进入标准化 evidence 流程，而不是停留在孤立 artifact。
